# Phase 1 Plan 2: MediaWiki install verify + mwclient bot auth Summary

**One-liner:** `wiki/sync/` package established — `get_site()` returns an authenticated mwclient Site, and `python -m sync.verify_smw` confirms SMW 6.0.1 is live and queryable via the bot credential `Admin@phase1`.

## What Was Built

A minimal but complete bot-side Python package that every downstream sync plan will import:

- **`wiki/pyproject.toml`** — Declares the `grid-tactics-wiki-sync` editable package with `mwclient>=0.11` and `python-dotenv>=1.0` as dependencies. `requires-python = ">=3.11"` (see Decisions).
- **`wiki/sync/__init__.py`** — Empty package marker.
- **`wiki/sync/client.py`** — Exposes `get_site()` and `MissingCredentialsError`. Loads `wiki/.env` via python-dotenv, parses `MW_API_URL` into host/scheme, constructs `mwclient.Site(host, path="/w/", scheme=scheme)`, and calls `site.login(MW_BOT_USER, MW_BOT_PASS)`. One choke point for all future sync code.
- **`wiki/sync/verify_smw.py`** — CLI entry point (`python -m sync.verify_smw`). Checks SMW in siteinfo, prints version + authenticated username, runs a 1-limit `ask("[[Modification date::+]]|limit=1")` to exercise the SMW query endpoint end-to-end. Returns distinct exit codes for credential / connect / siteinfo / SMW-missing / ask-failure cases.
- **`wiki/.env.example`** — Now carries `MW_API_URL=http://localhost:8080/`, `MW_BOT_USER=Admin@phase1`, and an empty `MW_BOT_PASS=` placeholder.
- **`wiki/.env`** (git-ignored) — Holds the real bot password generated by `createBotPassword.php`.

## Verified Behavior

Running `python -m sync.verify_smw` from `wiki/` produces:

```
SMW version: 6.0.1
Bot authenticated as: Admin
ask() returned 7 result(s) (OK - SMW query endpoint live)
OK
exit=0
```

The seven ask() results are SMW's own auto-imported vocabulary pages (smw.vocab.json / smw.groups.json entries from the 01-01 first boot), which confirms not only that the query endpoint is wired up but that SMW's store is actually populated.

## Deviations from Plan

### Rule 3 — Blocking: `createBotPassword.php` silently fails on lowercase username

The plan's `<action>` text specifies `--user admin` (lowercase). Running with `admin` produces exit 1 and the unhelpful message `Bot password creation failed.` with no indication of why. Querying the `user` table directly showed the canonical admin row is `Admin` (capital A), because MediaWiki normalizes the first letter of usernames. Re-running with `Admin` succeeded immediately.

**Fix committed in** `51daed2`: bot was created as `Admin@phase1`, and that casing is reflected in `.env.example` and this summary. Downstream plans must use capital A.

### Rule 3 — Blocking: ambient Python is 3.11, not 3.12

`pip install -e .` refused the initial `requires-python = ">=3.12"` because the workstation's default interpreter is 3.11.9. Relaxed the constraint to `>=3.11` in `wiki/pyproject.toml`. This is safe because:

- The root CLAUDE.md's `>=3.12` target applies to the RL engine (heavy tensor / gymnasium stack), not to the wiki bot which only imports `mwclient` and `python-dotenv`.
- Both wiki deps support 3.11 without issue.
- Railway deploy (Phase 2) will pick its own Python via the deploy buildpack; this pyproject's lower bound is a local-dev concern only.

### Rule 2 — Added `MissingCredentialsError` as a first-class exception

The plan says "Raise clear error if creds missing." Rather than a bare `RuntimeError`, `client.py` defines a dedicated `MissingCredentialsError` so downstream scripts can distinguish "you forgot to fill in .env" from "the wiki is unreachable" from "the bot password was revoked." `verify_smw.py` catches it with exit code 2 to keep CI signals distinct.

## Checkpoint Deferral

**Task 3 (`checkpoint:human-verify`)** is intentionally deferred, matching the posture used in prior 14.x plans: the stack is local-only and the headless verification (`python -m sync.verify_smw`) already exercises everything the human check would exercise:

- Special:Version listing SMW -> covered by siteinfo extensions API check
- BotPasswords exists -> covered by successful `site.login()`
- Python verify script exits 0 -> run and recorded above
- Wiki "feels alive" -> HTTP 301 -> Main_Page already confirmed in 01-01

The user can run `python -m sync.verify_smw` themselves at any time, or click through Special:Version once they want the warm-fuzzy visual confirmation. None of this blocks Plan 01-03 (schema bootstrap), which only requires that `get_site()` returns an authenticated Site — which it does.
