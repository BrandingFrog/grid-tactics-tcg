## MW_DEBUG handling

`MW_DEBUG=1` remained set on the Railway wiki service throughout the rollout and verification, ensuring any secondary exception would have surfaced in HTTP 500 response bodies. After user sign-off at the Task 4 checkpoint, flipped to `MW_DEBUG=0` via `variableUpsert` GraphQL mutation. Post-flip health check confirmed all canonical pages still return 200.

## Surprises / deviations from plan

1. **MySQL mesh DNS staleness on fresh Railway containers.** Both deployments built from commit `2712922` initially couldn't reach `mysql.railway.internal:3306`. The first deployment (`5486d08f`) cycled through the entrypoint retry loop without recovering. Manual `serviceInstanceRedeploy` (`87bd7f5e`) recovered on its own after ~2 min of retries. Captured as a decision in STATE.md ŌĆö this is a Railway-side issue, not anything in the stack, and the fix is always `serviceInstanceRedeploy`. Do NOT touch MySQL.

2. **SMW parser cache does not invalidate on `action=purge`.** After the upgrade, `Semantic:Showcase` was stuck showing partial pre-upgrade renders (only Q2 and Q4 had tables; the other 5 were blank). `action=purge` with `forcerecursivelinkupdate=1` via POST did not fix it. The fix was to run `python -m sync.sync_showcase` which performs an authenticated edit via the bot account; that forces a full re-parse. Captured as a decision in STATE.md ŌĆö Phase 9.2 execution will need to re-run the taxonomy/showcase/homepage sync scripts after installing Drilldown so the faceted pages don't serve stale cached renders.

3. **Plan test-name error: `Category:Minion` vs `Category:Minions`.** The plan tested a category name that doesn't exist on this wiki (only the plural form does). Easy fix: substituted the correct name in verification and captured the canonical naming convention in STATE.md.
