# Phase 9.1: SMW DisplayTitleLookup Backtick Fix — Research

**Researched:** 2026-04-11
**Domain:** Semantic MediaWiki (SMW) compatibility with hardened MediaWiki 1.43 Rdbms layer
**Confidence:** HIGH

## 1. Summary

`GET /wiki/Category:Card` (and every other SMW-touched page on the Grid Tactics wiki) returns HTTP 500 because SMW 5.1.0's `DisplayTitleLookup::fetchFromTable` pre-wraps the `smw_fpt_dtitle` table name with `$connection->tablename(...)` before passing it to `$db->select()`. MediaWiki 1.43's hardened `Wikimedia\Rdbms\Platform\SQLPlatform::addIdentifierQuotes` then tries to re-quote the already-backticked identifier and throws `InvalidArgumentException: "cannot contain quote, dot or null characters: got '\`smw_fpt_dtitle\`'"`.

The bug is already fixed upstream in SMW **6.0.0** (released 2025-08-12) by commit `600f520` / PR **#6172**, which closed issues **#6150** and **#6158** (the latter has our exact stack trace). The fix is a one-line change: replace `$connection->tablename( $propTable->getName() )` with `$propTable->getName()`. SMW **6.0.1** (released 2025-08-26) is a bugfix release that retains this change and adds MW 1.44 compatibility patches.

**Primary recommendation:** Upgrade SMW from `~5.0` to `~6.0` in `wiki/Dockerfile` (resolves to 6.0.1). SMW 5.1.x is officially obsolete and only supports MW 1.43.1 — we are on 1.43.8. No `update.php` / `setupStore.php` / schema migration is required for 5.x → 6.x. **Fallback** if the upgrade surfaces unforeseen regressions: in-place `sed` patch of `DisplayTitleLookup.php` line 122 in the Dockerfile build step (the exact upstream hunk from PR #6172).

## 2. Root Cause Confirmation

Re-verified against the 5.1.0 release tag (not re-derived). Source: `https://raw.githubusercontent.com/SemanticMediaWiki/SemanticMediaWiki/5.1.0/src/SQLStore/Lookup/DisplayTitleLookup.php`.

The offending method (file `src/SQLStore/Lookup/DisplayTitleLookup.php`, in the 5.1.0 tree the `$connection->select(...)` call begins on line **122** and the tablename wrap is **line 123**; stack trace from the running Railway instance cites line 124 because of whitespace / banner differences between 5.0.0 and 5.1.0 — same offense, same hunk):

```php
private function fetchFromTable( $list ) {
    $property = new DIProperty( '_DTITLE' );
    $connection = $this->store->getConnection( 'mw.db' );

    $propTableId = $this->store->findPropertyTableID(
        $property
    );

    $propTables = $this->store->getPropertyTables();

    if ( !isset( $propTables[$propTableId] ) ) {
        throw new RuntimeException( "Missing $propTableId!" );
    }

    $propTable = $propTables[$propTableId];

    $rows = $connection->select(
        $connection->tablename( $propTable->getName() ),    // <-- THE BUG (line 123)
        [
            's_id',
            'o_hash',
            'o_blob'
        ],
        [
            's_id' => array_keys( $list )
        ],
        __METHOD__
    );

    return $rows;
}
```

**Mechanism:** `$connection->tablename( $propTable->getName() )` returns the MySQL-quoted form `` `smw_fpt_dtitle` `` (MW's `Database::tableName()` backward-compat behavior wraps identifiers in backticks for the current DB type). SMW then passes that backticked string as the `$table` argument to `$connection->select()`. Inside MW 1.43's `Wikimedia\Rdbms\Platform\SQLPlatform::tableNameWithAlias()` the value is re-fed through `addIdentifierQuotes()`, which now (MW 1.43 hardening) rejects any string containing a backtick, quote, dot, or null byte, throwing the `InvalidArgumentException` observed on Railway.

**Upstream call chain confirmed:** CacheWarmer → DisplayTitleFinder → DisplayTitleLookup → fetchFromTable → Database::select → SQLPlatform::addIdentifierQuotes → throw. Matches the Railway live-trace the user provided exactly, including frame #9 pointing at `DisplayTitleLookup.php(124)`.

## 5. Local Constraints (Task 4)

### `wiki/Dockerfile` — composer install flow

Key lines (from `D:\windsurf\card game\wiki\Dockerfile`):

- Line 34–38: composer require block that installs SMW. Runs **as `root`** (`COMPOSER_ALLOW_SUPERUSER=1`), uses `--no-audit --update-no-dev`, and pins `mediawiki/semantic-media-wiki "~5.0"`. Followed immediately by `composer dump-autoload --optimize --no-dev` and `chown -R www-data:www-data /var/www/html`.
- Line 30: existing cache-bust comment `# Cache bust: 2026-04-09T17 — force rebuild for ParserFunctions`. Bumping this forces the Railway build pipeline to skip the layer cache.
- Line 44–46: Citizen skin install (separate `composer require`), must remain untouched.

**Constraint for the fix:**
1. Whatever change the planner makes, it must land **between the SMW `composer require` (line 35-36) and the final `chown` (line 38)** — otherwise www-data can't serve the patched file, or a later `dump-autoload` runs before the patch is in place.
2. The `~5.0` constraint needs to become `~6.0` (Option C) **or** a `RUN sed -i ...` must fire after the composer require (Option A). Either way, the cache-bust comment should be bumped to force a rebuild.
3. `--no-audit --update-no-dev` must stay — the mediawiki:1.43 base image ships a composer.json with a vulnerable phpunit in require-dev (PKSA-z3gr-8qht-p93v) which is handled by the `php -r` transform on line 34.

### `wiki/LocalSettings.php` — DisplayTitle settings

Lines 80–82 (verified):
```php
# Allow DISPLAYTITLE to hide namespace prefix (e.g. Card:Ratchanter -> Ratchanter)
$wgAllowDisplayTitle = true;
$wgRestrictDisplayTitle = false;
```

SMW's `_DTITLE` property is populated by the `{{DISPLAYTITLE:...}}` parser call in every card template. **The fix must not break this** — cards rely on `Card:Ratchanter` rendering as `Ratchanter`. Both the Option A sed patch and the Option C upgrade are functionally equivalent on this axis: both produce `$connection->select( $propTable->getName(), ... )`, which is what master / 6.0.x do, and DisplayTitle rendering works on every current SMW 6.0.x deployment.

MW_DEBUG block at lines 116–141 (commit `4c7eeb1`) is gated on `MW_DEBUG=1` env var. Keep it on for the 9.1 rollout so secondary exceptions (if any) surface immediately in the HTTP response body.

### `wiki/sync/templates/Card.wiki` — why CacheWarmer touches `smw_fpt_dtitle`

Every card page ends with:
```wikitext
{{DISPLAYTITLE:{{{name|{{PAGENAME}}}}}}}{{DEFAULTSORT:...}}{{#set:Name=... }}...[[Category:Card]]
```

The `{{DISPLAYTITLE:...}}` call is what populates the `_DTITLE` SMW property into the `smw_fpt_dtitle` fixed-property table. When MediaWiki renders `Category:Card`, it needs to resolve the display titles for every page linked from the category listing — which invokes `CacheWarmer::prepareCache()` → `DisplayTitleFinder::prefetchFromList()` → our crash site. This is why the bug manifests on *every* category page (not just `Category:Card`) and why it is the correct single bug to fix for Phase 9.1. Any category page covering pages that set `DISPLAYTITLE` will hit the same code path.

### `wiki/sync/schema.py` — no SMW property changes required

Declares 20 custom SMW properties (`Name`, `StableId`, `CardType`, `Element`, `Tribe`, `ManaCost`, `Attack`, `HP`, `Range`, `ArtFile`, `FirstPatch`, `LastChangedPatch`, `Deckable`, `LastModified`, etc.). `_DTITLE` is a core SMW property, not one of our custom 20, so the fix does not touch schema.py. No bootstrap_schema re-run needed after the fix.

### `wiki/composer.json`

**Does not exist.** The image uses the mediawiki:1.43 base's own composer.json, patched in-place by the Dockerfile's `php -r` transform on line 34. This is relevant for evaluating Option B (cweagans/composer-patches): we'd have to either create a new `wiki/composer.json` (adding a second managed-file layer) or `composer require` the package in the same run that installs SMW.

## 6. Fix Option Evaluation (Task 5)

### Option A — Dockerfile `sed` patch

**How it works:** Add a `RUN sed -i 's/$connection->tablename( $propTable->getName() )/$propTable->getName()/' extensions/SemanticMediaWiki/src/SQLStore/Lookup/DisplayTitleLookup.php` step (or a more robust `patch` step with a HEREDOC'd diff) immediately after the SMW `composer require` and **before** the `chown`.

**Lines touched:** `wiki/Dockerfile` — one new `RUN` block (~3 lines). No other files.

**Brittleness:**
- Survives Docker image rebuilds (cache-bust comment handles this).
- **Fragile across SMW minor upgrades**: if we later bump the `~5.0` pin and upstream has already refactored around that line (e.g. moved it to `fetchFromTableBatch`), `sed` silently no-ops and we re-introduce the 500. Mitigation: add a post-sed verification step like `grep -c 'tablename( $propTable' extensions/SemanticMediaWiki/src/SQLStore/Lookup/DisplayTitleLookup.php` that exits non-zero if the line is still present after the patch.
- Invisible to a human reading the SMW source tree — future maintainers will be confused that the installed file diverges from the composer lock.
- Composer dump-autoload re-runs after the sed is safe (autoloader doesn't cache method bodies).

**Rollback cost:** Revert one Dockerfile commit and rebuild. ~5 min.

**Recommendation:** Use only as a *fallback* if Option C surfaces a real regression. It is the minimal-surface-area change, but it leaves the codebase on obsolete-and-unsupported SMW 5.1.0 which has an explicit MW-1.43.1-only support claim — we're already outside that window at 1.43.8.

### Option B — `cweagans/composer-patches`

**How it works:** Create `wiki/composer.json` with the cweagans package declared in `require`, a `patches` entry under `extra` pointing at `wiki/patches/smw-displaytitle-backtick-fix.patch`, and commit the patch file verbatim from PR #6172. The Dockerfile `composer require` step needs to either (a) copy this composer.json into `/var/www/html` before running composer, or (b) chain an additional `composer require cweagans/composer-patches` plus a manual `patch -p1` step.

**Lines touched:**
- New: `wiki/composer.json` (~15 lines)
- New: `wiki/patches/smw-displaytitle-backtick-fix.patch` (~12 lines)
- Modified: `wiki/Dockerfile` — COPY step and/or extra composer require

**Brittleness:**
- Survives Docker rebuilds.
- Survives SMW minor bumps **only if the patched file hasn't been touched** — otherwise the patch fails to apply cleanly and the Docker build loudly errors (this is actually a *feature*, not a bug: fail-loud is better than silent sed no-op).
- Creates a tension with the existing `php -r` transform on Dockerfile:34 that mutates the base image's composer.json in place. Merging our composer.json with the base's is non-trivial: composer doesn't natively merge two unrelated composer.json files in the same directory, and the mediawiki base image's composer.json has quirky `require-dev` entries.
- Composer-patches support matrix: cweagans/composer-patches v1 works with Composer 2, but its interaction with `composer require --update-no-dev` in the Dockerfile flow is untested here.
- More machinery = more code to own. For a one-line fix to one file in one codebase, this is over-engineering.

**Rollback cost:** Delete `wiki/composer.json`, `wiki/patches/`, revert Dockerfile changes. ~10 min, more surface area to unwind than Option A.

**Recommendation:** **Reject.** The operational complexity of reconciling a new composer.json with the existing Dockerfile composer flow outweighs the "readable patch file" benefit, especially when Option C gets us the same patch via a one-character version pin bump.

### Option C — Upgrade SMW to `~6.0` (primary)

**How it works:** Change `wiki/Dockerfile` line 36 from `mediawiki/semantic-media-wiki "~5.0"` to `mediawiki/semantic-media-wiki "~6.0"`. Bump the cache-bust comment on line 30 to force a fresh build. That is the entire code change.

SMW 6.0.1 (the highest `~6.0` resolves to at the moment) contains the merged PR #6172 fix. It supports MW 1.43 - 1.44 and PHP 8.1 - 8.4. We run MW 1.43.8 and PHP 8.3.30 — squarely in-range. The 6.0.0 release notes state explicitly: *"No need to run 'update.php' or any other migration scripts."* No SMW schema migration. No setupStore.php. Composer resolves, autoloader regenerates, `chown` runs, container restarts.

**Breaking changes introduced by 6.0.0 (audited for our usage):**

| Breaking change | Impact on Grid Tactics wiki |
|-----------------|-----------------------------|
| Reverted support for "additional formatting in ask queries" | **None** — Phase 9.2's planned ask queries are simple property filters (`[[CardType::Minion]]`), not formatted sub-expressions |
| `$wgFooterIcons` array key structure change | **None** — we don't customize footer icons in LocalSettings.php |
| Dropped support for MW <1.43 | **Helps us** — we're on 1.43.8 |

**Lines touched:** `wiki/Dockerfile` — two lines modified (cache-bust comment + version pin).

**Brittleness:**
- **Most robust option.** The fix lives where upstream put it. Future SMW 6.0.x bugfix releases will flow in via `~6.0` (which is `>=6.0,<7.0`) automatically on rebuild.
- `~6.0` could resolve to a future 6.0.x or 6.1.x with its own regressions — but that's a general concern with constraint-based dependency pins, not unique to this fix, and is mitigated by controlling rebuild timing via the cache-bust comment.
- **No local diverge from upstream source.** Any future maintainer can `cat extensions/SemanticMediaWiki/src/SQLStore/Lookup/DisplayTitleLookup.php` and see the same code the SMW repo has.

**Rollback cost:** Revert one commit on `wiki/Dockerfile`, redeploy. ~5 min. Same or simpler than Option A.

**Recommendation:** **PRIMARY PATH.** Smallest code footprint, cleanest maintenance story, and as a bonus it moves us from an *obsolete* SMW release to a *stable* one that actually claims MW 1.43 support.

## 7. Recommended Fix Path

**Primary: Option C — upgrade SMW to `~6.0` in `wiki/Dockerfile`.**

The planner should produce a PLAN.md with these task actions:

1. Bump the cache-bust comment on `wiki/Dockerfile:30` to the current timestamp so Railway's build cache misses this layer.
2. Change `wiki/Dockerfile:36` from `mediawiki/semantic-media-wiki "~5.0"` to `mediawiki/semantic-media-wiki "~6.0"`.

4. Run verification steps from §8.
5. After 24h of green, consider disabling `MW_DEBUG` in a follow-up (not part of 9.1 scope).

**Fallback: Option A — Dockerfile `sed` patch.**

Trigger the fallback if, after Option C redeploy, any of these happen:
- Composer resolution fails (dependency conflict with Citizen skin, parser-hooks, etc.)
- New exception in a different SMW code path that wasn't on our radar (6.0.0's ~500-file delta vs 5.1.0 is non-trivial)
- Card page rendering breaks, category listings show wrong titles, or Phase 9.2 ask queries misbehave

The fallback change is self-contained — insert this block in `wiki/Dockerfile` **after** line 37 (the composer dump-autoload) and **before** line 38 (the `chown`), and revert the `~6.0` pin back to `~5.0`:

```dockerfile
# Fix SMW 5.1.0 DisplayTitleLookup backtick bug (upstream PR #6172, MW 1.43 hardened Rdbms compat).
# See wiki/.planning/phases/09.1-smw-displaytitle-backtick-fix/09.1-RESEARCH.md
RUN DTFILE=extensions/SemanticMediaWiki/src/SQLStore/Lookup/DisplayTitleLookup.php \
 && test -f "$DTFILE" \
 && grep -q '\$connection->tablename( \$propTable->getName() )' "$DTFILE" \
 && sed -i 's|\$connection->tablename( \$propTable->getName() )|\$propTable->getName()|' "$DTFILE" \
 && ! grep -q '\$connection->tablename( \$propTable->getName() )' "$DTFILE" \
 && echo "SMW DisplayTitleLookup backtick fix applied"
```

The pre-`grep -q` asserts the vulnerable line exists (fails loud if SMW bumped and moved it). The post `! grep -q` asserts the line is gone after sed (fails loud if the substitution didn't apply).

## 9. Rollback Plan (Task 7)

### Primary rollback — Option C fails

Single-commit revert: `git revert <commit-that-bumped-to-~6.0>` on `wiki/Dockerfile`, push, redeploy. Image rebuilds with SMW 5.1.0 pinned again. **The site goes back to the known-broken state** — this rollback only makes sense if the upgrade introduced something *worse* than the displaytitle 500 (e.g., data corruption, auth breakage, or a hard crash before the MW 500 handler runs).

In practice, if Option C triggers a regression, the correct move is **forward** to Option A, not **back** to the 5.1.0 baseline:

1. Revert the `~6.0` pin to `~5.0`.
2. In the **same commit**, add the Option A sed patch block (from §7).
3. Push, redeploy. Now we're back on 5.1.0 with the one-line DisplayTitleLookup fix applied, which is the minimum viable state.

### Secondary rollback — Option A fails

Revert the sed-patch commit, redeploy. Site returns to the fully-broken 5.1.0 baseline. At that point the bug is blocking and the user needs to be looped in — both Options A and C failing simultaneously would be a **RESEARCH BLOCKED** outcome that this phase's planning would need to escalate.

### Debug envelope stays on

`MW_DEBUG=1` is already set as a Railway env var on service `08a52887-a4ab-4f22-b03d-006da72e07f7`. **Do not turn it off** during the 9.1 rollout. It stays on for at least one full deployment cycle so that any secondary exception appears in the HTTP 500 response body (not just in Railway logs), which is fetchable with a single curl. After 9.1 is verified green for 24h, a follow-up commit can toggle it off. That toggle is **not** in scope for 9.1's PLAN.md.

### Data safety note

Neither option touches the database. SMW 5.1.0 → 6.0.1 is explicitly a "no update.php" upgrade per the 6.0.0 release notes. The `smw_fpt_dtitle` table and all 20 custom properties in `schema.py` remain untouched. No backup is strictly required, but a `mysqldump` of the Railway MySQL before redeploy is cheap insurance and is recommended if the planner wants to be extra careful — the existing `wiki/sync/backup.py` presumably handles this.

1. **Upstream PR — not needed.** The fix is already merged (PR #6172) and released (SMW 6.0.0 / 6.0.1). No upstream contribution is warranted. Both referenced issues (#6150, #6158) are closed.

2. **SMW 5.1.x is obsolete — document this in the wiki runbook.** The upstream compatibility page explicitly calls SMW 5.1.x obsolete and caps it at MW 1.43.1. Anyone who reinstates the 5.1.x pin in the future will silently re-introduce this bug (and likely others) on MW 1.43.8. Consider adding a one-line comment in the Dockerfile next to the version pin: `# ~6.0+ required for MW 1.43.2+ per https://www.semantic-mediawiki.org/wiki/Compatibility`.

3. **MW_DEBUG toggle-off follow-up.** A separate tiny phase (9.1.1 or just a `/gsd:quick`) should flip `MW_DEBUG=0` on the Railway service after the 9.1 fix has been stable for 24h. Leaving exception traces in production response bodies is a minor info-disclosure risk.

Verification step #5 in §8 is the canary. If #5 passes on the 6.0.1 upgrade, 9.2 is unblocked. If it fails, 9.2 planning will need its own research sub-task.

I confirmed patches to `SpecialConcepts`, `ListLookup`, `UsageStatisticsListLookup`, and `DisplayTitleLookup`, but did not exhaustively diff all 300 files in the PR.
