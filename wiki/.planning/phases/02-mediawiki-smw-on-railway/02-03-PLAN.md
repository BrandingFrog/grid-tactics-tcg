Phase 1 (01-02) established that the bot credential format is `Admin@phase2` (capital A — lowercase makes `createBotPassword.php` silently fail with exit 1). Use `phase2` as the appid to distinguish from the local Phase 1 credential which lived in a different DB.

Run the MediaWiki maintenance script inside the live container via `railway run`. Taqasta's maintenance scripts live at `/var/www/mediawiki/w/maintenance/`:

```bash
# From wiki/ directory, already linked to the Railway project
railway run --service mediawiki -- php /var/www/mediawiki/w/maintenance/createBotPassword.php \
  --appid phase2 \
  --grants basic,highvolume,editpage,createeditmovepage \
  Admin
```

Expected output: the script prints the generated BotPassword (a ~32-char string) to stdout. CAPTURE THIS VALUE — it cannot be retrieved later, only reset.

If the path differs (Taqasta's MediaWiki root varies by build), first locate it:
```bash
railway run --service mediawiki -- bash -c "find / -name 'createBotPassword.php' 2>/dev/null"
```
Re-run with the correct path.

<task type="auto">
  <name>Task 2: Run Phase 1 bootstrap sequence against live wiki</name>
  <files>wiki/.planning/STATE.md</files>
  <action>
Phase 1's STATE.md explicitly calls out the required sequence (watch item under "Phase 2 watch item"):
> post-deploy bootstrap sequence on Railway must run in order — `bootstrap_schema.py` → `verify_schema.py` → `bootstrap_template.py` → `create_sample_card.py`. All four scripts are idempotent.

Run them in that exact order from `wiki/`, with `.env` pointing at the Railway URL (from Task 1):

```bash
cd wiki/
python -m sync.bootstrap_schema
python -m sync.verify_schema
python -m sync.bootstrap_template
python -m sync.create_sample_card
python -m sync.verify_smw
```

Expected behavior per script:
1. `bootstrap_schema`: creates 25 Property: pages (20 core + 5 effect subobject fields). May encounter `smw-change-propagation-protection` errors — the script's `_edit_with_retry()` handles this with linear backoff up to 6 times (per Phase 1 01-03 decision). Allow up to ~5 minutes total on first run.
2. `verify_schema`: confirms each property page exists and has the `[[Has type::X]]` marker. Phase 1 01-03 noted `ask([[Property:+]])` is a soft signal on SMW 6.0.1 — the page-level marker is the authoritative gate. Script exit code 0 = pass.
3. `bootstrap_template`: uploads `sync/templates/Card.wiki` as `Template:Card`. Idempotent via the `page.text().rstrip() == expected.rstrip()` comparison (Phase 1 01-03 pattern).
4. `create_sample_card`: creates `Card:Ratchanter` from the real `data/cards/ratchanter.json`. Phase 1 01-04 noted sample cards MUST source from real JSON, not illustrative values.
5. `verify_smw`: final smoke test (Phase 1 decision: canonical smoke test for all downstream sync plans). Runs an SMW #ask query for Card:Ratchanter and asserts Cost=4, HP=30.

If any script fails:
- Schema errors → likely SMW change-propagation lock; retry the script (each script is idempotent, safe to re-run).
- Template errors → check `wiki/sync/templates/Card.wiki` exists (committed in Phase 1 01-04).
- Card verification fails → the Card.wiki template might not be present or SMW hasn't re-parsed Card:Ratchanter. Wait 30s and re-run `verify_smw`.
- Auth errors → re-check `wiki/.env` (especially `MW_BOT_USER=Admin@phase2` with capital A per Phase 1 01-02 decision).

After all 5 scripts pass, update `wiki/.planning/STATE.md`:
- Set `status: phase-2-complete`
- Set `stopped_at: completed_02-03`
- Set `progress.phase: 2`, `progress.phase_name: MediaWiki + SMW Deployment on Railway`, `progress.plan: 03`, `progress.plans_completed_in_phase: 3`, `progress.plans_total_in_phase: 3`, `progress.phases_completed: 2`, `progress.percent: 22`
- Append to Accumulated Context → Decisions: Phase 2 deploy decisions (Taqasta pinned tag, Railway service topology, live URL, bot credential appid `phase2`)
- Clear the "Phase 2 watch item" entries from Pending Todos (they're now addressed)
