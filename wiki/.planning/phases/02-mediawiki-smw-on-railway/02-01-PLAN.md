<objective>
Stand up the MariaDB service on Railway as the first and lowest-risk piece of the Phase 2 stack. This plan creates the Railway project, defines service config, and deploys ONLY the database so the Phase 1 MariaDB-error-1133 fix can be validated in isolation before any MediaWiki boot attempt.

Purpose: Phase 1 burned an entire plan debugging first-boot auth failures. Deploying the DB first and verifying the `wiki` user exists BEFORE the MediaWiki service ever tries to connect de-risks the most painful historical failure mode.
Output: A live `mariadb` Railway service with the `wiki` DB and user pre-provisioned, reachable on Railway's private network, plus `railway.json` committed so the topology is reproducible.
</objective>

<task type="auto">
  <name>Task 1: Install Railway CLI and initialize project</name>
  <files>wiki/.env.railway.example</files>
  <action>
Verify Railway CLI is installed (`railway --version`). If missing, install via the documented method for the host (Windows: `npm i -g @railway/cli` or scoop; macOS: `brew install railway`). Do NOT use a method requiring admin escalation without asking.

Run `railway login` — this will open a browser. If the login fails or the user is not authenticated, STOP and create a `checkpoint:human-action` note (this is a legitimate auth gate — Railway login requires a browser redirect).

Once authenticated:
1. `cd wiki/`
2. `railway init` — choose "Empty Project", name it `grid-tactics-wiki`.
3. `railway link` if not auto-linked.
4. Record the project ID printed by `railway status` in the summary.

Then create `wiki/.env.railway.example` (committed — it is a template, not secrets) documenting every env var the Railway dashboard will need:

```
# Railway env var template for grid-tactics-wiki
# Copy these into the Railway dashboard per-service. DO NOT commit real values.

# --- mariadb service ---
MARIADB_ROOT_PASSWORD=<generate-strong-password>
MARIADB_DATABASE=wiki

# --- mediawiki service (set in plan 02-02) ---
MW_SITE_NAME=Grid Tactics Wiki
MW_SITE_SERVER=https://<your-railway-public-url>
MW_SITE_LANG=en
MW_ADMIN_USER=Admin
MW_ADMIN_PASS=<generate-strong-password>
MW_DB_TYPE=mysql
MW_DB_SERVER=${{mariadb.RAILWAY_PRIVATE_DOMAIN}}
MW_DB_NAME=wiki
MW_DB_USER=wiki
MW_DB_PASS=wikipass
# CRITICAL: installer user MUST equal runtime user (Phase 1 fix for error 1133)
MW_DB_INSTALLDB_USER=wiki
MW_DB_INSTALLDB_PASS=wikipass
MW_MAIN_CACHE=CACHE_REDIS
MW_REDIS_SERVERS=${{redis.RAILWAY_PRIVATE_DOMAIN}}:6379
MW_LOAD_EXTENSIONS=SemanticMediaWiki,SemanticResultFormats,PageForms,ParserFunctions,Arrays,Scribunto,CategoryTree
MW_ENABLE_UPLOADS=true
MW_FILE_EXTENSIONS=png,jpg,jpeg,webp,svg
MW_SECRET_KEY=<generate-64-hex-chars>
```

NOTE on `MW_DB_PASS=wikipass`: Phase 1's db-init SQL hardcodes `wikipass`. Either (a) keep this for Railway too and accept the Phase 1 `[01-01 minor]` blocker, or (b) template the SQL later. For Phase 2 we deliberately take option (a) to stay identical to local — do not change this without also editing `wiki/db-init/01-create-wiki-user.sql`.

<task type="auto">
  <name>Task 2: Write railway.json and deploy the mariadb service with init SQL</name>
  <files>
wiki/railway.json
wiki/db-init/01-create-wiki-user.sql
  </files>
  <action>
Create `wiki/railway.json` describing the three services the phase will need (mariadb, mediawiki, redis) with image pins. Only the mariadb service will be deployed in THIS plan — mediawiki and redis are pre-declared so plan 02-02 can simply `railway up` them.

```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "services": [
    {
      "name": "mariadb",
      "source": {"image": "mariadb:10.11"},
      "volumes": [{"mountPath": "/var/lib/mysql", "name": "db_data"}]
    },
    {
      "name": "redis",
      "source": {"image": "redis:7-alpine"},
      "volumes": [{"mountPath": "/data", "name": "redis_data"}]
    },
    {
      "name": "mediawiki",
      "source": {"image": "ghcr.io/wikiteq/taqasta:1.43.5-20260408-97"},
      "volumes": [{"mountPath": "/mediawiki", "name": "mw_data"}]
    }
  ]
}
```

NOTE: Taqasta tag `1.43.5-20260408-97` is pinned explicitly (verified as the latest stable on ghcr.io/wikiteq at plan time). Do NOT use `:latest` — Phase 1 decision to move off `:latest` for reproducibility.

Railway's `railway.json` schema does not currently support every field we want (specifically, mounting a host file into the container's `/docker-entrypoint-initdb.d/`). So the init SQL must be delivered via a different mechanism. Use ONE of the following, in order of preference:

**Option A (preferred): Use a custom mariadb image**
Create `wiki/docker/mariadb/Dockerfile`:
```dockerfile
FROM mariadb:10.11
COPY ./db-init/01-create-wiki-user.sql /docker-entrypoint-initdb.d/01-create-wiki-user.sql
```
Change the mariadb service `source` in `railway.json` to `{"repo": "wiki", "dockerfilePath": "docker/mariadb/Dockerfile"}` (Railway will build from the linked GitHub repo). The file copy ensures the init SQL runs on first boot of the volume.

**Option B (fallback): Use `railway run` with a shell escape after first boot to bootstrap the user manually**
Only if Option A fails. Run `railway run --service mariadb -- bash -c "mysql -uroot -p$MARIADB_ROOT_PASSWORD < /path/to/sql"`. This is brittle — prefer Option A.

Pick Option A. Create the Dockerfile, update `railway.json` accordingly.

Now set the mariadb env vars via CLI (or direct the user to do it in the dashboard — this is a `checkpoint:human-action` candidate but we can automate with `railway variables --service mariadb set KEY=VALUE`):

```bash
railway variables --service mariadb --set MARIADB_ROOT_PASSWORD=<generated>
railway variables --service mariadb --set MARIADB_DATABASE=wiki
```

Use a locally-generated strong password (`openssl rand -hex 24`) for `MARIADB_ROOT_PASSWORD`. Print it to the terminal but DO NOT write it to any committed file. Instruct the user to also save it to their password manager.

Deploy the mariadb service only:
```bash
railway up --service mariadb --detach
```

Watch logs until you see `mariadbd: ready for connections`:
```bash
railway logs --service mariadb
```
