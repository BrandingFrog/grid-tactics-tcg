<task type="auto">
  <name>Task 1: Deploy redis service and set MediaWiki env vars</name>
  <files>wiki/railway.json</files>
  <action>
Deploy the redis service first (no env vars needed beyond defaults):
```bash
railway up --service redis --detach
railway logs --service redis
```
Wait until logs show `Ready to accept connections`. Confirm the service has a `RAILWAY_PRIVATE_DOMAIN` populated: `railway variables --service redis` should list it.

Now set ALL mediawiki env vars on the mediawiki service. This is where the Phase 1 error-1133 fix gets cemented — pay CRITICAL attention to `MW_DB_INSTALLDB_USER`:

```bash
# Generate a strong secret key first
MW_SECRET=$(openssl rand -hex 32)

# Use railway variables --service mediawiki --set KEY=VALUE for each:
railway variables --service mediawiki --set MW_SITE_NAME="Grid Tactics Wiki"
railway variables --service mediawiki --set MW_SITE_LANG=en
railway variables --service mediawiki --set MW_ADMIN_USER=Admin
railway variables --service mediawiki --set MW_ADMIN_PASS="$(openssl rand -hex 16)"
railway variables --service mediawiki --set MW_DB_TYPE=mysql
railway variables --service mediawiki --set MW_DB_SERVER='${{mariadb.RAILWAY_PRIVATE_DOMAIN}}'
railway variables --service mediawiki --set MW_DB_NAME=wiki
railway variables --service mediawiki --set MW_DB_USER=wiki
railway variables --service mediawiki --set MW_DB_PASS=wikipass
# CRITICAL — installer user == runtime user. This is the Phase 1 fix.
railway variables --service mediawiki --set MW_DB_INSTALLDB_USER=wiki
railway variables --service mediawiki --set MW_DB_INSTALLDB_PASS=wikipass
railway variables --service mediawiki --set MW_MAIN_CACHE=CACHE_REDIS
railway variables --service mediawiki --set MW_REDIS_SERVERS='${{redis.RAILWAY_PRIVATE_DOMAIN}}:6379'
railway variables --service mediawiki --set MW_LOAD_EXTENSIONS=SemanticMediaWiki,SemanticResultFormats,PageForms,ParserFunctions,Arrays,Scribunto,CategoryTree
railway variables --service mediawiki --set MW_ENABLE_UPLOADS=true
railway variables --service mediawiki --set MW_FILE_EXTENSIONS=png,jpg,jpeg,webp,svg
railway variables --service mediawiki --set MW_SECRET_KEY="$MW_SECRET"
# Temporarily set SITE_SERVER to a placeholder; we'll fix after getting the public URL
railway variables --service mediawiki --set MW_SITE_SERVER="https://placeholder.up.railway.app"
```

IMPORTANT about variable quoting: the `${{mariadb.RAILWAY_PRIVATE_DOMAIN}}` reference MUST be literal in the Railway variable value (Railway resolves it server-side). Single-quote in bash to prevent local shell interpolation.

Save the generated `MW_ADMIN_PASS` and `MW_SECRET_KEY` to the user's password manager — print them once to the terminal.
  </action>
  <verify>
1. `railway variables --service redis` shows a `RAILWAY_PRIVATE_DOMAIN`.
2. `railway variables --service mediawiki` lists all 18 variables above, with `MW_DB_INSTALLDB_USER=wiki` (NOT `root`) and `MW_DB_SERVER` referencing `${{mariadb.RAILWAY_PRIVATE_DOMAIN}}`.
3. Redis logs show `Ready to accept connections`.
  </verify>
  <done>Redis is live, mediawiki env is fully configured, and installer credentials are wired to skip the Phase 1 failure path.</done>
</task>

<task type="auto">
  <name>Task 2: Deploy mediawiki service, expose public URL, watch first-boot install</name>
  <files>wiki/railway.json</files>
  <action>
Deploy the mediawiki service. This triggers Taqasta's first-boot installer which runs `maintenance/install.php` against the mariadb we set up in plan 02-01:

```bash
railway up --service mediawiki --detach
```

IMMEDIATELY start tailing logs — the install runs in the first 30–90 seconds and errors are most visible here:
```bash
railway logs --service mediawiki --follow
```

Watch for these log markers (in order of expected appearance):
1. `Taqasta entrypoint` — container started
2. `Running maintenance/install.php` OR similar install script invocation
3. `Database type:` lines acknowledging mysql
4. Installer connecting to the mariadb private domain
5. `Installation complete` or `MediaWiki ... successfully installed`
6. Apache `started` / listening on port

**HISTORICAL FAILURE MODE to watch for:**
```
MariaDB error 1133: Can't find any matching row in the user table
```
If you see this, the `MW_DB_INSTALLDB_USER != MW_DB_USER` mismatch is present — the fix from Phase 1 is not applied. Abort, re-check task 1's env vars (especially `MW_DB_INSTALLDB_USER=wiki`), and retry. Do NOT attempt to patch live — destroy the mediawiki volume (`railway volume` operations in dashboard) and redeploy, since partial installs corrupt the DB state.

Once install succeeds, generate the public domain:
```bash
railway domain --service mediawiki
```
This prints/creates a `*.up.railway.app` subdomain. Record it as `$PUBLIC_URL`.

Fix the `MW_SITE_SERVER` env var to the real public URL:
```bash
railway variables --service mediawiki --set MW_SITE_SERVER="https://<PUBLIC_URL>"
railway up --service mediawiki --detach   # triggers a restart to pick up the new URL
```

Headless sanity check:
```bash
curl -sS -o /dev/null -w "%{http_code}\n" "https://<PUBLIC_URL>/wiki/Main_Page"
curl -sS "https://<PUBLIC_URL>/w/api.php?action=query&meta=siteinfo&format=json" | head -c 500
```

Expected:
- Main_Page returns `200` (or `301`→`200` if you follow redirects with `-L`)
- siteinfo JSON includes `"generator":"MediaWiki 1.43.x"`

Then check SMW is loaded:
```bash
curl -sS "https://<PUBLIC_URL>/w/api.php?action=query&meta=siteinfo&siprop=extensions&format=json" | python -c "import sys,json; d=json.load(sys.stdin); names=[e['name'] for e in d['query']['extensions']]; print('SMW' if 'Semantic MediaWiki' in names else 'MISSING'); print(names)"
```
Expected: `SMW` printed, followed by a list including `Semantic MediaWiki`, `Page Forms`, `Scribunto`, `CategoryTree`, `Arrays`, `ParserFunctions`.
  </action>
  <verify>
1. `railway logs --service mediawiki` shows successful install, NO occurrence of `error 1133`.
2. `railway domain` printed a public URL.
3. `curl https://<PUBLIC_URL>/wiki/Main_Page` returns 200 (with `-L`).
4. siteinfo JSON reports MediaWiki 1.43.x and includes Semantic MediaWiki in extensions.
5. `railway variables --service mediawiki` shows `MW_SITE_SERVER=https://<real-url>`.
  </verify>
  <done>Live public wiki serving Main_Page over HTTPS with SMW + all required extensions loaded. First-boot error 1133 did NOT occur.</done>
</task>
