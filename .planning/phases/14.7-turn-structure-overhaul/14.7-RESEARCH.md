## 1. Mapping Spec → Engine Surface

### §1 Turn Banner (TURN X / PLAYER X)
- `src/grid_tactics/server/static/game.js` — new `_showTurnBanner(turnNumber, playerIdx)` helper. Trigger from `applyStateFrame` when `frame.turn_number > prev.turn_number` OR on initial join. - No server change — turn banner is purely client-derivable from the existing `turn_number` + `active_player_idx` fields.

### §2 Phase Sequence (Start / Action / End)
- `src/grid_tactics/enums.py:11-15` — expand `TurnPhase` IntEnum. Current: `ACTION=0, REACT=1`. Add `START_OF_TURN`, `END_OF_TURN`. Keep single `REACT` and add a parallel `react_context` field (which kind of react window is open).
- `src/grid_tactics/game_state.py:85` — `phase: TurnPhase` widens; add `react_context: Optional[ReactContext] = None` and `react_return_phase: Optional[TurnPhase] = None` (where to transition back to after PASS-PASS).
- `src/grid_tactics/react_stack.py:284-450` (`resolve_react_stack`) — today monolithic: resolve LIFO → cleanup → flip active player → tick burns → fire PASSIVE → regen → auto-draw. Must split into phase-specific handlers:
  - `enter_start_of_turn(state, library)` — ticks + start-of-turn triggers, then opens START react window.
  - `close_start_react_and_enter_action(state)` — transitions START → ACTION.
  - `enter_end_of_turn(state, library)` — end-of-turn triggers, opens END react window.
  - `close_end_react_and_advance_turn(state)` — flips `active_player_idx`, increments `turn_number`, resets fatigue/discard flags, enters NEW turn's START_OF_TURN.
- `src/grid_tactics/action_resolver.py:1353-1855` (`resolve_action`) — existing phase-gate at 1495-1498 is structurally correct; extend.

### §3 Start of Turn (booleans → triggered effects → react window)
- New `fire_start_of_turn_triggers(state, library)`:
  1. Tick burn (currently `tick_status_effects`, `react_stack.py:57`).
  2. Fire `ON_START_OF_TURN`-trigger effects (new TriggerType — see §3 below).
  3. Enqueue into simultaneous-trigger queue (see §5).
- Critical redesign: today `_fire_passive_effects` resolves effects *inline with no react window*. Spec §3.1 step 4 requires a react window AFTER start-of-turn triggers and BEFORE ACTION.

### §4 Action Phase (incl. Melee dual-action & compound windows)
- `src/grid_tactics/action_resolver.py:586-652` (`_cast_magic`) — **immediate Acidic Rain/Prohibition bug site.** Today line 624-651 iterates `card_def.effects` filtered by `trigger == ON_PLAY` and calls `resolve_effect` *immediately*. Change: push **cast_mode originator** onto `react_stack`, enter REACT. ON_PLAY effects fire only when stack unwinds.
- Melee dual-action (§4.2): **already implemented** as `pending_post_move_attacker_id` (`action_resolver.py:240-249`, `1747-1781`). Phase 14.1 decision: one logical action with one react window. Spec §4.2 mandates TWO. NOT no-op — plan 14.7-08 must either split or declare a spec deviation. **Blocker flagged.**
- Compound windows (§4.3): `_deploy_minion` (`action_resolver.py:385-453`) calls `resolve_effects_for_trigger(state, ON_PLAY, ...)` inline at 449 — same bug class as `_cast_magic`. Treatment: declaration → react window A → land minion → fire Summon: triggers → react window B.

### §5 End of Turn (passives → react window → pass turn)
- Mirror of §3. Semantic shift: Fallen Paladin's `PASSIVE_HEAL` currently fires at start of owner's NEXT turn (`react_stack.py:161-162`). Spec §5.1 says passives resolve on OUTGOING turn player's side before priority passes. **Decision needed** — redefine PASSIVE as ON_END_OF_TURN of outgoing player, or keep PASSIVE semantics and add ON_END_OF_TURN as distinct new trigger.

### §6 React Windows (opening/closing, LIFO, chaining)
- `react_stack.py:178-281` (`handle_react_action`, `_play_react`) — reusable. New: close window into the CORRECT phase (not hardcoded `TurnPhase.ACTION`). Today `resolve_react_stack:415-419` hardcodes `phase=TurnPhase.ACTION`. Use `react_return_phase` field.
- `src/grid_tactics/legal_actions.py:750-846` (`_react_phase_actions`) — reusable. `_check_react_condition` (line 654-747) already has React-condition matching.

### §7 Priority + Modal (turn-player-first simultaneous triggers)
- `react_stack.py:284-450` — today simultaneous effects processed via `_fire_passive_effects` with `(row, col)` sort (line 146-149). Spec §7.2 requires turn player to pick order via modal when they have 2+ simultaneous triggers. Add:
  - `pending_trigger_queue_turn: tuple[PendingTrigger, ...]`
  - `pending_trigger_queue_other: tuple[PendingTrigger, ...]`
  - `pending_trigger_picker_idx: Optional[int]`
- `action_resolver.py` — new action types `TRIGGER_PICK(trigger_idx)` and `TRIGGER_DECLINE`. Reuse PLAY_CARD[0:N] + slot 1001 (same trick as 14.1/14.2).
- `_cleanup_dead_minions` (`action_resolver.py:1283-1316`) uses canonical sort `(active_first, instance_id)` at 1183-1188. Spec §7.2 requires turn player picks order for their own deaths. Same priority-queue/modal split.

### §7.3 Fizzle Rule
- `src/grid_tactics/effect_resolver.py` — today `resolve_effect` uses whatever `target_pos` was bound at play time. Fizzle: re-validate at resolution time.
  - `TargetType.SINGLE_TARGET` — re-check minion at `target_pos` is alive and matches faction.
  - Minion-by-instance_id references — same check.
  - Area effects (`ALL_ENEMIES`, `ALL_MINIONS`, `ADJACENT`, `ALL_ALLIES`) — already fizzle naturally (recomputed).
  - New logic applies ONLY to targeted effects.

## 4. Compound Window Mechanic (§4.3)

### Constraint
Summon with Summon: effect opens TWO sequential react windows:
- Window A — before minion lands (negate = full refund)
- Window B — after minion lands, fired by Summon: trigger (negate = only Summon: effect cancelled, minion stays)

### Current model (cannot express)
`_deploy_minion`: cost → place → resolve ON_PLAY inline → return → one react window.

### Design options

**Option A: Sub-action queue** (recommended)
`pending_sub_actions: tuple[SubAction, ...]` on GameState, discriminated union:
- `SummonDeclaration(card_numeric_id, position, ...costs...)`
- `SummonResolve(minion_id)` — pushes `summon_declaration` originator
- `SummonEffectResolve(minion_id, effect_idx)` — pushes `summon_effect` originator

Pros: identical machinery to `_cast_magic` (push → REACT → PASS-PASS → resolve). Centralized. Generalizes to "action with N sub-triggers."

Cons: new state field; another `pending_*` to serialize.

**Option B: Phase transitions mid-action**
After Window A, re-enter REACT with different `react_context`.

Pros: fewer fields.

Cons: still needs "what comes next" marker — effectively Option A without the name.

**Option C: Nested stacks** — REJECTED (breaks flat to_dict, view_filter, spell-stage animation).

### Recommendation: Option A

Plan materialization: 14.7-04 introduces `pending_sub_actions` and wires Summon. Generalizes later to ACTIVATE_ABILITY, ATTACK triggers.

### Interaction with Phase 14.1 melee chain
Phase 14.1 combines move+attack with one react window. Spec §4.2 mandates two. Options:
- (a) Split: move → react window → attack → react window. Loses Advance-Wars chain feel.
- (b) Declare 14.1 a spec deviation.
- (c) Insert mid-chain react window after move.

**Blocker — flag for user clarification.**

## 5. Priority / Fizzle Resolution (§7)

### Interaction with death cleanup
`_cleanup_dead_minions` (`action_resolver.py:1283-1316`) uses canonical sort `(active_first, instance_id)` at 1183-1188. Spec §7.2 contradicts: turn player picks order via modal.

**Replacement flow**:
1. Detect dead minions.
2. Split by `m.owner == active_side`.
3. If ≥2 turn-player deaths → open modal.
4. If 1 → auto-pick.
5. Drain turn-player queue, repeat for other-player.
6. Each picked effect opens its own react window (§7.5 step 3).

### Interaction with _fire_passive_effects
Same split. All PASSIVE/ON_END/ON_START triggers enqueue into `pending_trigger_queue_turn` or `_other`; turn player picks order if ≥2.

### Is today's `(row, col)` compatible?
Deterministic yes, spec-compliant no. Behavioral change that WILL break tests.

### Fizzle check — which effects?

Re-validate at resolve time:
- `TargetType.SINGLE_TARGET` — minion at `target_pos` alive + faction matches
- `TargetType.ADJACENT` — recompute from source minion's current position
- `TargetType.SELF_OWNER` when source is minion — source minion alive
- `TargetType.OPPONENT_PLAYER` — always valid

Self-fizzling (no change):
- `ALL_ENEMIES`, `ALL_ALLIES`, `ALL_MINIONS` — target list recomputed

Implementation: `effect_resolver.resolve_effect` gets `fizzle_check(state, effect, source_minion_id, target_pos)` — short-circuits to no-op when invalid.

### Worked example (§7.4) validation
RGB Laser vs Giant Rat requires:
- Turn-player-first ordering (new)
- Fizzle at resolve time (new)
- Simultaneous death detection (already today)

Plans 14.7-05 (priority) and 14.7-06 (fizzle) ship together or 14.7-06 strictly depends on 14.7-05.

## 7. Risks & Blockers

### Resolved by spec v2 (2026-04-18)

1. **Melee §4.1** — TWO separate react windows (one post-move, one post-attack). Supersedes Phase 14.1's combined single-window decision. Plan 14.7-08 is real, not no-op.
2. **End of turn §5** — the word "Passive" was a typo in v1. `End:` is the first-class trigger keyword. Fallen Paladin = End: (end of owner's turn). Dark Matter Battery + Emberplague Rat = End: as well. No existing PASSIVE semantics to preserve — retag all three minions to `on_end_of_turn`.
3. **Start of turn §3** — no "exhausted / summoning sickness" flag-clearing step. Start of turn: Burn ticks + any `Start:` triggered effects + react window. Clean and small.
4. **Negate-on-summon §4.2** — player loses BOTH the cost and the summon (no refund). Window A negate is harsh-by-design.
5. **Priority modal §4.3** — full-card-face modal reusing existing tutor modal pattern (`renderDeckBuilderCard`) for picking order among a player's simultaneously-triggered effects.
6. **UI reuse §6** — every react window uses the existing spell-stage animation. Existing modals (tutor-style) drive priority picking. No new UI primitives.

### Remaining unknowns

1. **Fatigue + discarded_last_turn** — flip on turn transition today (`action_resolver.py:1439-1443`). Under 3-phase model, recommend flipping at END_OF_TURN (before priority passes, so the outgoing player's flags reset before turn hand-off). Planner to confirm placement.

### Hidden coupling

1. `rl/observation.py:103` — phase is binary feature; widens. Stale-debt, not blocker.
2. `view_filter.py:23` — byte-stable entire 14.6; ReactEntry shape changes touch it. Call out.
3. `events.py:674-681` — auto-pass loop; must not infinite-loop on empty start/end phases.
4. Sandbox `to_dict`/`from_dict` — all new fields need reconstruction or old saves don't round-trip.
5. `game.js:3393-3556` spell-stage — scoped to magic casts + hand rect. Spec §11: all react windows. Extend `_spellStageSourceRect` to board tiles.

### Test invariants that break

~40-50 tests minimum across `test_react_stack.py`, `test_action_resolver.py`, `test_legal_actions.py`, `test_game_flow.py`, `test_integration.py` (~151 total in critical path).

- Immediate-effect tests: ~15-25
- PASSIVE timing tests
- Death order tests
- `test_integration.py` (10 tests likely all break)

Plan 14.7-10 is the migration.
