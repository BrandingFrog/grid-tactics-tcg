**`_cleanup_dead_minions` now enqueues ON_DEATH effects into `pending_trigger_queue_{turn,other}` (spec §7.2 turn-player-first); spec §4.3 RGB-vs-Giant-Rat worked example now works end-to-end with the pre-existing `pending_death_target` modal preserved.**

## Accomplishments

- `_cleanup_dead_minions` refactored: collects dead minions, removes them from board + minions tuple + adds to grave, then builds `PendingTrigger(trigger_kind="on_death", ...)` entries for each ON_DEATH effect and splits by owner vs active-player (turn player → `queue_turn`, other → `queue_other`). Drains via `drain_pending_trigger_queue` unless the reentrancy guard `_cleanup_skip_drain` is set.
- `_resolve_trigger_and_open_react_window` gained an on_death branch: (a) DESTROY/SINGLE_TARGET with a legal enemy target opens `pending_death_target` modal for the dying minion's owner (the pre-14.7-05 UX preserved); (b) PROMOTE with 2+ promote candidates opens the same modal with filter="friendly_promote"; (c) other on_death effects resolve inline via `resolve_effect` and open an AFTER_DEATH_EFFECT react window. Chain-reaction cleanup inside this helper runs with skip_drain=True.
- `resolve_action` pending_death_target gate: after `apply_death_target_pick` resolves the modal effect, calls `_cleanup_dead_minions` for chain-reaction deaths, then opens the AFTER_DEATH_EFFECT react window (preserving `react_return_phase` for nested drains).
- `resolve_react_stack` gained an identity-based check: if cleanup replaced state with a fresh AFTER_DEATH_EFFECT window, return state as-is instead of falling through to the turn-advance dispatch. This prevents a subtle correctness bug where a newly-opened death-trigger window would be prematurely closed.
- Existing callers (conjure resume, tutor resume, post-move resume, main resolve_action) gained two defensive checks after `_cleanup_dead_minions`: if cleanup opened a picker modal OR set `phase == REACT`, return state as-is instead of clobbering with a generic AFTER_ACTION window.
- Spec §4.3 / §7.4 worked example (RGB Lasercannon + Giant Rat simultaneous deaths with turn-player-first resolution) works end-to-end against the REAL card library. Integration test added.
- Sandbox `to_dict`/`load_dict` round-trip test validates on_death PendingTriggers survive serialization — no serializer changes needed (14.7-05's schema already handles the `trigger_kind` string field).

## Decisions Made

- **Reentrancy guard pattern (`_cleanup_skip_drain`):** Chose a module-level bool with try/finally over threading a parameter through every `_cleanup_dead_minions` caller. The parameter approach would have rippled across ~8 callers in two modules; the module-level flag confines the change to the 2 call sites inside `_resolve_trigger_and_open_react_window` where the guard is actually needed. Single-thread server model + Python frozen-state semantics make this safe (no cross-thread or cross-request contamination — the guard is always reset in finally).
- **Identity-based cleanup-opened-window detection:** `state is not _pre_cleanup_state` is the signal in `resolve_react_stack` for "cleanup actually did work and may have opened a new window". Considered alternatives: (a) queue-length snapshot (works but only for queue mutations; misses the case where queues were non-empty BEFORE cleanup), (b) react_context inequality (works but relies on pre-cleanup context being known, which it isn't always). Identity is the most robust.
- **Preserve pending_death_target modal verbatim:** The effect-level modal (DESTROY/SINGLE_TARGET pick, PROMOTE-with-2+-candidates pick) is orthogonal to the ordering-level picker modal (which on_death trigger fires next). Both legitimately coexist at different points in the flow. Spec §7.5 picker opens when 2+ deaths exist for the same owner; then within each resolving on_death, the effect's own target pick opens if applicable. The plan made this coexistence explicit and it shipped as designed.
- **Keep `pending_death_queue` / `PendingDeathWork` as defensive no-op:** Not strictly needed post-migration, but removing it would have meant touching `apply_death_target_pick`, the GameState serializer, sandbox state, and view_filter — all out of scope. Safer to leave the shape present and unused. 14.7-10 can prune if desired.
- **Chain-reaction order preservation via drain-recheck:** Rather than processing chain-death enqueue-and-drain inside `_resolve_trigger_and_open_react_window`, the skip_drain guard prevents nested drain; the outer `resolve_react_stack`'s drain-recheck hook (landed in 14.7-05) picks up the new entries once the current window closes. This keeps turn-player-first ordering correct across chains without complex bookkeeping.

## Issues Encountered

1. **`resolve_react_stack` AFTER_DEATH_EFFECT detection:** Initial implementation checked `state.phase == REACT and state.react_context == AFTER_DEATH_EFFECT` for the early-exit condition. This incorrectly matched when resolve_react_stack was CLOSING an existing AFTER_DEATH_EFFECT window (the test_game_flow full-game run got stuck in a PASS loop on iter ~385). Fixed by adding state identity check (`state is not _pre_cleanup_state`) so the early-exit only fires when cleanup ACTUALLY replaced state. Caught by a standalone game-simulation script before the test suite run; fix landed in the same commit.

2. **Chain-reaction nested drain inconsistency:** Initial implementation had `_cleanup_dead_minions` always call drain at the end. This caused a state inconsistency when called from inside `_resolve_trigger_and_open_react_window`: the nested drain would see the outer trigger STILL at queue[0] (not yet popped) + the new chain entry, treat it as 2-entries-in-one-queue, and open the picker modal — while the outer resolver was still mid-resolution. Fixed by introducing the `_cleanup_skip_drain` module-level reentrancy guard. The outer resolver's code path now: (a) resolves its effect, (b) calls cleanup with skip_drain=True (enqueues chain entries, does NOT drain), (c) pops its own entry, (d) opens its window. The outer `resolve_react_stack`'s drain-recheck picks up chain entries on window close.

3. **test_game_flow flakiness (1/10):** During the full test suite run, `tests/test_game_flow.py::TestCompleteGame::test_complete_game` fails intermittently with "LEAP target row exceeds leap distance" when run after `TestRandomGamesDoNotCrash`. Investigation: baseline (pre-14.7-05b) also has ~5% flake rate on this test (1/20). My branch has ~10-20% flake rate. Root cause is a pre-existing legal_actions/apply_move consistency bug in the LEAP code path (legal_actions computes landing_row one step past the max leap distance when enumerating 2+ enemy blockers with leap_amount=1). **Out of scope** for this plan per the scope fence (plan explicitly excludes touching legal_actions/fizzle for non-death effects). Documented for future plan (likely 14.7-10 cleanup or a separate debug task). My refactor did not introduce this bug; it may marginally increase trigger rate due to board-state changes but the root defect is in legal_actions, not my code.
