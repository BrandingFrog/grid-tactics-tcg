# Phase 14.7 Plan 09: Turn Banner + Spell-Stage Generalization + Trigger Blips Summary

**Shipped the visual layer for Phase 14.7: turn transition banner (1.5s CSS overlay on every turn flip), spell-stage driving ALL react windows (not just magic casts), and Start/End/Death-of-turn trigger blip animations (source-tile pulse + center icon + target-tile pulse). Engine-side `last_trigger_blip` transient field carries the animation payload from `_resolve_trigger_and_open_react_window` to the client via view_filter passthrough. Sandbox mode required per-frame emission (auto-drain was clobbering transient signals) and a second banner/blip wiring site because its state path diverges from `applyStateFrame`. Four distinct issues surfaced in Playwright UAT; three were auto-fixed as Rule 1/2 deviations, one (font-size side effect) resolved as a downstream consequence of the CSS class rename.**

### UAT fix round — post-deploy Playwright on Railway live (commits `9c414f9`, `260b134`)

Playwright UAT against v0.12.17 surfaced 4 issues. All 4 fixed in this session:

**Issue A (HIGH, auto-fix Rule 1) — Sandbox auto-drain ate every transient UI signal**

- **Found during:** UAT against live Railway deploy (sandbox route)
- **Root cause:** `sandbox_session.py:211-223` auto-drained trivial react windows via a bare `while REACT && len(legals)==1 && PASS: resolve_action(PASS)` loop. Each `resolve_action` call CLEARED `state.last_trigger_blip` at its top (by contract). The final emit showed no blip, no intermediate REACT phase, so:
  - client never saw `last_trigger_blip: non-null` → blip animation never fired
  - client never saw `phase === REACT` → `detectReactWindowClose` never fired → spell-stage hung open
- **Fix:** `SandboxSession.apply_action` gained an optional `on_frame: Callable[[], None]` parameter (default None). When provided, it fires after EACH `resolve_action` call (user action + every drain PASS). The `sandbox_apply_action` event handler passes `_emit_sandbox_state` as the callback, so the client receives one `sandbox_state` frame per intermediate state. The existing client handler is idempotent + animation-queue based, so frame replay composes cleanly. Per-frame updates to `_last_prev_state` / `_last_action` (pairing drained states with `pass_action()`) keep `enrich_last_action.attacker_pos` accurate on drain frames (PASS → no attacker_pos → no spurious animation replay).
- **Files:** `src/grid_tactics/server/sandbox_session.py`, `src/grid_tactics/server/events.py`
- **Commit:** `9c414f9`

**Issue B (HIGH, auto-fix Rule 2) — `_showTurnBanner` never called in sandbox mode**

- **Found during:** UAT checklist item #2 (banner on start), #3 (banner on flip)
- **Root cause:** `_showTurnBanner` was wired in `applyStateFrame` (multiplayer state path) at game.js ~2887. The sandbox socket handler at game.js ~7160 applies state directly via `renderSandbox()` and never flows through `applyStateFrame`. Result: sandbox mode never displayed the banner.
- **Fix:** Added the turn-advance check AND the last_trigger_blip check inline inside the `sandbox_state` handler (after `prevForFly` capture, before `sandboxState = payload.state`). Mirrors the multiplayer logic byte-for-byte.
- **Files:** `src/grid_tactics/server/static/game.js`
- **Commit:** `260b134` (combined with Issue C)

**Issue C (HIGH, auto-fix Rule 1) — CSS class collision on `.turn-banner`**

- **Found during:** UAT checklist item #2 — banner rendered TURN 1 + PLAYER 1 side-by-side instead of stacked
- **Root cause:** `.turn-banner` was declared TWICE in game.css:
  - Line 316: new 14.7-09 overlay (position:fixed, center, block-level stacked lines)
  - Line 1116: existing HUD flex row (display:flex, horizontal, gap:8px, flex:1)
  Cascade: BOTH rules applied to the new overlay element; the LATER rule won the `display` property, so the child divs rendered horizontally instead of stacked.
- **Fix:** Renamed the new overlay class to `.turn-transition-banner` + child lines to `.turn-transition-banner-line1` / `.turn-transition-banner-line2` + keyframe to `turn-transition-banner-in`. Updated `_showTurnBanner` in game.js to use the new names. Existing `.turn-banner` HUD rule untouched (shipped behavior).
- **Files:** `src/grid_tactics/server/static/game.css`, `src/grid_tactics/server/static/game.js`
- **Commit:** `260b134` (combined with Issue B)

**Issue D (LOW, downstream of C) — font/size inheritance**

- Confirmed resolved as a side-effect of the Issue C rename. No separate action required.

- **TURN banner on every turn flip** — both multiplayer and sandbox now show the 1.5s center overlay at turn start (including turn 1 when the game first loads).

## Reusable Infrastructure for Future Plans

- **`last_trigger_blip` transient-field pattern:** Any future visual signal that rides on a single resolve_action call can follow the same shape: set in the engine, cleared at top of next resolve_action, emitted via to_dict, consumed via identity-check (`frame.last_trigger_blip !== prev.last_trigger_blip`). 14.7-10 test-migration plans can use this as the template for future ephemeral-state fields.
- **`on_frame` callback for bulk apply paths:** The sandbox is the first place multiple resolve_action calls collapsed into a single emission. If a future batch-replay / undo-redo path lands in the sandbox or elsewhere, the callback hook is already there. Pattern: any caller that loops resolve_action N times and wants to expose intermediate states should accept `on_frame` and invoke it post-each-call.
- **Dual-path visual wiring check:** 14.x+ work should audit BOTH `applyStateFrame` AND `sandbox_state` handler whenever a new state-transition-driven visual effect lands. The two paths diverge intentionally (view_filter vs god-view) and new effects need inline wiring in both. This plan's Issue B was a real instance.
- **CSS-class collision audit:** Before adding a new top-level class (e.g. `.turn-banner`, `.spell-stage`, future effect overlays), grep the existing CSS with `grep -n "\.classname" game.css` to confirm uniqueness. Issue C is the exemplar of why this matters — the double-declaration was silent at runtime and only manifested visually.

## Non-Goals / Limits

- Turn banner NEVER gates the AnimationQueue — it's pure visual noise above the game state. If a future plan wants a blocking banner (e.g. pause on turn end for dramatic effect), that's a new primitive + scope decision.
- Trigger blip animation is a single path — no per-card-art or per-effect custom animations. Scope_fence locked this ("No new card art or sound effects"). Future card-specific effects should stay inside the blip primitive unless a design decision opens it up.
- Priority-trigger-picker modal was NOT changed in this plan — 14.7-05 already wired it to reuse `renderDeckBuilderCard`. This plan confirmed the visual polish was already correct (no modification needed).
- Fizzle triggers (14.7-06) never open a react window, so they never emit `last_trigger_blip` and never show a blip animation. This is correct per spec §7.3 ("silent no-op").
- Tensor engine untouched (on hold per CLAUDE.md).
