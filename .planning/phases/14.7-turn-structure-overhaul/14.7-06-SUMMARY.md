**Effects whose targets vanish between trigger-enqueue and resolution now silently no-op (spec §7.3), enabling the strategic pre-empt from spec §7.4 — a turn player's death effect can eliminate an enemy's future promote target before that trigger fires.**

## Accomplishments

- New `_validate_target_at_resolve_time` helper gates `resolve_effect` on target liveness per TargetType (SINGLE_TARGET / ADJACENT / SELF_OWNER). OPPONENT_PLAYER + area effects are never fizzled by the gate.

## Decisions Made

**1. on_death triggers skip the source-liveness fizzle check (target-only validation).**

The fizzle gate's source-liveness logic applies to non-`on_death` trigger kinds only. For death triggers, the source minion is DEAD by definition — that's the trigger condition. Validating source liveness would cause EVERY death trigger to fizzle (always). The fix: in `_resolve_trigger_and_open_react_window`, when `trigger.trigger_kind == "on_death"`, pass `source_minion_id=None` to `resolve_effect`. The SINGLE_TARGET target validation still runs (target must be alive).

**2. Fizzle is identity-based, not equality-based.**

`resolve_effect` returns the incoming `state` (the exact object) on fizzle. The caller can then detect fizzle via `state is prev_state` rather than deep-equality. This plays well with the frozen-dataclass engine: any state mutation goes through `replace()` which always produces a new object, so identity inequality reliably signals "something changed."

**3. Fizzled triggers pop + re-drain, no react window.**

When a trigger fizzles, the drain does NOT open an AFTER_DEATH_EFFECT / AFTER_START_TRIGGER / BEFORE_END_OF_TURN react window for it. Rationale: the effect produced no observable outcome — opening a "react?" prompt would be dead-air UX and would confuse players about what was counterable. Spec §7.5 step 3 ("react window opens while the effect's outcome is pending") implicitly requires a non-empty resolution. The drain re-enters `drain_pending_trigger_queue` directly to process the next entry (or exit cleanly if queues are empty).

**4. Kept `_fire_passive_effects` dead code in place.**

The plan suggested considering removal. Audit confirmed zero card JSONs currently use `"trigger": "passive"` (grep empty after 14.7-03 retagged Fallen Paladin / Emberplague Rat / Dark Matter Battery). Removing it now would break the 14.7-10 migration-test scaffolding. Safer to thread `source_minion_id` through defensively so future PASSIVE re-introduction doesn't need a fizzle retrofit.

**5. Source-position vs source-minion-id semantics.**

`_validate_target_at_resolve_time` takes both `source_pos` and `source_minion_id` but uses only `source_minion_id` for liveness checks. `source_pos` is passed through for symmetry with `resolve_effect`'s signature — future extensions (e.g. positional-fizzle for moved sources) can use it without an API break.

## §7.4 Worked Example Status

**Full end-to-end test, no scope reduction.** Task 2's integration test exercises the full flow:

1. P1 attacks P2's Giant Rat with RGB Lasercannon → both die.
2. `_cleanup_dead_minions` enqueues RGB's DESTROY to turn queue, Giant Rat's PROMOTE to other queue (14.7-05b's priority).
3. Drain turn queue → auto-resolve RGB → DESTROY modal opens for P1.
4. P1 picks the lone Rat — the ONE available promote candidate for Giant Rat's PROMOTE.
5. Rat destroyed; AFTER_DEATH_EFFECT react window opens.
6. PASS closes the window; drain-recheck pops Giant Rat's PROMOTE from other queue.
7. `_count_promote_candidates` returns 0 → `_apply_promote_on_death` inline no-op; cleanup runs (nothing to clean); AFTER_DEATH_EFFECT window opens for the PROMOTE branch.
8. PASS closes the window; queues empty; no new Giant Rat on the board.

Note: the PROMOTE inline-no-op path in `_apply_promote_on_death` already existed pre-14.7-06 (the function returns state unchanged when no candidates). This plan's fizzle gate does NOT gate PROMOTE (PROMOTE doesn't use `_validate_target_at_resolve_time` — it's handled by `_apply_promote_on_death`'s candidate filter). The fizzle BEHAVIOR (silent no-op) is identical either way; the code path differs. The §7.4 integration test documents this.
