<objective>
Implement spec §4.1: melee minions take up to TWO actions per turn (move + attack) with TWO separate react windows, one after the move and one after the attack. This SUPERSEDES Phase 14.1's combined-single-window decision per key_user_decisions #1.

Purpose: Spec v2 requires two independent react windows so the opponent can react separately to the move and to the attack. Today (post-14.1) the melee chain produces TWO sequential server actions (MOVE then ATTACK) but only opens ONE react window (after the ATTACK). We need to insert a react window BETWEEN the move and the attack.

Output:
- _apply_move for a melee minion with in-range post-move targets still sets pending_post_move_attacker_id (14.1 behavior preserved)
- _apply_move now ALSO enters REACT with AFTER_ACTION context and return_phase=ACTION so a react window opens post-move
- resolve_react_stack's AFTER_ACTION dispatch: if pending_post_move_attacker_id is set, return to ACTION (not END_OF_TURN) so the player can make the ATTACK or DECLINE sub-action
- After the ATTACK resolves, the standard AFTER_ACTION react window opens (second react window); after it closes, advance to END_OF_TURN normally
- DECLINE_POST_MOVE_ATTACK in ACTION phase (with pending set) clears the pending flag and advances to END_OF_TURN (no second react window because no second action occurred)
- Ranged minions unchanged (no pending_post_move state set)
- Legal actions enumerate correctly during the pending state (only ATTACK to in-range targets + DECLINE_POST_MOVE_ATTACK)
</objective>
