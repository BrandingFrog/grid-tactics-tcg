# Phase 14.7 Plan 08: Melee Two React Windows Summary

**Melee move+attack chains now open TWO independent react windows (one after the move, one after the attack) per spec v2 §4.1, superseding Phase 14.1's combined-single-window decision. The pending_post_move_attacker_id flag persists across the post-move REACT window so the caster still owes ATTACK or DECLINE when it closes; DECLINE short-circuits to end-of-turn to avoid a phantom second window. Ranged minions unchanged.**

## Accomplishments

### Task 1 — engine rewiring

- `_apply_move` for melee minions: existing 14.1 behavior (set `pending_post_move_attacker_id` when range=0 + in-range enemies) preserved byte-for-byte. The REACT transition happens at the caller site in `resolve_action`.
- `resolve_action` MOVE fall-through: when pending is set, now enters REACT with `react_context=AFTER_ACTION` / `react_return_phase=ACTION` (prior: returned inline in ACTION). This opens the post-move window.
- `resolve_action` pending_post_move gate (line ~1773): split ATTACK vs DECLINE handling.
  * ATTACK → resolves combat, clears pending, enters REACT for the second window (existing behavior; one branch).
  * DECLINE → clears pending, does NOT enter REACT, directly calls `enter_end_of_turn` (new behavior — 14.7-08 change).
- `resolve_react_stack` AFTER_ACTION dispatch: before clearing react bookkeeping + calling `enter_end_of_turn`, checks `pending_post_move_attacker_id`. If set, returns to ACTION with cleared react bookkeeping but pending intact. This is the core of the two-window semantic.
- `pending_death_target` resume path (action_resolver.py ~line 1454): pending_post_move no longer returns inline in ACTION; falls through to the REACT transition so the post-move window opens even when cleanup yielded a death modal.
- `legal_actions`: the pending-post-move enumeration branch is now gated on `phase==ACTION`. During the post-move REACT window, the reacting player gets standard REACT-phase enumeration (PASS + legal react cards), not the attack-or-decline menu that was meant for the caster.

### Task 2 — client UI + integration coverage

- `syncPendingPostMoveAttackUI` in game.js: gates the attack-pick-mode + Decline button on `gameState.phase === 0` (ACTION). During REACT the picker UI hides and reappears when the window closes.
- `TestMeleeTwoReactWindowsIntegration` in test_integration.py (+2 tests):
  * `test_full_melee_chain_with_two_react_windows`: Common Rat vs Common Rat end-to-end. Move → REACT (W1) → PASS → ACTION (pending still set) → legal_actions check (ATTACK + DECLINE only) → ATTACK → REACT (W2) → PASS → P2 turn. Turn counter +1 (not +2).
  * `test_decline_after_post_move_window_advances_turn`: Move → W1 → PASS → DECLINE → P2 turn directly. Both rats survive (no combat).
- `TestMeleeTwoReactWindows` in test_action_resolver.py (+7 tests): unit coverage for all branches (opens-window-1, opens-window-2-on-attack, skips-window-2-on-decline, ranged-unchanged, no-targets-single-window, legal_actions-during-window, legal_actions-between-windows).

## Decisions Made

1. **Phase 14.1 superseded** — spec v2 §4.1 + key_user_decisions #1 mandate two react windows. Opt-out: none. Plan 14.1's Advance-Wars-flavored single-action feel is retired.
2. **DECLINE bypasses enter_end_of_turn's REACT window** — the player chose not to take a second action, so there's no action to react against. Prior 14.1 code opened a window after DECLINE as a side effect of the shared `pending_action=action` cleanup path; 14.7-08 hard-gates DECLINE to skip the REACT transition.
3. **legal_actions phase-gated pending enum** — without this, the reacting player during the post-move window would see the attacker's ATTACK/DECLINE menu instead of REACT options. One-line fix (`and state.phase == TurnPhase.ACTION`) in the branch condition.
4. **Client picker hides during REACT** — phase gate in syncPendingPostMoveAttackUI. Users never see a Decline button they can't click.
5. **Ranged stays single-window** — plan explicitly fenced this. The `attacker_card.attack_range == 0` check in `_apply_move` is the natural gate; no other code changes needed.

## Observable Gameplay Impact

- Moving a melee minion into in-range state now opens a react window **immediately** — before the player can choose ATTACK or DECLINE. The opponent gets first pass on the move itself (e.g. play a buff-react on a targeted defender before the attack resolves).
- If the attacker chooses ATTACK, a second react window opens after the damage exchange (standard post-attack behavior; the second window was already there in 14.1, but it was the *only* window).
- If the attacker DECLINEs, the turn advances directly. No phantom REACT opportunity wasted on a non-action.
- Ranged minions move and get one react window as before.

- **14.7-10 (test migration):** The in-place migration pattern (drain W1 via single PASS before asserting between-window state) is the reference for any remaining 14.1-era tests that encode single-window assumptions. No xfails were needed.
