<objective>
Fix the Acidic-Rain-vs-Prohibition bug by deferring magic ON_PLAY effects behind a cast_mode originator at the bottom of the react stack. This is INDEPENDENTLY SHIPPABLE — closes roadmap SC4 (magic costs resolve on play, effects sit at stack bottom, Prohibition on top negates the cast) and requirements TURN-08 and TURN-11. Delivers the first user-visible win of Phase 14.7 before the larger turn-structure overhaul lands.

Purpose: The user has been bitten by this bug multiple times today. Today `_cast_magic` resolves ON_PLAY effects inline before pushing anything to the react stack, so Prohibition played in response fires AFTER damage has already resolved. The spec §4.2 and §6.3 require magic effects to defer: costs (mana, HP, destroy-ally, discard) resolve on play; ON_PLAY effects become an "originator" entry at the bottom of the react stack; the chain closes LIFO; the originator resolves last. Prohibition above the originator negates it.

Output:
- ReactEntry extended with is_originator/origin_kind/source_minion_id/effect_payload/destroyed_attack/destroyed_dm fields (all backward-compatible defaults)
- _cast_magic no longer resolves effects inline — pushes originator, enters REACT, opponent gets react window
- resolve_react_stack handles originator entries (dispatches to effect payload or resolves TUTOR/REVIVE via existing pending-state entry points)
- NEGATE above an originator cancels the cast entirely (no effects fire, card still goes to grave — mana was already spent, scorched-earth by design)
- to_dict/from_dict round-trips new fields (sandbox save/load unaffected)
- view_filter preserves new fields so the UI can render "pending magic cast" on the stack
- Tests updated: Acidic Rain + Prohibition → zero damage; Acidic Rain + PASS → damage resolves; multi-purpose react arm still works
</objective>
