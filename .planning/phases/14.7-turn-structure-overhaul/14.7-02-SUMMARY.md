## How events.py auto-pass loop was extended

Original loop (pre-14.7-02):

```python
while not session.state.is_game_over:
    next_actions = legal_actions(session.state, session.library)
    if len(next_actions) > 0:
        break
    session.state = resolve_action(session.state, pass_action(), session.library)
```

Extended loop (14.7-02):

```python
_auto_advance_counter = 0
_AUTO_ADVANCE_MAX = 50
while not session.state.is_game_over:
    _auto_advance_counter += 1
    if _auto_advance_counter > _AUTO_ADVANCE_MAX:
        raise RuntimeError("Phase auto-advance loop exceeded safety counter …")
    if session.state.phase == _TurnPhase.START_OF_TURN:
        session.state = _enter_start_of_turn(session.state, session.library)
        continue
    if session.state.phase == _TurnPhase.END_OF_TURN:
        session.state = _enter_end_of_turn(session.state, session.library)
        continue
    next_actions = legal_actions(session.state, session.library)
    if len(next_actions) > 0:
        break
    session.state = resolve_action(session.state, pass_action(), session.library)
```

**Why**: `resolve_action` raises `ValueError(f"Cannot resolve action in phase {state.phase.name}, expected ACTION")` for START/END phases. If we routed START/END via `pass_action()` through `resolve_action`, it would crash. Instead we detect the phase at the loop boundary and call the helper directly.

**Safety counter**: 50 iterations is generous. Today's 14.7-02 never runs more than 2 start/end passes per turn (START_OF_TURN placeholder flips to ACTION immediately; END_OF_TURN runs tail + enters next START which flips to ACTION — at most 3 loop iterations). 14.7-03 will add react windows which increase the count slightly but 50 remains a huge safety margin. The loop raises a `RuntimeError` (500 surfaced to the user) rather than wedging the server.

## Decisions Made

- **TurnPhase append-only (not IntEnum reorder)**: Adding START_OF_TURN and END_OF_TURN at the end preserves ACTION=0 / REACT=1 binary encodings for any numpy/tensor code. The tensor engine is on hold (per CLAUDE.md) but stability here costs nothing.
- **ReactContext as a separate enum**: Phase and context are orthogonal. A REACT window can open AFTER_ACTION, AFTER_START_TRIGGER, AFTER_SUMMON_DECLARATION, etc. Forcing them onto TurnPhase would conflate concerns.
- **react_return_phase=None defaults to ACTION**: Preserves strict backward compat. No pre-14.7-02 caller sets the field; all of them get the legacy after-action turn-advance path. Verified via `test_react_window_returns_to_action_by_default`.
- **Placeholder helpers in 14.7-02**: The plan explicitly fences start/end trigger firing into 14.7-03. Keeping 14.7-02 as pure state-machine wiring (no behavior change) makes the diff reviewable and the blast radius tiny. The new helpers exist but do nothing user-visible — they're there so 14.7-03 can hook into them.
- **Dedup via helper, not inline refactor**: The pending_death_target resume path previously duplicated the turn-advance tail inline. Rather than leave the duplication and add scope for 14.7-03, Step 8 of the plan explicitly deduplicates now. One source of truth for "outgoing player's end-of-turn" logic.
