- Extended `ReactCondition` IntEnum with three append-only values: `OPPONENT_SUMMONS_MINION=15`, `OPPONENT_START_OF_TURN=16`, `OPPONENT_END_OF_TURN=17`. Card JSON loader picks them up automatically via reflective `enum_cls[value.upper()]` lookup — no allowlist edit required.

Legacy conditions (OPPONENT_PLAYS_MAGIC, OPPONENT_PLAYS_MINION, element conditions, etc.) keep their pre-14.7 semantics under `AFTER_ACTION` / `None`, with the 14.7-01 deferred-magic originator flow explicitly handled.
- **Prohibition (OPPONENT_PLAYS_MAGIC) now correctly filtered OUT** of `legal_actions` during AFTER_SUMMON_DECLARATION / AFTER_SUMMON_EFFECT — the previous behavior would have included Prohibition as legal during a minion summon even though the card text says "magic only." This fulfills must-have truth #1 of the plan.
- Per-context dispatch table now documented in the function docstring for easy audit.

## Decisions Made

1. **OPPONENT_SUMMONS_MINION matches BOTH summon windows (A and B), not just declaration.** The plan's sketch said "matches OPPONENT_SUMMONS_MINION (and legacy OPPONENT_PLAYS_MINION still matches for back-compat)" under AFTER_SUMMON_DECLARATION, and "matches OPPONENT_SUMMONS_MINION" under AFTER_SUMMON_EFFECT — so both. A react card tagged OPPONENT_SUMMONS_MINION can now be played in either window, leaving it to card designers to scope via other means. This keeps the condition semantically simple ("the opponent summoned a minion") rather than introducing two variants.
2. **Context-tagged conditions are evaluated BEFORE the counter-react branch.** This means a counter-react inside an AFTER_SUMMON_DECLARATION window still has OPPONENT_SUMMONS_MINION as a legal condition — the window's "why" doesn't change just because a react got stacked on top. This matters for 4-player future scenarios or any deeper counter-react chain; for the 2-player game today it's a harmless invariant.
3. **Prohibition behavior reverted to spec intent.** Pre-14.7-07, the counter-react branch matched `top_card.card_type == REACT` for OPPONENT_PLAYS_MAGIC — so Prohibition could technically counter-react itself. Under the new dispatch, a counter-react in an AFTER_SUMMON_DECLARATION window still won't match OPPONENT_PLAYS_MAGIC (because the context-tagged block returns early for new conditions, and the counter-react branch returns False for OPPONENT_PLAYS_MAGIC when the top card is a REACT type only during summon windows — actually wait, the counter-react branch DOES still return True for REACT-type tops during AFTER_ACTION). Kept the AFTER_ACTION counter-react fallback (Prohibition-vs-Prohibition in a magic chain still works). The BEHAVIORAL CHANGE is only: Prohibition is no longer legal as a FIRST react in a summon window.
4. **No card JSON retagged.** Per plan scope fence. The new conditions are forward-compat for future cards (e.g., a "Watch the Throne" card that fires when opponent's Start-of-turn triggers go off). Existing cards retain their current react_condition values.

## Issues Encountered

**Subtlety: pre-existing integration tests play Prohibition during summon windows via direct resolve_action.**

The existing `test_prohibition_on_window_a_negates_full_summon_and_tutor` and `test_prohibition_on_window_b_preserves_minion_cancels_tutor` (added in 14.7-04) call `resolve_action(state, play_react_action(...))` directly, bypassing `legal_actions` filtering. They continue to pass because `_play_react` (in react_stack.py) does NOT re-validate the condition — it just pushes and resolves.

**Not a bug — the tests were exercising what the CODE can do, not what players can do.** With my change, players can no longer pick Prohibition in a summon Window A (it's filtered out of legal_actions). Those tests remain green as a stale-but-harmless check that if a server somehow let Prohibition through, the negate effect would still resolve. No action taken.

The new integration tests add an explicit assertion that Prohibition is NOT in legal_actions during AFTER_SUMMON_DECLARATION, covering the behavior change cleanly.
