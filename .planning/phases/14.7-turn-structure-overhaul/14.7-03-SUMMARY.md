## Decisions Made

1. **Shortcut-when-no-triggers policy**: enter_start_of_turn / enter_end_of_turn open a REACT window only when the active player has at least one minion carrying the relevant trigger. Rationale: avoids breaking ~40 direct resolve_react_stack callers in tests that expect immediate turn-advance on minion-less scenarios. Also makes dead-air turns snappy in the live game. 14.7-07 will extend the gate to also check for opponent react cards matching OPPONENT_START_OF_TURN / OPPONENT_END_OF_TURN when those react conditions land.

2. **Data-rename + bridge**: Retagged 6 Summon: minions' effects on_play -> on_summon in JSON, then made `_deploy_minion` fire BOTH ON_PLAY and ON_SUMMON trigger dispatches. Preserves full functionality through the rename. The explicit scope fence says "Do NOT touch _deploy_minion compound-window logic (that's 14.7-04)" — the bridge is a *separate* one-line addition, not the compound-window logic. 14.7-04 will replace the bridge with proper two-window dispatch (declaration react -> fire ON_SUMMON -> post-effect react).

3. **Kept magic cards at trigger=on_play**: 14.7-01 made magic casts deferred-resolve via the cast_mode originator pattern (originator pushed on react-stack bottom; ON_PLAY effects resolve at react-stack-resolve time, not at cast time). Renaming their trigger to "on_cast" would be cosmetic churn without semantic effect. Same for react cards (NEGATE resolves at react-stack-resolve time regardless of trigger value).

4. **Kept surgefed_sparkbot at trigger=on_play**: its entire on-play behavior lives in `react_effect` (DEPLOY_SELF), which resolves through a separate react-stack code path that doesn't filter by TriggerType. The `effects=[]` list is empty, so the ON_PLAY trigger dispatch no-ops at deploy anyway.

5. **Kept _fire_passive_effects as LEGACY no-op**: After retagging, no card JSON uses `trigger="passive"`. Function retained for forward-compat in case a future card re-introduces PASSIVE (e.g. a true always-on effect). Removed from the _close_end_of_turn_and_flip call path; nothing else calls it.

6. **Moved tick_status_effects from _close_end_of_turn_and_flip to enter_start_of_turn**: The plan explicitly redistributes this. Burn tick timing is preserved (fires for new active player's burning minions at their turn start) but now lives inside the Start-of-Turn phase boundary. Ordering: burns tick first, THEN Start triggers fire. Plan note says this is a judgment call; kept mana regen / auto-draw in _close_end_of_turn_and_flip (they sit between the flip and the new active player's start).

### Scope Clarifications Applied

**Shortcut-when-no-triggers policy**: The plan text describes a fully-open-always react-window model: "open a REACT window with ReactContext.AFTER_START_TRIGGER / BEFORE_END_OF_TURN." Taken literally, every turn would open two empty react windows that then close via the server's auto-PASS loop. The user's upstream context explicitly clarified: "When triggers exist, open REACT window... When no triggers, auto-advance (existing placeholder behavior stays)." I implemented the user's clarification: the window opens only if the active player has minions carrying the relevant trigger. Without this, ~40 direct resolve_react_stack test callers would stall in REACT (they go through the tests layer, not the server auto-PASS loop). 14.7-07 will open the gate further to account for opponent-side react conditions.
