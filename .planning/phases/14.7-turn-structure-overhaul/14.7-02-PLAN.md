<objective>
Expand the turn state machine from 2 phases (ACTION, REACT) to 4 phases (START_OF_TURN, ACTION, END_OF_TURN, REACT). Introduce ReactContext so REACT knows WHICH phase it's returning to. Split the monolithic resolve_react_stack into phase-specific handlers. Update legal_actions + events.py so the server auto-advances through Start and End phases (they become meaningful in 14.7-03 when triggered effects and start/end react windows land).

Purpose: The spec §2 mandates three ordered phases per turn. Today's engine collapses Start (burn tick + passive) and End (nothing — passives fire at next owner's start) into the turn-flip tail of resolve_react_stack. This plan is infrastructure: it creates the phases as first-class TurnPhase values, wires the state machine, and preserves all existing behavior (14.7-03 will populate Start/End with the actual trigger firings). Keeping it scope-fenced to the state machine (not the triggered-effect pipeline) keeps the blast radius controllable.

Output:
- TurnPhase extended: START_OF_TURN=2, END_OF_TURN=3 (append-only, IntEnum-safe)
- ReactContext IntEnum: AFTER_START_TRIGGER, AFTER_ACTION, AFTER_SUMMON_DECLARATION, AFTER_SUMMON_EFFECT, AFTER_DEATH_EFFECT, BEFORE_END_OF_TURN (6 values)
- GameState fields: react_context: Optional[ReactContext], react_return_phase: Optional[TurnPhase] (both default None)
- react_stack.py split into 4 phase transition helpers
- resolve_react_stack uses react_return_phase instead of hardcoding TurnPhase.ACTION
- legal_actions returns [] during START_OF_TURN/END_OF_TURN (auto-advanced by server)
- events.py auto-pass loop handles Start/End gracefully (no infinite loop)
- to_dict/from_dict preserves new fields
</objective>
