## Decisions Made

1. **Use ReactEntry originator pattern, not pending_sub_actions.** The plan header already documented this deviation from RESEARCH §4 option A. One queue is simpler, re-uses the 14.7-01 serializer, and composes naturally with the LIFO negate logic (a summon_declaration at the bottom of the stack is negated exactly like a magic_cast originator).

2. **Window A negate is HARSH.** Mana, discard, and destroy-ally costs are all forfeit on negate. Per spec §4.2 and key_user_decisions #2. `test_summon_window_a_negate_loses_cost_and_summon` pins this.

3. **Gargoyle's two ON_SUMMON effects share ONE Window B.** The effect_payload tuple carries both indices; the opponent reacts to the set once. 14.7-05 may layer priority-picker + modal on top for simultaneous-trigger edge cases, but this plan's compound window is single-fire-per-window.

4. **Minion ON_PLAY triggers are orphaned.** Only ON_SUMMON fires through the compound pipeline. No card JSON currently uses trigger=on_play on a minion (grep-verified), and future minions that want on-deploy effects should tag ON_SUMMON. `test_on_play_effect_triggers_after_deploy` was updated to assert the new (non-firing) behavior.

5. **Stack-identity snapshot over sentinel fields.** The compound-window hand-off compares `state.react_stack is not _pre_resolution_stack` rather than adding a new GameState field. Locally scoped, doesn't touch serialization, and the check fires iff a helper called `replace(state, react_stack=...)` during resolution — precisely the semantic we want.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Fixed resolve_action overwriting _deploy_minion's react_context**

- **Found during:** Task 1 (first test run of test_summon_declaration_opens_window_a)
- **Issue:** The outer `resolve_action` dispatcher unconditionally sets `react_context=AFTER_ACTION` at the end of an ACTION-phase action, but `_deploy_minion` (and `_cast_magic`) already set their specific context (AFTER_SUMMON_DECLARATION / AFTER_ACTION respectively) and transitioned to REACT inline. The outer block was clobbering the summon context to AFTER_ACTION.
- **Fix:** Added an early-return at the top of the outer REACT-transition block: if `state.phase == TurnPhase.REACT` already (meaning the handler set up its own react window inline), skip the generic AFTER_ACTION transition.
- **Files modified:** `src/grid_tactics/action_resolver.py`
- **Verification:** `test_summon_declaration_opens_window_a` + magic-cast tests all green. _cast_magic was already emitting AFTER_ACTION so no behavior change there.
- **Committed in:** 8b093af (Task 1 commit)

**2. [Rule 1 — Bug] Fixed latent 14.7-01 pending-modal-during-react bug exposed by Diodebot Window B**

- **Found during:** Task 1 (full-suite test run showed "Invalid action type TUTOR_SELECT during REACT phase")
- **Issue:** When a magic_cast or summon_effect originator fires a TUTOR or REVIVE effect during stack resolution, it sets `pending_tutor` / `pending_revive` without changing phase. After the LIFO loop, resolve_react_stack's dispatch advanced the turn to P2 — but pending_tutor is still for P1. The pending_tutor gate in resolve_action is phase-agnostic, so magic_cast tutors worked by coincidence (the gate caught the wrong-player action). However, Window B kept phase=REACT, so resolve_action routed TUTOR_SELECT to handle_react_action which rejected it.
- **Fix:** Added a post-LIFO-loop check in resolve_react_stack: if pending_tutor_player_idx or pending_revive_player_idx is set, close the react window (phase=ACTION, react_stack=(), bookkeeping cleared) WITHOUT advancing the turn. The modal owner is then the correct decision-maker via active_player_idx.
- **Files modified:** `src/grid_tactics/react_stack.py`
- **Verification:** 30-seed random-games regression passes; test_complete_game in isolation passes; test_diodebot_tutor_through_compound_windows end-to-end passes. Side benefit: Ratmobile-style magic tutors no longer also advance the turn before the caster picks (was an existing 14.7-01 pre-bug).
- **Committed in:** 8b093af (Task 1 commit — same commit since both fixes were needed for the unit tests to pass)

**3. [Rule 1 — Bug] Updated 6 deploy tests + 1 on_play test in test_action_resolver.py**

- **Found during:** Task 1 (initial regression run)
- **Issue:** 6 existing deploy tests asserted on `len(new_state.minions) == 1` immediately after `resolve_action(PLAY_CARD)`. With the new compound-window flow, the minion lands only after Window A drains (PASS-PASS). These tests needed an explicit `resolve_action(PASS)` call to drain Window A before asserting.
- **Fix:** Added `resolve_action(PASS)` to each affected test. `test_on_play_effect_triggers_after_deploy` additionally updated to assert that minion ON_PLAY triggers are now orphaned (only ON_SUMMON fires through the compound pipeline), since no real card uses trigger=on_play on a minion.
- **Files modified:** `tests/test_action_resolver.py`
- **Verification:** All 100 tests in test_action_resolver.py + test_react_stack.py pass.
- **Committed in:** 8b093af (Task 1 commit)
