# Phase 14.7 Plan 01: Deferred Magic Resolution Summary

**Magic ON_PLAY effects now defer behind a cast_mode originator on the react stack; Prohibition on top of an Acidic Rain cast cancels the burn entirely (closes SC4, TURN-08, TURN-11)**

## Performance

## Accomplishments

- **Fixed the Acidic-Rain-vs-Prohibition bug** that has been biting the user repeatedly. Before: `_cast_magic` resolved ON_PLAY effects INLINE before pushing anything to the react stack — Prohibition played in response fired AFTER damage had already resolved, contradicting every other card game's ordering intuition. After: costs (mana/HP/destroy-ally/discard) resolve on play; ON_PLAY effects sit at the BOTTOM of the react stack as a cast_mode originator; Prohibition on top cancels the cast per spec §4.2 / §6.3.
- **Extended ReactEntry** with six additive fields (`is_originator`, `origin_kind`, `source_minion_id`, `effect_payload`, `destroyed_attack`, `destroyed_dm`). All default-valued, fully backward-compatible with existing react-entry construction in `_play_react`.
- **Added a branch to `resolve_react_stack`** that dispatches magic_cast originators through the same TUTOR / REVIVE / generic-resolve_effect routing the old inline `_cast_magic` path used — preserves destroy-ally scale_with, target_pos capture, and caster-owner attribution.
- **Round-tripped originator fields through `to_dict` / `from_dict`** with a dedicated test that exercises JSON serialization (`json.dumps(d); json.loads(...); GameState.from_dict(...)`) across both an originator entry AND a legacy react entry on the same stack. Sandbox save/load of a mid-cast state now survives verbatim.
- **Added three integration tests** in `TestAcidicRainProhibition` + one in `TestMultiPurposeMagicReactArm` that cover the full user-facing scenarios: (1) cast defers the burn, (2) PASS resolves the originator, (3) Prohibition on top negates the cast, (4) multi-purpose card played as a REACT still fires only its react_effect.

## Decisions Made

- **ReactEntry field types** — `is_originator: bool`, `origin_kind: Optional[str]` (str for extensibility: "magic_cast" today, "summon_declaration" / "start_of_turn" later), `source_minion_id: Optional[int]` (reserved for origin-kinds that reference a specific minion), `effect_payload: Optional[tuple]` (tuple-of-tuples so the frozen dataclass stays hashable-friendly), `destroyed_attack: int = 0` + `destroyed_dm: int = 0` (plain ints, zero defaults — only set non-zero when the cast had a destroy-ally cost).
- **Chose tuple-of-tuples over dict for `effect_payload`** — plan §2 research described the payload as a dict shape, but the existing ReactEntry is `@dataclass(frozen=True, slots=True)` with a tuple in `target_pos`. Keeping the payload as tuple maintains immutability and equality semantics (two originators with identical payloads compare equal).
- **`_cast_magic` sets `phase=REACT` and `react_player_idx=1-active` internally** — the subsequent unconditional phase transition at the end of `resolve_action` (lines 1849-1853) does the same thing, so it's effectively a no-op there. Setting it inside `_cast_magic` makes the function's contract self-contained and future-proofs against someone moving the final transition.
- **NEGATE handling required zero code change** — the existing `negated_indices.add(i + 1)` pattern at `react_stack.py:324` already cancels "the next entry in LIFO iteration," which correctly describes "the entry directly below me on the stack." When Prohibition sits atop a magic_cast originator, Prohibition appears at LIFO index 0, the originator at LIFO index 1, and `add(1)` negates the originator exactly as intended. Plan verified this assertion empirically via `test_acidic_rain_negated_by_prohibition`.
- **view_filter preserves new fields via `copy.deepcopy`** — confirmed by reading view_filter.py. `filter_state_for_player` receives a `state_dict` that was already produced by `GameState.to_dict()` (caller's job), deep-copies it, and only mutates `players[opponent].hand` + `deck` + strips `seed`. The react_stack dicts pass through untouched, so originator fields flow to the client transparently. (Recorded this finding per plan Task 1 Step 6.)
- **Test-migration scope kept minimal** — only `test_magic_card_resolves_effect_and_discards` was rewritten (it asserted inline damage resolution on exactly the path this plan changed). Full test migration for the 3-phase turn structure is tracked by plan 14.7-10. No `@pytest.mark.xfail` marks were added in this plan — nothing else in the suite depended on inline magic resolution.
- **`_play_to_completion` iteration cap bumped 500→1500** — deferred magic resolution doubles the loop iteration count for any game with multiple magic plays (each cast needs its resolve-via-pass). The bump also incidentally repaired the pre-existing `test_complete_game` failure that was sitting at the 500 cap.

## Issues Encountered

Fixed by collapsing to one PASS and updating the test docstring.
- **Flaky "multi-purpose react arm" test on first draft** — originally used `to_the_ratmobile` as the originator magic with P1's deck holding a rat, expecting the tutor to be a no-op. In practice `_enter_pending_tutor` found the rat and set `pending_tutor_player_idx` after the stack resolved. The test's final PASS then raised `ValueError("Pending tutor: must TUTOR_SELECT or DECLINE_TUTOR")`. Fixed by emptying P1's deck — `_enter_pending_tutor` finds zero matches and exits without setting pending state, so the originator resolves cleanly.
