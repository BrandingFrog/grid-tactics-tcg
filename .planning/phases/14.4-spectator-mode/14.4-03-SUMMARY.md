# Phase 14.4 Plan 03: Events Layer Spectator Wiring Summary

**One-liner:** events.py now speaks spectator end-to-end — join handler, action gating, state/game_start/game_over fanout, chat, and disconnect cleanup — without altering the existing player code paths.

## What Was Built

**1. spectate_room handler** (new, inside register_events)
- Validates display_name + room_code
- Calls `room_manager.join_as_spectator(room_code, display_name, sid, god_mode)`
- `sio_join_room(room_code)` so room-wide broadcasts (chat) reach the spectator
- Acks with `spectator_joined { room_code, session_token, god_mode }`
- If a GameSession already exists for the room, immediately emits a synthetic `game_start` with filtered state + card_defs so a mid-game joiner is fully bootstrapped

**2. submit_action gate**
- Immediately after `get_token_by_sid`, returns `error: "Spectators cannot submit actions"` if `room_manager.get_role(token) == 'spectator'`
- Placed above the room/session/turn checks so spectators get a clear error rather than a misleading "Not in a game"

**3. State fanout**
- New `_fanout_state_to_spectators(session, state, base_state_dict, resolved_action, event_name)` helper
- Wired into `_emit_state_to_players` and `_emit_game_over` after their existing per-player loops
- New `_fanout_game_start_to_spectators(session, base_state_dict, card_defs)` helper wired into `handle_ready` (initial start) and `handle_request_rematch` (rematch start)
- Each spectator gets their own filtered state via `filter_state_for_spectator(state_dict, god_mode=slot.god_mode, perspective_idx=0)`
- Non-god spectators additionally get `enrich_pending_tutor_for_viewer(... idx=0 ...)` so opponent tutoring stays hidden
- Spectator payloads include `is_spectator: True` for client branching

**4. Chat for spectators**
- `handle_chat_message` now resolves sender name via `get_role` → `get_spectator` for spectator tokens, falling back to the existing player/waiting-room lookup for player tokens
- Existing `emit("chat_message", ..., to=room_code)` already broadcasts to everyone in the SocketIO room — combined with the new `sio_join_room(room_code)` in spectate_room, spectators both send and receive chat

**5. Disconnect cleanup** (new handler)
- New `@socketio.on("disconnect")` handler
- For spectator tokens: calls `room_manager.remove_spectator(token)` and returns
- Player disconnect is intentionally a no-op (Phase 15 reconnection territory) — documented in the handler docstring

## Deviations from Plan

**1. Plan suggested refactoring `_emit_state_to_players` to take `room_manager` as a parameter.**
Instead used the existing module-level `_room_manager` global that `register_events` already populates. Same effect, zero call-site churn, consistent with how the rest of the file accesses room_manager.

**2. Plan said "the existing disconnect handler" — there was no existing disconnect handler in events.py.**
Wave 1 already flagged this in its summary. Added a brand-new minimal disconnect handler scoped to spectator cleanup only. Player disconnect remains a no-op until Phase 15 reconnection, which is the correct posture (cleaning a player sid here would silently break Phase 15's reconnection contract before it lands).

**3. Plan only mentioned state_update fanout. Added game_start fanout too.**
A spectator who joins the lobby BEFORE start_game would otherwise miss the initial game_start frame and never bootstrap card_defs/opponent_name. Wired `_fanout_game_start_to_spectators` into both `handle_ready` and `handle_request_rematch`. The spectate_room handler also synthesizes a game_start for mid-game join.

**4. Spectator payload includes legal_actions=[] and your_player_idx=0.**
Plan didn't specify the wire shape. Mirroring the player schema (with `is_spectator: True` discriminator) lets the frontend reuse its existing state_update reducer with a single branch instead of two parallel decoders.

No deviation rules (Rule 1-4) triggered.
