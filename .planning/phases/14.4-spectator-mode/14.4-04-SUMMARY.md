# Phase 14.4 Plan 04: Spectator Frontend UI Summary

Frontend spectator layer: lobby entry (Spectate button + God Mode checkbox), client flag, interaction suppression, dual-hand god view, and SPECTATING badge. After this plan the full spectator feature is reachable from the browser.

### Task 1 тАФ Lobby UI + client flag (commit 701ff25)
- game.html: Added `#btn-spectate-room` button next to Join Room, and `#god-mode-checkbox` in a `.god-mode-label` under the room-code row. Added `#spectating-badge` span to the game room bar.
- game.js:
  - New module-level flags: `isSpectator`, `spectatorGodMode`

### Task 2 тАФ Gating + dual-hand + badge (commit b63d2ec)
- game.js action suppression (early `if (isSpectator) return;` at the top of each):
  - `submitAction(actionData)`
  - `onBoardCellClick(row, col)`
  - `onBoardMinionClick(minion)`
  - `onHandCardClick(handIdx)`
  - `renderActionBar()` тАФ clears the bar + hides the how-to-play hint and returns
- `onGameStart` тАФ if `data.is_spectator` OR `data.state.is_spectator`, set `isSpectator=true` and pick up `spectator_god_mode`. This matches the server's `handle_spectate_room` flow which emits `game_start` with `is_spectator:true` when joining a running game.
- `_applyStateFrameImmediate` тАФ re-syncs `isSpectator` + `spectatorGodMode` from every frame (`frame.is_spectator`, `frame.spectator_god_mode`). No drift possible.
- `renderHand` тАФ refactored to an inner `appendHand(playerObj, label)` helper. God-mode spectators get BOTH hands rendered top-to-bottom with `.spectator-hand-label` dividers ("Player N hand"). Non-god keeps the single-hand legacy render.
- `renderOpponentInfo` тАФ hand-count fallback: `oppPlayer.hand_count ?? oppPlayer.hand?.length ?? 0`. God-mode spectators get the full raw state dict (no `hand_count` field), so fallback is required.
- `renderRoomBar` тАФ toggles `#spectating-badge.visible` and flips label between `SPECTATING` and `SPECTATING (GOD MODE)` based on `isSpectator` + `spectatorGodMode`.
- game.css тАФ added `#spectating-badge` (gold chip, display:none тЖТ .visible), `.god-mode-label` (block label w/ checkbox margin), `.spectator-hand-label` (gold uppercase divider inside hand container).

## Deviations from Plan

### Auto-fixed issues

**[Rule 2 тАФ Missing Critical] Opponent hand-count fallback for god-mode full-state dict**
- Plan did not specify how `renderOpponentInfo` should render when `oppPlayer.hand_count` is absent. God-mode spectators receive a `copy.deepcopy(state_dict)` via `filter_state_for_spectator`, which means the player dicts use the raw-state schema (no `hand_count` тАФ the field is added by `filter_state_for_player` for non-self players). Without the fallback, the opponent HAND stat would display `undefined`.
- Fix: `oppPlayer.hand_count ?? oppPlayer.hand?.length ?? 0`
- Commit: b63d2ec

**[Rule 3 тАФ Blocking] Spectator mode source-of-truth resolution**
- Plan instructed a separate `renderSpectatorView(state)` branch in `onStateUpdate`. Rejected in favor of routing through the existing `applyStateFrame тЖТ renderGame` pipeline because:
  1. Phase 14.3's AnimationQueue is the single point of state application тАФ a spectator bypass would desync animations (spectators should still see summon/move/attack/popups)
  2. The existing render functions (renderBoard, renderOpponentInfo, renderSelfInfo, renderHand) already handle perspective_idx=0 correctly for spectators
  3. Less duplicated code, same behavior
- Instead: gated the action seams and extended renderHand for dual-hand god view. `onStateUpdate` and `applyStateFrame` are unchanged except for flag sync.
- This is a cleaner pattern that preserves Phase 14.3's animation contract and gets dual-hand rendering for free.

## Decisions Made

- **Spectator flag propagation:** `isSpectator` + `spectatorGodMode` are re-synced from EVERY state frame (in `_applyStateFrameImmediate`) in addition to `onGameStart` + `onSpectatorJoined`. This guarantees reconnection and late-join both flip the client into spectator mode automatically with no drift.
- **No separate spectator renderer:** Reuse the existing `renderGame тЖТ renderBoard / renderHand / renderOpponentInfo` pipeline. Plan's proposed `renderSpectatorView` would have fragmented the render tree and broken Phase 14.3 animation integration.
- **Dual-hand layout:** Both hands stack vertically in the existing `#hand-container` with gold labeled dividers. Top hand = opponent seat, bottom = perspective-0 seat. No new DOM container, no separate CSS grid тАФ just a label + existing `renderHandCard` for each card.
- **isMyTurn passthrough:** `renderHandCard(..., isMyTurn && !isSpectator)` тАФ spectator cards are always rendered dimmed/unplayable regardless of whose turn it is. Prevents "my turn" glow from appearing on spectator hands.
- **Transition posture:** `onSpectatorJoined` does NOT call `showScreen('screen-game')`. The server emits `game_start` immediately if a game is running, which already does the transition. If no game is running, the spectator sits in the lobby with a status message until `game_start` arrives. This matches how player joins work.
