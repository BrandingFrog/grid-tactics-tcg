# Phase 14.4 Plan 01: Spectator Data Model + Join API Summary

**One-liner:** RoomManager now tracks spectators by room_code in a manager-level dict so they survive the WaitingRoom→GameSession transition without touching either dataclass, with role-aware token lookups via `_token_role`.

## What Was Built

- `SpectatorSlot` dataclass: `token`, `name`, `sid`, `god_mode`.
- `RoomManager._room_spectators: dict[room_code, dict[token, SpectatorSlot]]` — chosen over storing on WaitingRoom because `start_game` pops the WaitingRoom, and storing on GameSession would force a second mutation site. Manager-level keying is the least invasive option.
- `RoomManager._token_role: dict[token, 'player'|'spectator']` populated in `create_room`, `join_room`, and `join_as_spectator`.
- New public API:
  - `join_as_spectator(room_code, display_name, sid, god_mode)` — works against waiting rooms OR active games; raises `ValueError("Room not found")` if neither exists.
  - `get_role(token) -> str | None`
  - `get_spectator_tokens(room_code) -> list[str]`
  - `get_spectator(token) -> SpectatorSlot | None`
  - `remove_spectator(token) -> str | None` — cleans `_token_to_room`, `_token_role`, `_sid_to_token`, and the per-room bucket; collapses empty buckets.

## Verification

Manual smoke (extended beyond the plan's one-liner):

- Spectator joins waiting room → role='spectator', appears in `get_spectator_tokens`.
- Multiple spectators per room.
- Removal cleans all four indexes.
- Spectator joins **after** `start_game` (room is now a `GameSession`) → still resolved, still tracked, `get_spectator_tokens` works.
- Existing player flow (`create_room`/`join_room`/`set_ready`/`start_game`) unchanged.

No existing tests for room_manager existed to regress; the verification snippet covers the new surface.

## Decisions Made

- **Spectator storage location:** RoomManager-level `_room_spectators` (not on WaitingRoom or GameSession). Survives `start_game` pop without touching either dataclass. Documented in the module docstring.
- **god_mode is per-spectator state**, not per-room — allows mixing audit/dev spectators with regular viewers in the same room later.
- **Spectator removal does not exist in connect/disconnect path yet** — Plan 01 explicitly scopes events.py out. Wiring `remove_spectator` into the disconnect handler is for a later wave.
- **No SpectatorSlot field added to GameSession** — kept the dataclass clean; manager-level keying is sufficient.

## Deviations from Plan

1. The plan offered two storage options ("WaitingRoom field that survives" vs "manager-level dict"). Chose manager-level dict and documented why in the module docstring (option 2 of the plan's bullet 7).
