# Phase 14.2 Plan 01: Python Engine Pending Tutor State Summary

**One-liner:** Replaced silent auto-tutor in the Python engine with a pending_tutor state machine mirroring 14.1's pending-post-move-attack pattern, and extended `tutor_target` to accept selector dicts (tribe / element / card_type) in addition to the legacy card_id string shorthand.

## Encoding Decision (locked in)

Action-space layout `[0:1262]` is **PRESERVED** — no resize.

- New `ActionType.TUTOR_SELECT = 9` and `ActionType.DECLINE_TUTOR = 10` exist in Python only (semantic).
- `TUTOR_SELECT` reuses the **PLAY_CARD[0:250]** slot space, reinterpreted while `pending_tutor_player_idx is not None`: slot `i` means "select match index `i` into `pending_tutor_matches`". Regular PLAY_CARD is illegal during pending_tutor anyway, so the channel is free.
- `DECLINE_TUTOR` reuses **slot 1001 (PASS)** while pending_tutor is set — same trick as 14.1's `DECLINE_POST_MOVE_ATTACK`. Regular PASS-as-end-turn is illegal during pending_tutor.
- Only ONE pending flavour active at a time. `pending_tutor` fires from `on_play` (PLAY_CARD path); `pending_post_move_attacker` fires from MOVE. Mutual exclusion is asserted (loud `AssertionError`) in `_enter_pending_tutor`.
- Action payload: `TUTOR_SELECT` reuses `Action.card_index` as the match index — no new dataclass field.

**Tensor parity (Wave 2):** `pending_tutor_player_idx` -> `[N]` int tensor with -1 sentinel; `pending_tutor_matches` -> `[N, K]` with K=8 headroom and -1 padding.

**RL training trade-off:** Existing checkpoints become stale (they were trained against the old auto-tutor model) but remain loadable. Same known-issue posture as Phase 14.1.

## tutor_target Selector Schema

`CardDefinition.tutor_target: Optional[Union[str, Dict[str, str]]]`

- **String shorthand:** `"minion_red_diodebot"` -> matches a card by exact `card_id`. Diodebot JSON files unchanged; back-compat verified.
- **Selector dict:** `{"tribe": "Robot"}`, `{"element": "metal"}`, `{"card_type": "minion"}`, or any AND combination.
  - Allowed keys: `{tribe, element, card_type}`. Loader raises `ValueError` for unknown keys at load time.
  - All provided keys must match (logical AND).
  - Comparisons are case-insensitive string compares against the candidate's stringified attribute.

`CardDefinition.tutor_matches(candidate)` is the single source-of-truth predicate.

- `src/grid_tactics/game_state.py` — Added `pending_tutor_player_idx: Optional[int] = None` and `pending_tutor_matches: tuple = ()` to the `GameState` frozen dataclass.
- `src/grid_tactics/effect_resolver.py`
  - Deleted body of `_resolve_tutor` and replaced with `NotImplementedError` shim (defense in depth).
  - New `_enter_pending_tutor`: scans the caster's deck, collects matching deck indices via `card_def.tutor_matches`, returns state unchanged if no matches, otherwise sets pending fields. Asserts mutex against `pending_post_move_attacker_id`.
  - `resolve_effects_for_trigger` dispatch updated to call `_enter_pending_tutor` for `EffectType.TUTOR`.
- `src/grid_tactics/action_resolver.py`
  - New pending_tutor gate before the existing pending_post_move_attacker gate. Only `TUTOR_SELECT` (with valid match index) or `DECLINE_TUTOR` legal while pending; everything else raises `ValueError("Pending tutor: ...")`.
  - `TUTOR_SELECT` pops the chosen deck card to the caster's hand, clears both pending fields, then runs cleanup + game-over check + the single deferred react window.
  - `DECLINE_TUTOR` clears pending fields and runs the same react window flow.
  - `TUTOR_SELECT` / `DECLINE_TUTOR` outside pending state raise `ValueError`.
  - After normal action dispatch, if `pending_tutor_player_idx` is set, react transition is deferred (mirrors 14.1's `pending_post_move_attacker_id` short-circuit).
- `tests/test_effect_resolver.py` — Added `TestPendingTutorEntry` (7 tests), `TestTutorSelectorLoaderValidation` (1), `TestPendingTutorResolution` (5). 13 new tests, all green.
