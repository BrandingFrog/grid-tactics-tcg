# Phase 14.2 Plan 03: Legal-Action Mask for Pending Tutor Summary

**One-liner:** Gated the action-space mask on `pending_tutor` in both Python and tensor engines, mirroring the 14.1 pending-post-move pattern: TUTOR_SELECT reuses PLAY_CARD slot space and DECLINE_TUTOR reuses slot 1001, with mutex assertions preventing both pending flavours from coexisting.

## Mask Reinterpretation (locked in)

Action-space layout `[0:1262]` is **PRESERVED** — no resize.

While `state.pending_tutor_player_idx is not None` (Python) / `pending_tutor_player[g] >= 0` (tensor):

| Logical action          | Reused slot                                  | Encoder convention                         |
| ----------------------- | -------------------------------------------- | ------------------------------------------ |
| `TUTOR_SELECT(match_i)` | `PLAY_CARD_BASE + match_i * GRID_SIZE` (cell pinned to 0) | `Action.card_index = match_i`              |
| `DECLINE_TUTOR`         | `PASS_IDX = 1001`                            | (no payload)                               |

The `ActionEncoder.decode` path checks `pending_tutor_player_idx` **before** the 14.1 `pending_post_move_attacker_id` check — this is safe because the two pending flavours are mutually exclusive (asserted both in `legal_actions` and in the resolver layer).

All other slots (PLAY_CARD non-tutor, MOVE, ATTACK, SACRIFICE, DRAW, regular PASS, PLAY_REACT, DECLINE_POST_MOVE_ATTACK) are explicitly masked OFF in the pending-tutor branch.

## Mutex Enforcement

Three layers:

1. `legal_actions()` (Python) — `assert not (pending_tutor_player_idx and pending_post_move_attacker_id)` runs **before** the branch dispatch. Loud `AssertionError` on violation.
2. `_apply_pending_tutor_override()` (tensor) — `assert not (has_pending_tutor & pending_post_move >= 0).any()`.
3. Pre-existing asserts in `_enter_pending_tutor` (Wave 1) and `_apply_tutor` (Wave 2) — defense in depth.

## Deviations from Plan

The plan's task snippets describe `legal_actions` as returning a `np.zeros(1262, dtype=bool)` mask directly. In the actual codebase, Python `legal_actions()` returns `tuple[Action, ...]` and the bool mask is built downstream by `rl/action_space.py::build_action_mask` via the `ActionEncoder`. I followed the actual architecture:

- `legal_actions()` returns `(TUTOR_SELECT(i)…, DECLINE_TUTOR)` Action objects.
- `ActionEncoder` was extended (Rule 2 — missing critical functionality) to encode/decode the new action types so `build_action_mask` produces the correct bool mask. Without this the encoder would have raised `ValueError("Unknown action type")` and broken every RL training step that hit a pending-tutor game.
- Tests cover both the Action-list path and the bool-mask path (`build_action_mask` round-trip).

This is a faithful implementation of the plan's intent, just routed through the existing encoder seam instead of duplicating mask construction inside `legal_actions.py`.
