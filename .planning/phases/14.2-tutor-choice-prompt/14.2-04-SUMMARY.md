## Decisions Made

- **Per-viewer enrichment runs AFTER filter_state_for_player.** The standard filter wipes both decks (security). The tutor enrichment then re-injects exactly the resolved match information the *caster* needs (numeric ids of matched deck cards + match indices) without leaking deck order or contents to the opponent. The opponent only receives `pending_tutor_player_idx` + `pending_tutor_match_count`.
- **Send numeric card ids, not full card defs.** `cardDefs` was already pushed at game_start, so the modal looks up the full def client-side via `renderDeckBuilderCard(numeric_id, -1)`. Smaller payload, single card-rendering codepath.
- **Reuse `renderDeckBuilderCard` verbatim.** User-locked UX requirement: tutored cards must render with the SAME art/stats/effects/element/tribe as the deck builder. The function takes a numeric id; passing `count=-1` suppresses the deck-quantity badge while keeping the full card frame.
- **Total-copies-owned counted across deck + hand + board.** This gives the "X of Y copies remaining" pill a stable denominator that reflects what the caster originally brought, not just what's currently in hand. Computed server-side to avoid duplicating ownership math in JS.
- **Modal sync mirrors 14.1's post-move pattern.** `syncPendingTutorUI()` reads `pending_tutor_player_idx` from each state frame and opens/closes the modal idempotently. Reconnection-safe.
- **Background click does NOT dismiss the modal.** The Skip button is the single explicit decline path — accidental clicks must not silently submit DECLINE_TUTOR.
- **Opponent passive toast, no modal.** Showing the opponent the picker would leak the caster's hidden information. A simple "Opponent is tutoring…" toast preserves agency without revealing deck contents.
- **`deck_count` was already serialized** by `filter_state_for_player` and rendered in `renderSelfInfo`/`renderOpponentInfo` — the must-have requirement was already satisfied; no extra work needed there beyond verification.
- **Action wire format reuses existing codec.** `{action_type: 9, card_index: <match_idx>}` for TUTOR_SELECT and `{action_type: 10}` for DECLINE_TUTOR round-trip cleanly through the existing `reconstruct_action`/`ActionType` enum without any codec changes — exactly the encoding decision locked in 14.2-01.

## Wire Format Reference

**Caster state frame (added fields):**
```
pending_tutor_player_idx: int               # 0 or 1
pending_tutor_match_count: int              # len(matches)
pending_tutor_matches: [                    # caster only
  {card_numeric_id: int, deck_idx: int, match_idx: int},
  ...
]
pending_tutor_total_copies_owned: {         # caster only; key is str(numeric_id)
  "12": 3, "8": 2, ...
}
```

**Opponent state frame (added fields):**
```
pending_tutor_player_idx: int               # 0 or 1 (caster)
pending_tutor_match_count: int              # only the COUNT is exposed
pending_tutor_matches: []                   # always empty for opponent
pending_tutor_total_copies_owned: {}        # always empty for opponent
```

**Client → server actions:**
- TUTOR_SELECT: `{action_type: 9, card_index: <match_idx>}`
- DECLINE_TUTOR: `{action_type: 10}`
