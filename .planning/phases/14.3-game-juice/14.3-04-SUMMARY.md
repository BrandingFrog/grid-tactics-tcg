# Phase 14.3 Plan 04: Attack Animation Summary

Attack animation polished end-to-end: rubber-band pullback, snap forward, target tile flash + shake, floating damage popup, return tween. Damage number is server-authoritative via a new minimal `last_action` payload added to `state_update`.

## last_action Payload Schema

Emitted on every `state_update` after a player action (None on game_over / lobby frames):

```json
{
  "type": "ATTACK" | "MOVE" | "PLAY_CARD" | "PASS" | ... ,
  "attacker_pos": [row, col] | null,
  "target_pos":   [row, col] | null,
  "damage":       int | null,
  "killed":       bool
}
```

| Field | ATTACK | MOVE | PLAY_CARD | other |
|---|---|---|---|---|
| `attacker_pos` | acting minion's pre-action pos | acting minion's pre-action pos | null | minion pos if action.minion_id |
| `target_pos` | target minion's pre-action pos | move destination | deploy position | null |
| `damage` | actual HP delta (or full HP if killed) | null | null | null |
| `killed` | true if target removed from minion list | false | false | false |

Computed in `view_filter.enrich_last_action(state_dict, prev_state, new_state, action)`. Wired through `_emit_state_to_players(session, state, prev_state, resolved_action)` from the `submit_action` handler in `events.py`.

## Attack Animation Phase Timeline

| Phase | Window | What happens |
|---|---|---|
| A | 0-180ms   | Add `.anim-attack-windup`, set transform = `translate(-0.3·dx, -0.3·dy)` (pullback away from target) |
| B | 180-280ms | Hold pullback (100ms anticipation pause) |
| C | 280-400ms | Swap to `.anim-attack-strike`, set transform = `translate(0.7·dx, 0.7·dy)` (snap toward target edge) |
| D | 400ms     | Target cell gets `.anim-target-hit` (flash + shake, 400ms keyframe). Damage popup `<div class="damage-popup">-N</div>` appended to target cell, animates `damage-float` 900ms |
| E | 400-700ms | Swap back to `.anim-attack-windup`, set transform = `translate(0,0)` for return tween |
| F | 700ms+    | Cleanup classes/transform, call `done()`. `runQueue` then calls `applyStateFrame(stateAfter)` which re-renders and removes any killed minion |

Total: ~800ms (A+B+C+E = 700, then state apply on next frame).

## Deviations from Plan

None — plan executed as written. Two small clarifications:

1. **`.board-cell` already has `position: relative`** (line 865 of game.css), so the damage-popup absolute-positioning anchors correctly without any CSS prerequisite changes. The plan's "verify in CSS, add if missing" was a no-op.
2. **State application timing** — the plan suggested calling `applyStateFrame(job.stateAfter)` explicitly inside PHASE F. Wave 2 (summon) had introduced a `job.stateApplied` flag for branches that apply state mid-animation. For attack, the natural choice is the *opposite*: don't set the flag, let `runQueue` do its default `applyStateFrame` after `done()`. This guarantees killed minions disappear AFTER the strike (which is what we want) without duplicating the call.

No Rule 1/2/3 fixes needed. No Rule 4 architectural decisions.
