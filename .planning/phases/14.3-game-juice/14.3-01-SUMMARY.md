# Phase 14.3 Plan 01: Animation Queue Infrastructure Summary

Client-side AnimationQueue + state buffering foundation for game juice. State frames for gameplay actions (summon/move/attack) are now buffered behind animations; pending UIs (react window, tutor modal, post-move-attack pick) wait for the queue to drain before opening.

## API Surface (for later waves)

### Job shape

```js
{
  type: 'summon' | 'move' | 'attack' | 'noop',
  payload: { /* type-specific fields */ },
  stateAfter: <state frame>,          // set by onStateUpdate
  legalActionsAfter: <legal actions>, // set by onStateUpdate
}
```

### Functions

| Function | Purpose |
|---|---|
| `enqueueAnimation(job)` | Push a job and kick the queue. |
| `runQueue()` | If idle, shift next job, run playAnimation, then applyStateFrame + recurse. Re-entrant-safe via `animRunning` guard. |
| `playAnimation(job, done)` | Switch on `job.type`. **Wave 1: every branch is `setTimeout(done, 0)`.** Waves 2-4 replace branches with real visuals; contract is "call `done()` when animation finishes". |
| `applyStateFrame(frame, legal)` | Single point of state application. Sets `gameState`, optionally `legalActions`, runs `logStateDiff`, calls `renderGame()`. |
| `isAnimating()` | Returns `animRunning || animQueue.length > 0`. |

### Flow

```
socket 'state_update'
  └─ onStateUpdate(data)
       ├─ first frame / noop diff → applyStateFrame(next) directly
       └─ action frame → enqueueAnimation({ type, payload, stateAfter, legalActionsAfter })
                              └─ runQueue()
                                   ├─ playAnimation(job, done)  [Wave 1: 0ms stub]
                                   └─ on done: applyStateFrame(job.stateAfter, job.legalActionsAfter)
                                                └─ renderGame()
                                                     ├─ renderReactBanner()         ← pending UI
                                                     ├─ syncPendingPostMoveAttackUI() ← pending UI
                                                     └─ syncPendingTutorUI()         ← pending UI
```

## Deviations from Plan

**Plan suggested** explicit `isAnimating()` guards wrapping each pending-UI opener. **Discovered during Task 3:** because all three sync calls (`renderReactBanner`, `syncPendingPostMoveAttackUI`, `syncPendingTutorUI`) already live inside `renderGame()`, and `renderGame()` is only called from `applyStateFrame()`, and `applyStateFrame()` is only called by `runQueue()` AFTER animation completion (or directly for bypass paths where there is no animation), the gating is **structurally implicit** — no runtime `isAnimating()` guard is needed.

Resolution: documented the implicit gate with Phase 14.3 comment blocks above each sync call rather than adding dead-code guards. Simpler, same guarantee.

No Rule 1/2/3 fixes needed. No Rule 4 architectural decisions.
