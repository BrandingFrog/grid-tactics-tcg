# Phase 14.3 Plan 02: Summon Animation Summary

First real animation plugged into the Wave 1 AnimationQueue. Deploying a minion now plays a 600ms scale-in (2x → 1x) at the destination tile with springy easing, accompanied by a 350ms board shake. React windows remain structurally gated behind animation completion.

## What changed

### CSS (`game.css`)
- `@keyframes summon-scale-in` — 2x → 1x, opacity 0 → 1, cubic-bezier(0.34, 1.56, 0.64, 1) for slight overshoot.
- `.anim-summon` — 600ms, transform-origin center, z-index 10 (so the scaling minion floats above neighbors during the animation).
- `@keyframes grid-shake` — 4-step ±2-3px jitter.
- `.anim-grid-shake` — 350ms ease-in-out.

### JS (`game.js`)
- New `animatingTiles` module-level registry: `{ "row,col": "summon"|... }`.
- `playSummonAnimation(job, done)`:
  1. Set `animatingTiles[key] = 'summon'`.
  2. `applyStateFrame(job.stateAfter, job.legalActionsAfter)` — minion appears NOW.
  3. Set `job.stateApplied = true` so `runQueue` skips its default apply.
  4. Add `.anim-grid-shake` to `#game-board` (force reflow first to retrigger).
  5. After 650ms: clear registry entry, re-render board, call `done()`.
- `playAnimation` — `case 'summon'` branch dispatches to `playSummonAnimation`; other branches still 0ms stubs (Waves 3/4).
- `runQueue` — checks `job.stateApplied` before its default `applyStateFrame`.
- `renderBoard` — checks `animatingTiles[row + ',' + col]`; if present, adds `anim-<kind>` class to the `.board-cell`.
- `deriveAnimationJob` — for `action_type === 0` (PLAY_CARD), now verifies a minion actually appeared at `pa.position` in `next` that wasn't in `prev` before emitting `summon`. Payload extended with `card_id` (numeric_id) for future use. Magic cards with positions correctly fall through to `noop`.

## API surface for Waves 3/4

The `animatingTiles` registry pattern is the reuse target:

```js
// Wave 3 (move): tag BOTH source and destination with kind 'move'
animatingTiles[from] = 'move-from';
animatingTiles[to]   = 'move-to';
// renderBoard adds .anim-move-from / .anim-move-to to those cells.

// Wave 4 (attack): tag attacker + target
animatingTiles[attackerPos] = 'attack';
animatingTiles[targetPos]   = 'hit';
```

For animations whose visuals require the OLD state to persist (e.g. attack
flash before HP drop), set `job.stateApplied = false` (default) and
`runQueue` will apply the state after `done()` fires. For animations
whose visuals require the NEW state up front (e.g. summon), apply the
frame yourself and set `job.stateApplied = true`.

## Deviations from Plan

**Plan said** "set animatingTiles, then call applyStateFrame, then have renderBoardMinion check animatingTiles." **Implementation** instead checks `animatingTiles` in `renderBoard` (the cell builder) rather than `renderBoardMinion` (the inner HTML builder). This places the class on the `.board-cell` (which has `transform-origin` clean and a stable layout box), not on the inner `.board-minion` div. Cleaner CSS, identical visual effect.

**Plan said** "650ms" for the clear-and-done timeout. Kept exactly 650ms (50ms slack over the 600ms scale-in).

**Plan implied** runQueue might need no changes. **Discovered during Task 2:** because summon applies state up-front (so the minion is visible during the scale-in), runQueue's default post-animation `applyStateFrame` would apply the same frame twice. Added a `job.stateApplied` opt-out flag — surgical, one-line, future-proof for any other "apply-at-start" animation.

**Bonus hardening:** `deriveAnimationJob` for PLAY_CARD now verifies a minion actually appeared in `next.minions` at `pa.position` before emitting `summon`. The Wave 1 version only checked `pa.position` was set, which would have misclassified magic-with-position effects (e.g. AOE-at-cell) as summons. Caught while reading the diff path.

No Rule 1/2/3 fixes beyond the runQueue tweak. No Rule 4 architectural decisions.
