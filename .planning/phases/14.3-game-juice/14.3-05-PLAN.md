<task type="auto">
  <name>Task 1: Chain melee move+attack into two sequential animation jobs</name>
  <files>src/grid_tactics/server/static/game.js</files>
  <action>
Phase 14.1 introduced melee minions that move-then-attack as a single server action. The current `last_action` (from Wave 4) likely reports this as a single ATTACK with attacker_pos = post-move position. The user wants to SEE the move first, then the attack.

Two options — implement Option A unless server already exposes pre-move pos:

**Option A (client-side):** In `deriveAnimationJob`, detect a melee post-move-attack by checking if `prev` had the attacker minion at a different position than `next.last_action.attacker_pos`. If so, emit TWO jobs back-to-back via `enqueueAnimation`:
  1. `{ type:'move', payload:{ from: prevPos, to: attackerPos }, stateAfter: <intermediate> }`
  2. `{ type:'attack', payload:{...last_action}, stateAfter: next }`

For the intermediate stateAfter, synthesize it by cloning `prev` and updating only the attacker minion's position. If that's too fragile, set `stateAfter: prev` for job 1 (so job 1 only animates and doesn't apply state) and `stateAfter: next` for job 2.

**Option B (server-side):** Add `pre_attack_pos` to `last_action` in view_filter.py if Option A's intermediate-state synthesis is messy. Document chosen option in summary.

Verify: tutor modal sync code from 14.2 should still work — the tutor modal opens based on `pending_tutor_player_idx` which is set in `applyStateFrame`, so it naturally waits behind animations. No additional changes needed unless smoke test reveals collision.
  </action>
