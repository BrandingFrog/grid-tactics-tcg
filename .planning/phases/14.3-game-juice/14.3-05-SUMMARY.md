## Why No Code Change for Task 1

The plan offered Option A (client-side intermediate state synthesis) and Option B (server `pre_attack_pos` field) under the assumption that a melee move+attack might arrive as a single ATTACK frame whose `attacker_pos` reflects the post-move position.

In practice, per the Phase 14.1-01 design (recorded in STATE decisions), the melee chain is "ONE logical action" only at the **react-window** level — it submits as **TWO sequential server actions**: a `MOVE` (which sets `pending_post_move_attacker_id` server-side, suppressing the react window) followed by an `ATTACK` or `DECLINE_POST_MOVE_ATTACK` (which clears pending and fires the single react window).

Each of those two server actions produces its own `state_update` event, each carrying its own `last_action`. The Wave 4 / Wave 3 `deriveAnimationJob` already branches on `last_action.type === 'MOVE'` and `=== 'ATTACK'` independently, and the AnimationQueue serializes both jobs back-to-back. The user sees: move animation → attack animation → react window. Exactly the desired sequence.

Option C (verified-to-already-work) was chosen.
