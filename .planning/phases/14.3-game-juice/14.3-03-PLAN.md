Replace `case 'move':` stub with `playMoveAnimation(job, done)`.

`function playMoveAnimation(job, done)`:
1. Read `{ from:[r1,c1], to:[r2,c2] }` from `job.payload`.
2. Locate the source tile DOM element via the same selector renderBoard uses (e.g. `document.querySelector('[data-row="r1"][data-col="c1"]')`). Grab the inner minion element.
3. Compute the pixel delta between source and destination tile centers via `getBoundingClientRect()`.
4. PHASE A — lift (0ms): add `anim-move-lift` to the minion element.
5. PHASE B — translate (120ms later): add `anim-move-translate` and set `style.transform = translate(<dx>px, <dy>px) scale(1.15)`. Wait 350ms.
6. PHASE C — apply state + drop (470ms total): call `applyStateFrame(job.stateAfter)` so the minion now lives in the destination tile in the DOM (renderBoard re-renders both tiles). The freshly rendered destination minion gets `anim-move-drop` class via the `animatingTiles` registry. Wait 120ms.
7. Clear `animatingTiles` entries for both tiles, call `done()`.

Edge case: if the source tile is removed mid-animation by renderBoard (because state hasn't been applied yet), guard by NOT calling applyStateFrame until phase C. The board between Phase A and C reflects `prev` state, not `next`. Wave 1's `applyStateFrame` defer-until-done contract makes this natural.

Update `deriveAnimationJob`: detect MOVE actions from `prev.pending_action` (or by diffing minion positions: same minion id at different `[row,col]`).
