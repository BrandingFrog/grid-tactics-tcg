# Phase 14.3 Plan 03: Move Animation Summary

Move actions now play a lift → translate → drop animation. The minion scales up
with a drop-shadow, slides along a 350ms cubic-bezier curve to the destination
tile, and drops back to scale 1.0 on landing. Source tile clears mid-flight;
destination fills only at the moment of landing.

## Animation Phase Timeline

| Phase | Window | What happens |
|---|---|---|
| A | 0ms      | Add `.anim-move-lift` to the source `.board-minion` (scale 1.15 + drop-shadow, 120ms ease-out) |
| B | 120ms    | Add `.anim-move-translate`, set inline `transform: translate(dx,dy) scale(1.15)`. Browser animates the slide over 350ms |
| C | 470ms    | `animatingTiles[destKey] = 'move-drop'` then `applyStateFrame(stateAfter)`. Board re-renders: source empty, destination holds minion. Destination `.board-cell` gets `.anim-move-drop` class which plays the drop keyframe (scale 1.15 → 1.0 + shadow off, 120ms ease-in) |
| D | 600ms    | Clear `animatingTiles` entries for source + destination, re-render board (drop class falls off), `done()` |

`runQueue` skips its default post-animation `applyStateFrame` because PHASE C
sets `job.stateApplied = true` (same opt-out flag introduced for summon in
14.3-02).

## What changed

### CSS (`game.css`)
- `.anim-move-lift` — transform scale(1.15) + drop-shadow, 120ms ease-out
- `.anim-move-translate` — 350ms cubic-bezier(0.4, 0.0, 0.2, 1) transition
- `.anim-move-drop` + `@keyframes move-drop` — 120ms ease-in drop back to scale 1.0

### JS (`game.js`)
- New `getTileDelta(fromPos, toPos)` helper — returns `{dx, dy, fromCell, toCell}` or `null` if either tile is missing from the DOM
- New `playMoveAnimation(job, done)` — orchestrates the four phases above
- `playAnimation` — `case 'move'` now dispatches to `playMoveAnimation` (was `setTimeout(done,0)` stub)
- `deriveAnimationJob` — prefers `last_action.type === 'MOVE'` before falling back to `pending_action` (server-authoritative attacker_pos/target_pos)

## Tile-Rect Helper (for reuse)

`getTileDelta(fromPos, toPos)` is the shared measurement primitive:

```js
var delta = getTileDelta([0,2], [1,2]);
// → { dx: 0, dy: 64, fromCell: <div>, toCell: <div> }
// Returns null if either tile is not in the DOM.
```

The Wave 4 attack animation (14.3-04) predates this helper and still computes
its own `getBoundingClientRect` deltas inline. It could be retrofitted to use
`getTileDelta()` but was left untouched to avoid touching shipped code.

## Deviations from Plan

**Plan said** "Total move animation runs 400-500ms." **Implementation** runs ~600ms (120 lift + 350 slide + 130 drop). Kept the plan's per-phase numbers (120/350/120) which sum to 590; the 10ms extra in the drop phase is queue-cleanup slack. The visual quality justifies the slightly longer total — within the same order of magnitude as summon (650ms) and attack (~800ms), so it composes naturally in the queue.

**Plan said** Phase C "applies state, drop class via animatingTiles registry. Wait 120ms." **Implementation** also calls `renderBoard()` again in PHASE D (after 130ms) so the `.anim-move-drop` class falls off the cell. Same defensive pattern as `playSummonAnimation`'s end-of-life cleanup.

**Plan suggested** detecting MOVE in `deriveAnimationJob` "by diffing minion positions: same minion id at different `[row,col]`." **Implementation** uses the existing `pa.source_position`/`pa.target_position` path that was already in place from Wave 1, AND additionally prefers `last_action.MOVE` (added in this wave) for server-authoritative coordinates. No minion-id diffing needed — the server already tells us.

**Bonus:** Extracted `getTileDelta()` as a shared helper instead of duplicating `getBoundingClientRect` math inline (the plan's "noting tile-rect measurement helper for reuse in Wave 4" output spec made this an explicit deliverable).

No Rule 1/2/3 fixes. No Rule 4 architectural decisions.
