  <action>
Add a new section near the top of game.js (after socket setup, before `onStateUpdate` at line ~1388) called "Section: AnimationQueue (Phase 14.3)".

Define:
- `var animQueue = [];` — array of pending jobs
- `var animRunning = false;` — guard
- `var bufferedStateFrame = null;` — most recent state frame waiting to be applied
- `function enqueueAnimation(job)` — job is `{ type: 'summon'|'move'|'attack'|'noop', payload: {...}, stateAfter: <frame>, durationMs: <int> }`. Pushes job and calls `runQueue()`.
- `function runQueue()` — if `animRunning` returns; else shifts next job, sets `animRunning=true`, calls `playAnimation(job)` which returns a Promise (or accepts a `done` callback). On completion: applies `job.stateAfter` via `applyStateFrame()`, sets `animRunning=false`, calls `runQueue()` recursively.
- `function playAnimation(job, done)` — switch on `job.type`. For Wave 1, ALL types call `setTimeout(done, 0)` (no-op stub). Waves 2-4 will replace these branches.
- `function applyStateFrame(frame)` — extracts the existing state-application body of `onStateUpdate` (the part that sets `gameState = next`, calls `renderBoard`, updates pending UI, etc.) into this helper so both the queue and the initial-state path can call it.
- `function isAnimating()` — returns `animRunning || animQueue.length > 0`. Exposed so other code can gate on it.

Do NOT add CSS yet. Do NOT remove any existing render logic — only refactor `onStateUpdate` to call the queue.
  </action>

<task type="auto">
  <name>Task 2: Rewire onStateUpdate to diff and enqueue</name>
  <files>src/grid_tactics/server/static/game.js</files>
  <action>
Refactor `onStateUpdate(data)` (around line 1388) so it no longer applies state directly. Instead:

1. Compute `prev = gameState` (may be null on first frame), `next = data`.
2. If `prev` is null OR this is a non-action frame (no `pending_action` AND no diff in minion positions/HP/hand sizes that indicates an action), bypass the queue: call `applyStateFrame(next)` directly. (Initial joins, lobby state, react open/close, etc., must NOT be queued.)
3. Otherwise, derive an animation job from the diff:
   - Implement `function deriveAnimationJob(prev, next)` that inspects `prev.pending_action` / `next.pending_action` and the board diff to decide:
     - PLAY_CARD of a minion → `{ type:'summon', payload:{ pos, card_id } }`
     - MOVE → `{ type:'move', payload:{ from, to, minion_id } }`
     - ATTACK → `{ type:'attack', payload:{ attackerPos, targetPos, damage, killed } }`
     - Anything else (pass, draw, react resolve, magic, sacrifice for now) → `{ type:'noop' }`
   - For Wave 1 the derivation can be conservative — return `noop` for everything except move/attack/summon detection stubs. Later waves will sharpen the diff.
4. Attach `stateAfter: next` to the job and call `enqueueAnimation(job)`.
5. Preserve the existing pending_action / phase-transition handling (lines ~1427+) but move it INSIDE `applyStateFrame` so it fires after the animation, not before.

Critically: the react window opening logic and tutor modal sync logic (~1670) must run inside `applyStateFrame`, NOT in the raw socket handler. This guarantees they wait for animations.
  </action>
  <verify>Play through several actions (deploy a minion, move, attack, pass). Game must behave identically to pre-refactor. Add a `console.log` in `enqueueAnimation` to confirm jobs flow through.</verify>
  <done>onStateUpdate diffs prev/next, enqueues a job, and applyStateFrame runs only after the queue's done callback. Pending UIs still open correctly because they live in applyStateFrame.</done>
