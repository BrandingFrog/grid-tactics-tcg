# Phase 14.3 Plan 06: Engine Status Effects (Burning) Summary

**One-liner:** Burning is now a fully-modeled per-minion status effect in both Python and tensor engines, applied via the new `APPLY_BURNING` effect type, ticked at end-of-turn for `stacks * BURN_DPT` damage, lethal-safe via existing death cleanup, serialized to clients, and proven by Emberplague Rat's on-attack effect.

## What Landed

### Python engine
- `MinionInstance.burning_stacks: int = 0` field on the frozen dataclass.
- `BURN_DPT = 1` module constant in `minion.py` (naming pattern `<STATUS>_DPT` reserved for future statuses).
- `EffectType.APPLY_BURNING = 15` appended to enum.
- `effect_resolver._apply_effect_to_minion`: APPLY_BURNING branch — additive stacking via `replace(burning_stacks=old + amount)`, default amount=1.
- `react_stack.tick_status_effects(state, library)`: walks board minions in `(row, col)` order, deals `stacks * BURN_DPT` damage to each minion with `burning_stacks > 0`, decrements stacks (clamped >= 0), then routes through `_cleanup_dead_minions` + `_check_game_over` so on-death triggers fire and game-over is honored.
- `resolve_react_stack` calls `tick_status_effects` AFTER turn flip, BEFORE mana regen / auto-draw. Game-over check short-circuits.
- `GameState.to_dict` / `from_dict` round-trip `burning_stacks`.

### Tensor engine (parity)
- `TensorGameState.burning_stacks: int32 [N, MAX_MINIONS]` (int32 chosen for uniformity per 14.2 precedent).
- Reset on `reset_batch` and on minion deploy (slot reuse).
- `step_batch` turn-flip path (after `should_advance` resolves, before draw): vectorized tick — `tick_mask = should_advance & alive & (burning_stacks > 0)`, applies `burn_damage = burning_stacks * BURN_DPT`, decrements stacks, then calls `cleanup_dead_minions_batch` + `check_game_over_batch` for lethal burns.
- `effects._apply_burning(state, active, etarget, eamount, target_flat_pos)` handler dispatched from `apply_effects_batch` on `etype == 15` (SINGLE_TARGET only; AoE adds a one-line branch later).
- `CardTable` stores `effect_type[i,j] = eff.effect_type.value` so APPLY_BURNING (15) appears automatically with no schema change.

### Card pipeline
- `data/cards/minion_emberplague_rat.json` gains a second effect: `{type: apply_burning, trigger: on_attack, target: single_target, amount: 1}` alongside its existing passive `burn` aura.
- `CardLoader._parse_enum` resolves `"apply_burning"` -> `EffectType.APPLY_BURNING` automatically (no loader changes).

### Serialization & client
- `GameState.to_dict` per-minion payload now includes `"burning_stacks": int(m.burning_stacks)`. This flows through `view_filter.filter_state_for_player` unchanged (deep-copy passthrough). Burning is public information — both players see it.

### Glossary
- `data/GLOSSARY.md` and `src/grid_tactics/server/static/game.js` `KEYWORD_GLOSSARY` both gain a new "Burning" entry: *"When this minion is burning, it takes 1 damage at the start of each turn and burning decreases by 1."* (Distinct from the legacy "Burn" passive aura keyword used by Emberplague Rat's pre-existing effect.)

### Tests (`tests/test_status_effects.py`, 4 passing)
1. `test_apply_burning_via_effect` — APPLY_BURNING through `resolve_effect` sets `burning_stacks=2` on the target only.
2. `test_burn_ticks_at_turn_flip` — HP=5 stacks=2 → tick → HP=3 stacks=1 → tick → HP=2 stacks=0 → tick (no-op) → HP=2.
3. `test_burn_lethal_kills_minion` — HP=1 stacks=1 → tick → minion removed via standard death cleanup.
4. `test_burn_stacks_additive` — Two amount=1 applications stack to 2.

## Burn Tick Semantics (locked, documented in `tick_status_effects` docstring)

- **Who:** ALL board minions with `burning_stacks > 0`, regardless of owner.
- **When:** Once per turn flip in `resolve_react_stack`, AFTER react resolves and active player flips, BEFORE mana regen.
- **Damage:** `stacks * BURN_DPT` (BURN_DPT=1).
- **Decrement:** `stacks -= 1` clamped at 0.
- **Death:** Routed through `_cleanup_dead_minions` + `_check_game_over` — on-death effects fire as in combat.
- **Order:** Python iterates `(row, col)` for determinism; tensor processes all slots in parallel (independent — no minion's burn affects another's).

## Naming Pattern for Future Statuses

Adding `frozen`, `stunned`, `poisoned`, etc., requires only:
1. Add `<status>_stacks: int = 0` (or `<status>_turns_remaining`) to `MinionInstance` and corresponding tensor field.
2. Add `EffectType.APPLY_<STATUS>` enum value (append).
3. Add resolver branch in `effect_resolver._apply_effect_to_minion` and tensor `effects.apply_effects_batch` dispatch.
4. Add tick branch in `tick_status_effects` and tensor `step_batch` turn-flip path.
5. Add serialization line in `GameState.to_dict`.
6. Add tests.

No refactor of the tick architecture, the dispatch shape, or the serialization seam is needed.

## Decisions Made

- **Tensor dtype int32, not int8** — uniformity with the rest of the engine (per 14.2-02 precedent for `pending_tutor_*`); memory cost of one int32 per minion slot per game is negligible.
- **Default amount=1** when `effect.amount` is 0/missing in both Python and tensor handlers.
- **No fog-of-war for burning_stacks** — burning is public information; both players see badges. View filter does not gate it.
- **APPLY_BURNING is distinct from legacy BURN (10)** — BURN is the existing passive aura keyword on Emberplague Rat; APPLY_BURNING is the new stack-based status. The "Burn" and "Burning" glossary entries are also distinct.
- **Lethal burns route through standard death cleanup** in both engines so on-death effects (Promote, Conjure, etc.) fire correctly.
