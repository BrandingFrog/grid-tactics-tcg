# Phase 14.5 Plan 06: Card-draw Animations Summary

**One-liner:** Drawing a card (any source: normal draw, Diodebot / Ratchanter tutor chains, conjure, future react-gain) now animates â€” own cards fly in from the deck-pile area and scale into their hand slot over ~600ms, and the opponent's face-down hand row gets a ~400ms pop-in â€” all driven from a single multiset hand diff feeding the Phase 14.3 AnimationQueue.

## What shipped

### JS (`game.js`)

Two new pure-visual animation branches wired into `playAnimation`'s switch (alongside the existing `summon`/`attack`/`move` cases):

- **`playDrawOwnAnimation(job, done)`**: resolves the live `.card-frame-hand[data-hand-idx="N"]` DOM node for `job.toSlotIndex` (with last-child fallback), builds a `position:fixed` floating clone via `renderCardFrame(def, { context:'hand', interactive:false })` at the target rect, stamps the sourceâ†’target delta as inline `--draw-fly-dx` / `--draw-fly-dy` CSS variables, and adds the `.draw-fly-in` class to fire the `draw-fly-in` keyframe. Hides the real slot during the flight (`visibility:hidden`) so there's no double-render, restores on `animationend` (with an 800ms safety timeout in case the browser swallows the event on hidden tabs). Plays the `card_play` SFX at flight start.

- **`playDrawOppAnimation(job, done)`**: stamps the `.pop-in` class on the LAST child of `#oppHandRow` (state has already been applied by the time this runs, so `renderOppHandRow` has already added a fresh `.opp-hand-card-back`), which fires the `opp-back-pop` keyframe. Removes the class on `animationend` with a 600ms safety timeout.

- **`_resolveDrawFromPoint(fromPos)`** helper: accepts `'deck'` (â†’ own-graveyard pile button rect as proxy origin), `'center'` (â†’ viewport center), or `{x,y}` absolute. Returns `{x,y}`.

New diff + dispatch logic in `onStateUpdate`:

- **`deriveDrawJobs(prev, next)`**: multiset-diffs `prev.players[me].hand` vs `next.players[me].hand` â€” counts prev occurrences, then walks next and emits a `draw_own` job for each entry that isn't cancelled by a prev occurrence. Duplicate draws (two copies of the same card on the same frame) produce two jobs. For opponent, computes `oppCount(next) - oppCount(prev)` where `oppCount` prefers `hand_count` and falls back to `hand.length` (for spectator god mode) â€” positive delta produces that many `draw_opp` jobs; negative delta (discard) produces zero.

- **`onStateUpdate` dispatch**: after computing the primary action job (existing `deriveAnimationJob`) and the draw jobs, it:
  - On noop + draws: `applyStateFrame(next, nextLegal)` then enqueues each draw job with `stateApplied=true`.
  - On summon/move/attack + draws: enqueues the primary job (which handles its own state-apply timing), then enqueues draw jobs with `stateApplied=true` so they animate into the post-state DOM.
  - On noop + zero draws: unchanged â€” immediate `applyStateFrame` bypass, no queue churn.

### CSS (`game.css`)

New Wave 6 block at the end of the stylesheet:

- **`@keyframes draw-fly-in`** â€” `translate(var(--draw-fly-dx), var(--draw-fly-dy)) scale(0.3)` opacity 0 â†’ fade in to opacity 1 by 30% â†’ `translate(0,0) scale(1)` opacity 1 at 100%. 600ms ease-out.
- **`.draw-fly-in`** â€” `position: fixed; z-index: 1000; pointer-events: none; will-change: transform, opacity;` + `animation: draw-fly-in 600ms ease-out forwards`.
- **`@keyframes opp-back-pop`** â€” `scale(0)` opacity 0 â†’ `scale(1.1)` opacity 1 at 60% â†’ `scale(1)` opacity 1 at 100%. 400ms ease-out.
- **`.opp-hand-card-back.pop-in`** â€” `animation: opp-back-pop 400ms ease-out`.

Both draw job types run with `stateApplied=true` so `runQueue()`'s default `applyStateFrame` fall-through is suppressed and the main primary-action branch retains full control of state-apply timing. Draws always run AFTER any primary action job, which means: a card that summons a minion AND draws a card (hypothetical on-play draw) will play summon-fly-in first, then draw-fly-in â€” never both on top of each other.

## Decisions Made

- **State-applied-first over per-job `stateAfter`.** Draws don't drive board state â€” they add hand entries. The cleanest way to make draw animations target DOM nodes that exist is to let the primary action job apply state (or apply it up-front on noop frames) and then run draw jobs as pure visuals against the live, already-rendered hand/opp-row. Alternative (per-job `stateAfter` with a staged apply cascade) would have added ordering bugs where the queue applied the same frame multiple times.

- **Multiset diff, single trigger point.** Instead of sprinkling `enqueueAnimation({type:'draw_own'...})` at every site that adds cards (draw action, Diodebot on-play, Ratchanter tutor cast, conjure, future react-gain effects), `deriveDrawJobs` catches them ALL structurally from the `prev.hand` â†’ `next.hand` delta. Zero chance of missing a code path. Reordered-but-identical hands produce zero jobs (multiset subtract is identity-preserving).

- **'deck' fromPos proxies off the own-graveyard pile button.** There is no dedicated deck-pile DOM element yet (Wave 5 shipped graveyard/exhaust buttons only). The own-graveyard button sits on the self info-bar, visually "off-board", and is always present â€” it's a solid proxy origin until a real deck-pile widget lands. The swap point is a single function: `_resolveDrawFromPoint`.

- **Opponent diff is count-only, matching Wave 5 hidden-info posture.** `deriveDrawJobs` reads `p.hand_count` first; `draw_opp` jobs carry zero identity data. The hidden-info leak-prevention test at the Wave 3 filter layer is unchanged by this wave.

- **`animationend` + safety timeout.** Both animation branches register `animationend` as the primary done-trigger and add a safety `setTimeout` (800ms for own, 600ms for opp) to guarantee `done()` fires even if the browser throttles the animation on a hidden tab. Consistent with the Phase 14.3 summon/attack pattern.

- **`playSfx('card_play')` on own draws only.** Matches existing convention â€” the summon branch also uses `card_play`. No new SFX file needed. Opponent draws are silent (the face-down pop is subtle enough; adding sound would be noisy across chained tutor draws).

## Deviations from Plan

### Deviation 1: `stateApplied=true` cascade instead of per-job `stateAfter`

**Found during:** Task 1 design review.

**Issue:** Plan Task 1 suggested `playDrawOwnAnimation` could either set `job.stateApplied=false` so `runQueue` applies state after the animation, OR apply inline. On closer inspection, apply-after-animation breaks when draws coincide with a primary action job (the primary job's `stateAfter` is already the final frame, and applying it again post-draw is a no-op at best, a re-render at worst). And apply-before-animation at the individual draw-job level means N draws apply state N times.

**Fix:** The draw jobs are pure-visual (`stateApplied=true`, no `stateAfter`). `onStateUpdate` applies state exactly once â€” either via the primary action job's own branch, or via the immediate `applyStateFrame` on noop frames. Draw jobs then animate against the live post-render DOM.

**Files modified:** `game.js` (`onStateUpdate`, `playDrawOwnAnimation`, `playDrawOppAnimation`).

**Commit:** `367158a` (Task 2).

**Impact:** Cleaner ordering, no duplicate state applies, correct primary-action timing preserved.

### Deviation 2: No `interactive:false` option in `renderCardFrame` â€” used `showReactDeploy:false`

**Found during:** Task 1 implementation.

**Issue:** Plan Task 1 referenced `renderCardFrame(card, {context:'hand', interactive:false})`. The floater needs no interaction regardless because it has `pointer-events:none` on the `.draw-fly-in` wrapper and is removed before state re-renders the real slot.

**Fix:** Passed `showReactDeploy:false` (existing option) for clarity. `pointer-events:none` on the floater parent handles interaction-blocking regardless of whether `renderCardFrame` grows an `interactive` flag.

**Files modified:** `game.js` (`playDrawOwnAnimation`).

**Commit:** `2958f07` (Task 1).

**Impact:** None â€” floater is uninteractive by CSS.

1. Normal draw: start turn, see own card fly in; the opposing tab sees a face-down pop in their `#oppHandRow`.
2. Tutor (Diodebot on-play): the Diodebot summon animation plays first, then the tutored card flies into the hand â€” in order, not overlapping.
3. Tutor chain (Ratchanter cast drawing multiple): N fly-in animations play one after another, queue-serialized, no skipped frames.
4. No false positives: reorder hand / play a card / discard â€” zero draw animations fire.
5. Spectator god mode: draws animate for both players' hands (both perspectives get the fly-in, since both hands are arrays).
