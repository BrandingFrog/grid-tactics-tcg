# Phase 14.5 Plan 05: Pile Buttons + Shared Modal + Opp Hand Row Summary

**One-liner:** Graveyards and exhaust piles are now public UI — 4 inline emoji buttons on the info-bars open a single shared pile modal that renders every card via Wave 4's `renderCardFrame`, and the opponent's hand is represented above the board as a row of face-down card backs whose count matches `hand_count` every frame.

## What shipped

### HTML (`game.html`)

- **4 pile buttons** as `<button class="pile-btn">` with `<span class="pile-emoji">` + `<span class="pile-count stat-emoji">`, placed inline at the end of each info-bar's `.stat-group`:
  - `#pileBtnOppGraveyard` / `#pileBtnOppExhaust` inside `.info-bar.opp-bar`
  - `#pileBtnOwnGraveyard` / `#pileBtnOwnExhaust` inside `.info-bar.self-bar`
- **`<div class="opp-hand-row" id="oppHandRow">`** placed between the opponent info bar and the board container.
- **`#pileModal`** overlay (header with `#pileModalTitle` + `#pileModalClose` ×, scrollable `.pile-grid` body) inserted before the existing game-over overlay.

### CSS (`game.css`)

Appended a Wave 5 block containing:

- `.pile-btn` / `.pile-btn:hover` / `.pile-btn:active` — transparent pill button, border turns cyan on hover, scale-down on press. Sits inline with existing stats so no absolute positioning / layout overhaul was needed.
- `.pile-emoji` (22px) and `.pile-count` (15px, 800 weight). The `.pile-count` element also carries the existing `.stat-emoji` class so the drop-shadow halo filter is picked up for free.
- `.opp-hand-row` — flex-wrap row, centered, gap 4px, min-height 0 (collapses on empty hand).
- `.opp-hand-card-back` — 42×60 tile with diagonal striped background built from `repeating-linear-gradient` over a dark cyan base, subtle cyan inner-shadow border, `pointer-events: none` so there's no hover/click affordance.
- `.pile-modal-overlay` / `.pile-modal` / `.pile-modal-header` / `.pile-modal-title` / `.pile-modal-close` — visual parity with the tutor modal (same backdrop blur, cyan border, `--card` / `--card2` chrome).
- `.pile-grid` — CSS grid `repeat(auto-fill, minmax(200px, 1fr))`, gap 18px, 200px fixed card width via `.pile-grid .card-frame-pile { width: 200px }`. `.pile-grid-empty` message fills the full row via `grid-column: 1 / -1`.
- Mobile `@media (max-width: 700px)` overrides: modal goes to 140px card tiles, card-backs shrink to 30×44, emoji/count sizes decrease.

### JS (`game.js`)

New functions added in a Wave 5 block just after the DOMContentLoaded bootstrap:

- **`showPileModal(title, cardNumericIds)`** — sets title, clears grid, iterates `ids`, looks up each via `allCardDefs[nid] || cardDefs[nid]`, calls `renderCardFrame(c, { context: 'pile', numericId, showReactDeploy: false })` and appends. Empty pile renders a `.pile-grid-empty` message. Shows the overlay with `display: flex`.
- **`hidePileModal()`** — sets `display: none`.
- **`renderOppHandRow(count)`** — clears `#oppHandRow`, appends `count` `<div class="opp-hand-card-back">` elements. Identity-free by construction.
- **`updatePileButtonCounts()`** — reads `gameState.players[me].graveyard.length` / `exhaust.length` and the opponent's, writes the number into each button's `.pile-count` child. Null-safe: missing pile arrays render `0`.
- **`setupPileHandlers()`** — wires the 4 button clicks to `showPileModal` with a closure over `gameState.players[idx].graveyard/exhaust` so every click reads the latest state at click time (not the state at bind time). Also wires:
  - `#pileModalClose` click → `hidePileModal`
  - Backdrop click on the overlay itself (not children) → `hidePileModal`
  - Global `Escape` keydown → `hidePileModal` if visible
- **`DOMContentLoaded` hook** extended to call `setupPileHandlers()` alongside the existing setup functions.
- **`renderGame`** gained a guarded IIFE that pulls the opponent view and calls `renderOppHandRow(hand_count ?? hand.length ?? 0)` + `updatePileButtonCounts()` every frame. Placed right after `renderOpponentInfo()` so the row and buttons update in lockstep with HAND/HP/MANA stats.

## Contract wired through the stack

```
Wave 3 view_filter                Wave 4 renderer              Wave 5 UI
─────────────────────             ─────────────────            ───────────────────
state.players[*].graveyard  ───▶  renderCardFrame(c,     ───▶  showPileModal(ids)
state.players[*].exhaust          { context:'pile' })
state.players[opp].hand=[]                                     renderOppHandRow(
state.players[opp].hand_count ─────────────────────────────▶   hand_count)
```

Opponent hand identity data never reaches the client (Wave 3 hidden-info test locks this down), and the UI now can't accidentally leak it because `renderOppHandRow` only takes a count.

## Decisions Made

- **Inline pile buttons on info-bars, not absolutely-positioned corner buttons.** The plan suggested absolutely positioning buttons in "play area corners." On inspection the existing layout has no clear absolute parent — the board container, info bars, and hand container are a vertical flexbox chain. Dropping the buttons inline at the end of each existing `.stat-group` (which is already `justify-content: flex-end`) puts them in the visual corners of the info bar (which reads as the top/bottom-right of the play area) with zero layout risk. Less code, responsive for free, consistent with the existing stat row aesthetic.
- **`.stat-emoji` reuse, not a new `.stat-emoji-halo` class.** The plan referenced `.stat-emoji-halo`; grep showed the actual existing class is `.stat-emoji` (filter-based drop-shadow halo for color-glyph emojis). Used that directly on `.pile-count` to inherit the halo instead of creating a duplicate name.
- **Single `showPileModal` + `setupPileHandlers` binder closures.** The 4 buttons share one modal and one render path. Each click handler is a closure over a getter that re-reads `gameState.players[idx]` at click time, so late binding means you always see the current pile even if the modal was opened after state updates.
- **Backdrop click is narrowly scoped** — only triggers close when `e.target === modal` (the overlay itself), not when clicking on a card inside. Prevents accidental dismiss when users click-through a card to inspect it.
- **Escape-to-close** added even though the plan didn't require it — it's 6 lines and matches modal UX expectations; no ROI in omitting it.
- **Count-only opponent hand row.** The row is pure decoration: `<div>` backs, `pointer-events: none`, no data attributes, no numeric ids anywhere. There is structurally no path for identity information to reach the row's DOM because `renderOppHandRow`'s signature is `(count)` — it physically cannot receive card data.
- **God-mode fallback.** `renderGame` reads `hand_count ?? hand.length` so spectator god-mode (which delivers the full opp hand array) still produces the right number of face-down backs in the row. Spectator god mode already shows both full hands via `renderHand`'s appendHand helper, so the face-down row is cosmetically redundant in that mode but not harmful.

## Deviations from Plan

### Rule 1 / Rule 2: Class name correction (`.stat-emoji-halo` → `.stat-emoji`)

**Found during:** Task 1 CSS drafting.
**Issue:** Plan referenced a `.stat-emoji-halo` class for the count halo. No such class exists in `game.css` — grep only finds `.stat-emoji` (defined at line 2819 as a drop-shadow filter halo for color-glyph emojis).
**Fix:** Used `.stat-emoji` directly on the pile count spans. Functionally identical to the plan's intent (same halo effect) with zero duplication.
**Files modified:** `game.html` (class="pile-count stat-emoji").
**Commit:** `42212b4` (Task 1).
**Impact:** None — the halo effect the plan wanted is now on the counts.

### Architecture deviation: Inline buttons vs absolute corner positioning

**Found during:** Task 1 HTML drafting.
**Issue:** Plan said buttons should be absolutely positioned in play-area corners (bottom-right of own, top-right of opp). The game-main container's children (room bar, info bars, board, hand) are a vertical flex chain with no natural absolute-positioning parent, and adding one would have required either a new wrapper `<div class="play-area">` or per-button absolute coordinates that would drift on window resize.
**Fix:** Placed the buttons inline at the end of each `.info-bar .stat-group` (which is already right-aligned via `justify-content: flex-end`). Visually this reads as the right side of the top bar (opp area) and right side of the bottom bar (own area) — effectively "top-right" and "bottom-right" of the play area as the plan intended.
**Files modified:** `game.html`, `game.css`.
**Commit:** `42212b4` (Task 1).
**Impact:** Less code, better responsive behavior, no layout wrapper churn. The "visual corner" goal is met without absolute positioning fragility.
