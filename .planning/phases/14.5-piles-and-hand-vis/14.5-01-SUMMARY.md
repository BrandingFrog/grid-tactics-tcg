# Phase 14.5 Plan 01: Engine piles (graveyard/exhaust/token-exclusion) Summary

One-liner: Engine now distinguishes deck-origin minions from tokens via MinionInstance.from_deck, routes discard-for-cost to a new Player.exhaust pile, and stops double-counting minion plays in the graveyard.

## What changed

- **MinionInstance.from_deck: bool = True** — new field. Normal PLAY_CARD deploys inherit the default; the activated `summon_token` path explicitly sets `from_deck=False`. Tokens vanish silently on death.
- **Player.exhaust: tuple[int, ...] = ()** — new field parallel to `graveyard`. `summon_sacrifice_tribe` discards now route here via a new `Player.exhaust_from_hand(card_id)` helper.
- **Player.remove_from_hand(card_id)** — new helper that removes a hand card WITHOUT appending to any pile. Used by the minion PLAY_CARD path so deployed minions no longer appear in graveyard on play. Magic/react one-shots still use `discard_from_hand` (graveyard on play).
- **_cleanup_dead_minions** now gates graveyard append on `minion.from_deck`. Board cleanup is unconditional.
- **_apply_sacrifice** now gates graveyard append on `minion.from_deck` (consistency with death cleanup).
- **game_state.to_dict/from_dict** mirrors both new fields with backward-compatible defaults (`from_deck=True`, `exhaust=()`).

## Decisions Made

- **from_deck is the source of truth, not a hand-origin flag.** Conjured-to-hand cards (dormant `_resolve_conjure` effect path) were out of scope — no current card uses that effect and the plan's Ratchanter Common Rat flow actually goes through pending_tutor (deck search), so those rats ARE from deck. Left that path unchanged.
- **Split hand removal into three verbs** (`remove_from_hand` / `discard_from_hand` / `exhaust_from_hand`) rather than adding a "destination" param to one method. Each call site is explicit about which pile (if any) the card lands in.
- **Gated sacrifice action on from_deck too** even though current gameplay can't sacrifice a token (no token is playable and no token is on the back row by design). Symmetry with death cleanup avoids a future footgun.
- **Tensor engine untouched** — plan explicitly defers parity to Wave 2. Existing tensor parity test (`test_random_games_match`) was already failing on master with a hand_size mismatch; not caused by this wave.

## Deviations from Plan

### Rule 1 — Bug fix: minion plays double-counted in graveyard

**Found during:** Task 2 (tracing the discard_from_hand flow)

**Issue:** Pre-existing behavior routed ALL PLAY_CARD through `discard_from_hand`, which appended the card to graveyard immediately. For minion plays this meant the card showed up in graveyard on play AND AGAIN on death via `_cleanup_dead_minions` — a double count that corrupted future pile UI and any card-text effect that counts graveyard size.

**Fix:** Split the PLAY_CARD path in `_play_card`: minions use the new `remove_from_hand`, magic cards still use `discard_from_hand`. Updated one stale assertion in `test_deploy_melee_to_friendly_row` (which was asserting the bugged behavior).

**Files modified:** src/grid_tactics/action_resolver.py, tests/test_action_resolver.py

**Commit:** aa93005

This fix was not called out explicitly in the plan but is a direct consequence of the plan's truths list:
- "When a from_deck=True minion dies, its card_numeric_id is appended to owner's graveyard"
- "Magic/react cards played as one-shots append to owner's graveyard"

The plan's truths silently imply minions do NOT belong in graveyard until they die. Removing the double-count was necessary for those truths to hold.

### Scope note — dormant `_resolve_conjure` effect path left untouched

The plan said to audit Diodebot-style conjure-to-hand paths. The `_resolve_conjure` effect in `effect_resolver.py` (L464) IS such a path, but no current card uses it (grep of `data/cards/*` shows zero `summon_token_target` usages via the CONJURE effect type). The active Ratchanter flow goes through `_apply_conjure_rat_and_buff` → `pending_tutor` → deck search, so conjured rats legitimately originate from the caster's deck and correctly stay `from_deck=True`. Tracking per-hand-card origin (to handle a hypothetical future CONJURE card) would require a parallel tuple on Player and is out of scope for this wave. A code comment in `_resolve_conjure` was not added — if a card ever lights up this path, the implementer will need to introduce the hand-origin flag at that time.
