# Phase 14.5 Plan 03: Server piles serialization Summary

One-liner: view_filter already passed graveyard and exhaust through via deepcopy; this wave documented the public-piles contract in the filter docstring and locked it down with three regression tests (both graveyards public, both exhausts public, opponent hand count-only).

## What changed

- **view_filter.py docstring** — Made the Phase 14.5 contract explicit: "BOTH players' graveyard and exhaust piles are serialized PUBLICLY as full card_numeric_id lists to every viewer. These piles are face-up info; only the opponent's hand (and both decks) remain hidden." No functional code change was needed — `filter_state_for_player` already starts with `copy.deepcopy(state_dict)` and only nulls `opp.hand`, `players[*].deck`, and `seed`. The Wave 1 exhaust field flows through untouched for the same reason graveyard always did.
- **tests/test_view_filter.py** — Seeded `exhaust` on both players in `_make_state_dict` (p0: `[15, 16]`, p1: `[]`) and gave p1 a non-empty graveyard (`[17, 18, 19]`) so the new tests have signal on both sides. Added three tests under `TestFilterStateForPlayer`:
  - `test_view_filter_emits_both_graveyards` — p0 and p1 both see p0's and p1's graveyard contents, from both viewer perspectives.
  - `test_view_filter_emits_both_exhausts` — same symmetric coverage for exhaust.
  - `test_view_filter_hides_opponent_hand_ids` — opponent hand is empty list + `hand_count` int, and none of the opponent's actual card_numeric_ids appear anywhere in the opp's `hand` field, from both viewer perspectives. This is the hidden-info security regression.
- Updated one stale assertion in `test_public_fields_preserved` (`p1.graveyard == []` → `== [17, 18, 19]`) to match the new fixture.

## Decisions Made

- **No functional filter change.** `filter_state_for_player` already preserves all dict keys it doesn't explicitly strip. Graveyard was always public (covered by the existing `test_public_fields_preserved` assertion); exhaust is a new Wave 1 field that inherits the same treatment for free. Adding an explicit whitelist of pile keys was considered and rejected — the current negative-list approach (strip hand/deck/seed) is more resilient to future public fields being added to `to_dict`.
- **Test both perspectives.** Each new test runs the filter twice (viewer=0 and viewer=1) to catch any viewer-idx-dependent regression. Cheap insurance.
- **No separate filter function for piles.** The plan's "serialize piles" language implied new code, but the simplest correct change is zero code — just documentation and tests. Resist NIH.

**Impact:** Cosmetic on git history only; the actual plan deliverables (docstring + tests) are all in and green. Reviewers looking for "Wave 3 tensor work" on commit `5157016` will be confused — that commit accidentally doubles as a Wave 2 feature commit.

### No engine-side work needed for Wave 3

The plan's truths list ("emit `players[me] = {hand: [...], graveyard: [...], exhaust: [...]}`") read as if `view_filter.py` needed new serialization code. On inspection, `GameState.to_dict` already emits `"exhaust": list(p.exhaust)` for both players (Wave 1, line 149 of `game_state.py`), and `filter_state_for_player` deep-copies the whole dict before any stripping. Wave 3 is therefore a no-op at the code level — just test coverage and a docstring contract. Documented so Wave 4 (UI) can safely assume `players[*].graveyard` and `players[*].exhaust` are always present in state frames.
