# Phase 14.5 Plan 02: Tensor Engine Parity for from_deck + Exhaust

One-liner: GPU tensor engine now mirrors Python Wave 1 pile semantics — minion_from_deck flag, exhaust piles parallel to graveyards, minion plays no longer double-count in graveyard, tokens vanish silently.

## What changed

- **TensorGameState.minion_from_deck: [N, MAX_MINIONS] bool** — new field. Default False; set True by `_deploy_minion_batch` on every normal PLAY_CARD. No tensor token-spawn path exists today, so the False default already covers the hypothetical token case. Cleared on death-slot vacation and on reset.
- **TensorGameState.exhausts: [N, 2, MAX_GRAVEYARD] int32 + exhaust_sizes: [N, 2] int32** — new piles parallel to graveyards. Same capacity (80), same sentinel (-1), same pointer pattern.
- **`_add_to_exhaust_batch`** — new helper in actions.py, structural copy of `_add_to_graveyard_batch`. Used only by `_apply_summon_sacrifice_batch`.
- **`apply_play_card_batch`** — Rule 1 fix: previously added EVERY play (minion or magic) to the graveyard on cast. Now only magic plays route to graveyard. Minion plays leave hand via remove-only; the card lands in graveyard later via death-cleanup gated on `from_deck`.
- **`_deploy_minion_batch`** — sets `minion_from_deck[arange_n, safe_slot] = True` for valid deploys, right next to the existing `minion_alive = True` write.
- **`_apply_summon_sacrifice_batch`** — the tribe-sacrifice discard now appends to exhaust via the new helper instead of graveyard. Mirrors Python `Player.exhaust_from_hand`.
- **`apply_sacrifice_batch`** (on-board sacrifice action) — graveyard append gated on `state.minion_from_deck[arange_n, safe_slot]`, symmetric with death cleanup. Defensive even though no tokens currently reach the back row.
- **`cleanup_dead_minions_batch`** (engine.py) — snapshots `dead_from_deck = s.minion_from_deck.clone()` alongside the other dead_* clones per cleanup pass. The per-player graveyard-append loop gates `is_p` on `slot_from_deck`. On slot vacation, `from_deck` is zeroed alongside `alive`.
- **`reset_batch`** (engine.py) — clears `exhausts`, `exhaust_sizes`, and `minion_from_deck` on reset, mask-respecting like the other pile fields.
- **`TensorGameState.clone`** propagates the three new tensors.

## Decisions Made

- **Default from_deck = False, set True on deploy** (rather than default True + set False on token spawn). Chosen because there is no tensor token-spawn path today, so False-by-default is the safest baseline — any future token code path starts from the right place. Every existing deploy gets the True write explicitly.
- **Death-cleanup gates on a cloned snapshot, not the live `minion_from_deck`**, matching the existing pattern for `dead_cid`/`dead_owner`/`dead_row`/`dead_col`. Two-pass cleanup could otherwise race against slot reuse within the same pass.
- **Did NOT touch react play-path** — `apply_react_batch` still appends react cards to graveyard on cast, which is correct (react one-shots go to graveyard, same as magic one-shots, same as Python Wave 1).
- **Did NOT touch activated-ability path** — `apply_activate_ability_batch` (Ratchanter conjure) enters `pending_tutor` which pulls from deck, so conjured rats are legitimately from_deck=True. Symmetric with Python Wave 1's decision to leave the conjure path alone.
- **No unit tests added for the new paths** — the conftest gates tensor tests on torch/sb3 availability AND the only cross-engine parity test (`test_random_games_match`) is on the pre-existing baseline-failure list (hand_size mismatch from a prior phase, unrelated to piles). Deferred until that gating flips, matching the 14.5-01 "tensor parity tests not exercised in this sweep" posture.

## RL Checkpoint Staleness — TRIPLY STALE

All existing RL checkpoints are now **triply stale** after Wave 2:

Specifically: pre-Wave-2 checkpoints observed graveyard populations that included minion plays on cast; post-Wave-2 observations never do. Any RL feature derived from graveyard contents (card counting, archetype detection, lethal detection via sacrificed minions) is subtly different.
