## Decisions Made

- **Opts refactor is additive + backward-compatible:** `opts = opts || {}; var target = opts.mount || document.getElementById('legacy'); var state = opts.state || gameState;` pattern keeps every existing zero-arg call site byte-identical. The alternative (adding a second `renderBoardInto(mount, state, ...)` entry point) was rejected because it would split the render logic and create a drift risk between live game and sandbox.
- **renderHand.opts.godView renders single-hand, not dual-hand.** The sandbox calls renderHand twice (once per player) with distinct mounts, so each call should render ONLY the ownerIdx hand. The spectator dual-hand branch is preserved untouched for the live-game spectator path. This keeps the sandbox hand logic symmetric with the two separate DOM mounts and avoids any branching inside renderSandbox.
- **animatingTiles is part of the snapshot set.** Even though it's not directly read by click handlers, it's read by renderBoard to apply `.anim-*` classes, and lingering live-game animation markers could leak into sandbox rendering. Resetting to `{}` on activation and restoring on exit is the clean fix.
- **Sandbox section placed at the bottom of game.js (append-only).** Keeps the diff blast radius minimal, groups all sandbox logic in one grep-able region, and lets plan 14.6-03 extend the section via anchor-comment insertion points rather than fragile line numbers.
- **Additive cardDefs mirror.** Sandbox might run BEFORE the user has ever entered a live game, in which case `cardDefs` would be empty and renderBoardMinion / renderHandCard would fail to resolve numeric ids. The sandbox_card_defs handler mirrors ONLY missing keys into cardDefs (and assigns allCardDefs only if null) so the live-game cardDefs never gets stomped.
- **Sandbox browser smoke test performed via Socket.IO client, not a real browser.** Playwright MCP was not available as a tool in this execution session, so the smoke test was done via the python-socketio client against a real `pvp_server.py` instance. This exercises the same Flask + Socket.IO layer a browser would, validates the full payload round-trip, and confirms the DOM markup served to clients (verified via curl against the running server). The JS was additionally syntax-checked via `node -c`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Sandbox needs cardDefs populated, not just allCardDefs**

- **Found during:** Task 2 (setupSandboxSocketHandlers implementation review)
- **Issue:** The plan specified mirroring `sandbox_card_defs` into `allCardDefs` only if empty. But the actual renderers (`renderBoardMinion`, `renderHandCard`) read from the module-level `cardDefs` variable, not `allCardDefs`. A user who opens the Sandbox tab BEFORE entering a live game would have `cardDefs = {}` and all hand/board rendering would fail silently.
- **Fix:** Extended the sandbox_card_defs handler to additively mirror entries into `cardDefs` as well (only filling keys not already present; no overwrites). `allCardDefs` mirror kept per the plan.
- **Files modified:** `src/grid_tactics/server/static/game.js` (`setupSandboxSocketHandlers` body)
- **Verification:** Socket.IO smoke test confirmed 36 card defs received and the sandbox render path reaches the HP/mana stats strip without errors. No console errors observed in the served HTML.
- **Committed in:** `7e3eb70` (Task 2)

**2. [Rule 2 - Missing Critical] Defensive null-guards in renderBoard / renderHand / renderSandboxStats**

- **Found during:** Task 2 (review before first run)
- **Issue:** The plan's refactor snippets did not guard against `state` being null inside the opts path (e.g. if renderSandbox is called BEFORE sandbox_state arrives, or in the split second between sandboxActivate and the first sandbox_state frame). The legacy zero-arg callers were already protected because the live-game code never calls renderBoard/renderHand before game_start. The sandbox path does NOT have that guarantee -- renderSandbox can fire after sandbox_card_defs but before sandbox_state (both events may arrive in different orders depending on Socket.IO buffering).
- **Fix:** Added `if (!state) return;` early-exits to renderBoard and renderHand after the opts fallbacks, and added `if (!sandboxState || !sandboxState.players) return;` at the top of renderSandboxStats. These guards are behaviorally invisible to the legacy callers (because state === gameState is already non-null by the time they fire).
- **Files modified:** `src/grid_tactics/server/static/game.js`
- **Verification:** Socket.IO smoke test produced zero errors in the server output and the payload round trip completed cleanly.
- **Committed in:** `7e3eb70` (Task 2)
