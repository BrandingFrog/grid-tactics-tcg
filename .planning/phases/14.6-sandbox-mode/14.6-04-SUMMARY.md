## Deviations from Plan

### Auto-fixed Issues (mid-phase orchestrator repair between Task 1 and Task 3)

**1. [Rule 2 - Missing Critical] Collapsed .sandbox-container with no vertical size**

- **Found during:** User reported visual breakage after Task 1 returned its UAT checkpoint
- **Issue:** `.sandbox-container { height: 100%; }` sat inside a `.screen { display: block; }` that had no height, so the container collapsed to 0px — the board, hands, and toolbar all rendered at zero height and appeared invisible despite the DOM being present
- **Fix:** Rebuilt #screen-sandbox to mirror #screen-game's `.game-layout` 3-column grid (commit `ca09eae`). The `.game-layout` in game.css already carries an explicit `height: 100vh` (or equivalent) and grid-template-rows, so the new container inherits proper sizing automatically
- **Files modified:** src/grid_tactics/server/static/game.html, src/grid_tactics/server/static/game.css
- **Verification:** All 5 Playwright tests now pass; `test_sandbox_screen_renders` asserts toolbar + search + zone-select + slot-name inputs are actually visible (not just present in DOM)
- **Committed in:** ca09eae (fix(14.6): rebuild sandbox screen to mirror live game layout)

**2. [Rule 2 - Missing Critical] #sandbox-board missing .game-board class → no grid layout**

- **Found during:** Same investigation as #1
- **Issue:** Plan 14.6-02 created `#sandbox-board` with class `sandbox-board` only, NOT `.game-board`. renderBoard emitted 25 cells expecting `.game-board` grid CSS to lay them out in a 5x5, but that class was never applied — the cells stacked in document order. Worse, `#screen-sandbox .sandbox-board { display: flex; }` in the original CSS actively OVERRODE any attempt at grid layout. Net effect: 25 cells in a vertical ribbon instead of a 5x5 board
- **Fix:** Rebuilt #sandbox-board with both `.game-board` and `.sandbox-board` classes and removed the conflicting `.sandbox-board { display: flex }` rule from game.css
- **Files modified:** src/grid_tactics/server/static/game.html, src/grid_tactics/server/static/game.css
- **Verification:** `test_sandbox_layout_is_fixed_god_view` now asserts the 5x5 board is positioned between the two hand mounts (via bounding-box y-ordering), which implicitly requires the board to have non-zero grid dimensions
- **Committed in:** ca09eae

**3. [Rule 2 - Missing Critical] #pileModal nested inside #screen-game → unreachable from sandbox**

- **Found during:** Same investigation as #1 and #2
- **Issue:** Plan 14.6-03 surfaced pile access via `renderSandboxStats` pile-open buttons calling `showPileModal(...)`, which was supposed to open the Phase 14.5 `#pileModal` element. But `#pileModal` lived inside `#screen-game` in the HTML — when the sandbox screen was active, `#screen-game` had `display: none`, so every descendant including `#pileModal` was also display:none. Opening a pile from the sandbox did literally nothing
- **Fix:** Moved `#pileModal` out of `#screen-game` to be a sibling of both screens (modal overlay pattern — always in-DOM, not scoped to any screen). Rebuild commit ca09eae includes this relocation
- **Files modified:** src/grid_tactics/server/static/game.html
- **Verification:** Orchestrator-run Playwright walkthrough clicked a pile button from the sandbox screen; modal opened correctly
- **Committed in:** ca09eae

**4. [Rule 1 - Bug] Initial 10-test Playwright suite targeted pre-repair DOM**

- **Found during:** After commit ca09eae fixed the 3 bugs above, the Task 1 10-test file no longer matched the new DOM structure (selectors like `.sandbox-container > .sandbox-hand-row` pointed at nodes that no longer existed after the rebuild)
- **Issue:** 10 tests targeting pre-repair selectors; maintaining them would mean tracking the pre-repair structure in test code even though production had moved on
- **Fix:** Rewrote the suite as 5 tests of increasing complexity targeting the rebuilt DOM — screen-renders → layout-fixed → search-add-to-hand → cheat-mana → server-slot round trip. Per-player-row assertions use `data-player` attributes for deterministic targeting. All 5 pass in 10.9s
- **Files modified:** tests/e2e/test_sandbox_smoke.py (full rewrite)
- **Verification:** `pytest tests/e2e/test_sandbox_smoke.py -v` against localhost:5000 → 5 passed in 10.9s
- **Committed in:** baa239e (test(14.6): replace e2e smoke with 5 tests matching rebuilt sandbox DOM)

The DOM mount ids preserve the ownership mapping (#sandbox-hand-p0 = P1, #sandbox-hand-p1 = P2) — only the visual positioning via `.game-main` reverses them. Documented extensively in this SUMMARY, REQUIREMENTS DEV-01 inline note, and STATE.md Phase 14.6 closeout section so future readers understand the override.
