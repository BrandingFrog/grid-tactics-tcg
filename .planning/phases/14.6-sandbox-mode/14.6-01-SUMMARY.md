## Reference snippets for downstream plans

### add_card_to_zone (sandbox_session.py)

```python
def add_card_to_zone(self, player_idx: int, card_numeric_id: int, zone: str) -> None:
    if player_idx not in (0, 1):
        raise ValueError("player_idx must be 0 or 1")
    if not (0 <= card_numeric_id < self.library.card_count):
        raise ValueError("unknown card_numeric_id")
    if zone not in ZONES:
        raise ValueError(f"zone must be one of {ZONES}")
    self._push_undo()
    old_player = self._state.players[player_idx]
    cid = card_numeric_id
    if zone == "hand":
        new_player = replace(old_player, hand=old_player.hand + (cid,))
    elif zone == "deck_top":
        new_player = replace(old_player, deck=(cid,) + old_player.deck)
    elif zone == "deck_bottom":
        new_player = replace(old_player, deck=old_player.deck + (cid,))
    elif zone == "graveyard":
        new_player = replace(old_player, grave=old_player.grave + (cid,))
    else:  # exhaust
        new_player = replace(old_player, exhaust=old_player.exhaust + (cid,))
    self._replace_player(player_idx, new_player)
```

### set_player_field (sandbox_session.py)

```python
def set_player_field(self, player_idx: int, field: str, value: int) -> None:
    """FULL CHEAT MODE — no validation against game rules."""
    if player_idx not in (0, 1):
        raise ValueError("player_idx must be 0 or 1")
    if field not in PLAYER_FIELDS:  # ('current_mana', 'max_mana', 'hp')
        raise ValueError(f"field must be one of {PLAYER_FIELDS}")
    int_value = int(value)
    self._push_undo()
    old_player = self._state.players[player_idx]
    new_player = replace(old_player, **{field: int_value})
    self._replace_player(player_idx, new_player)
```

def save_to_slot(self, slot_name: str) -> None:
    """Persist via to_dict() to data/sandbox_saves/<slot_name>.json."""
    path = self._slot_path(slot_name)
    with path.open("w", encoding="utf-8") as f:
        json.dump(self.to_dict(), f)

### _validate_slot_name (sandbox_session.py)

```python
@staticmethod
def _validate_slot_name(slot_name: str) -> str:
    if not isinstance(slot_name, str):
        raise ValueError("slot_name must be a string")
    if not SLOT_NAME_RE.match(slot_name):  # ^[a-zA-Z0-9_-]{1,64}$
        raise ValueError("slot_name must be 1-64 chars of [a-zA-Z0-9_-]")
    if os.path.basename(slot_name) != slot_name:
        raise ValueError("slot_name must not contain path separators")
    return slot_name
```

## Decisions Made

- **Empty state construction:** Build the empty starting `GameState` directly from `Board.empty()` + `Player.new(PlayerSide.X, ())` rather than calling `GameState.new_game(seed=0, deck_p1=(), deck_p2=())`. The `new_game` path crashes on empty decks because it unconditionally draws `STARTING_HAND_P1=3` / `STARTING_HAND_P2=4` cards via `Player.draw_card`, which raises `ValueError("Cannot draw from empty deck")`.
- **No RNG attribute on session:** Sandbox is deterministic god-mode scratch space. Any randomness comes from the engine calls (`resolve_action`) themselves when applicable; the sandbox itself never holds an RNG.
- **HISTORY_MAX = 64:** DEV-09 requires undo/redo depth >= 50. 64 is the next power-of-2 cushion above the minimum.
- **Slot validation is intentionally simple:** Single regex + `os.path.basename` identity check, no sanitization library, no Unicode normalization, no retry. Bad names raise; the user picks a different name.
- **Sandbox handlers are god view:** `_emit_sandbox_state` deliberately does NOT call `filter_state_for_player` / `filter_state_for_spectator`. The whole point of sandbox is to see and edit everything.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `disconnect` handler must clean up sandbox SIDs unconditionally**

- **Found during:** Task 3 (events.py disconnect wiring)
- **Issue:** The plan instructed adding "ONE additional line at the end" of `handle_disconnect`. But the existing handler returns early when `get_token_by_sid` is `None`, and sandbox users have NO session token (sandboxes are keyed by SID, not by token). A literal trailing line would never run for sandbox users — their sandbox would leak on disconnect.
- **Fix:** Restructured the handler so the spectator check is gated on `token is not None`, then unconditionally calls `remove_sandbox(request.sid)` regardless of token state. The spectator path remains functionally identical for token-bearing users.
- **Files modified:** `src/grid_tactics/server/events.py` (handle_disconnect body)
- **Verification:** `test_disconnect_cleans_up_sandbox` in `test_sandbox_events.py` creates a sandbox, disconnects the test client, and asserts `len(rm._sandboxes) == 0`.
- **Committed in:** `fafc7f1` (Task 3 commit)

**2. [Rule 3 - Blocking] Test infrastructure: tests/server/ subdirectory + create_app instead of global `app`**

- **Found during:** Task 4 (test file creation)
- **Issue:** The plan's test file headers import `from grid_tactics.server.app import app, socketio` — but the real `src/grid_tactics/server/app.py` only exposes `create_app(testing=True)`, no module-level `app` global. - **Fix:** Followed the existing `tests/test_pvp_server.py` pattern (`application = create_app(testing=True)`). Created the `tests/server/` subdirectory with an empty `__init__.py` since the plan explicitly specified that path.
- **Files modified:** `tests/server/__init__.py` (created), `tests/server/test_sandbox_events.py` (uses `create_app` instead of importing a non-existent global)
- **Verification:** Both test files import and run cleanly: `pytest tests/server/ -q` → 62 passed.
- **Committed in:** `2603ec7` (Task 4 commit)

**3. [Rule 1 - Bug] `apply_action` test cases must not assume PASS is always legal**

- **Found during:** Task 4 (`test_apply_action_pass_is_legal` and `test_sandbox_apply_action_pass`)
- **Issue:** The plan's test cases assume `pass_action()` is always legal during ACTION phase. The `legal_actions.py` module docstring even claims "PASS is always present (D-16)". But on inspection, `_action_phase_actions` does NOT explicitly append PASS — fatigue bleed at the GameSession layer handles the no-actions case via auto-PASS in `submit_action`. On a truly empty sandbox state, the legal-actions tuple is `()`, so submitting PASS via `apply_action` raises "Illegal action".
- **Fix:** Replaced both `*_pass_is_legal` test cases with engine-driven equivalents that seed a deck card, pull `legal_actions()[0]` (DRAW becomes legal once the deck is non-empty), and apply that. The new tests still verify `apply_action` works through the real engine surface.
- **Files modified:** `tests/server/test_sandbox_session.py` (`test_apply_action_legal_path_round_trip`), `tests/server/test_sandbox_events.py` (`test_sandbox_apply_action_legal_draw`)
- **Verification:** Both tests pass; no test assumes any specific action is "always legal".
- **Committed in:** `2603ec7` (Task 4 commit)

**4. [Rule 1 - Bug] `_find` event-search helper must return the LAST match, not the first**

- **Found during:** Task 4 (`test_sandbox_undo_redo_via_events`)
- **Issue:** When the test emits multiple sandbox events back-to-back (e.g. two undos) and then drains, the queue holds multiple `sandbox_state` messages. A first-match helper returns the state after the FIRST undo, not the most recent. The undo/redo test was failing because it asserted on the latest hand contents.
- **Fix:** `_find` now iterates the full event list and returns the LAST matching payload. Tests that expect "no event of type X" still work because returning `None` from an empty match list is unchanged.
- **Files modified:** `tests/server/test_sandbox_events.py` (`_find` helper)
- **Verification:** `test_sandbox_undo_redo_via_events` passes; the assertion that `state_after = _find(received, "sandbox_state")` is `None` after a bad set_player_field still holds (no sandbox_state was emitted at all).
- **Committed in:** `2603ec7` (Task 4 commit)
