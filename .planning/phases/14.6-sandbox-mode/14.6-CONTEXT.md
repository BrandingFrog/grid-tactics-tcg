# Phase 14.6 Context: Sandbox Mode (Dev Tooling)

## What
A new "Sandbox" tab in the top nav (placed BEFORE the existing Wiki tab — Wiki moves to the right) where the developer can manually set up and play through game scenarios for testing. This is dev tooling, not a player-facing feature.

## Why
- Manual UAT of new mechanics is currently painful — you have to find another tab, create a real lobby, draw real hands, hope you get the cards you need
- Playwright tests need reproducible scenarios — being able to share save files / state codes makes bug reports trivial
- Debugging combat / react / tutor / spectator edge cases needs the ability to set up exact board states
- Useful as a future regression-testing tool: save a JSON file → replay it after refactors

## Functional requirements
1. **New Sandbox tab** in `game.html` top nav, placed before the Wiki link. Wiki link moves over.
2. **Fixed dual-perspective layout** — Player 1's hand on top, Player 2's hand on bottom, both face-up at all times (god mode is always on in sandbox; there is no "view toggle" or "flip the grid" button). This is different from the live game's hide-opponent-hand layout — sandbox is always full-information.
3. **Active player toggle (whose turn it is)** — A control that sets `state.active_player_idx` to P1 or P2 directly. This actually mutates the game state, so legal actions and turn-related rules update accordingly. The toggle is auto-driven during normal play (e.g. react phases switch active player automatically via `resolve_action`); manual toggle is for setting up scenarios. Reacts and modal windows must work — auto-flips during react or pending-tutor phases come from the engine, not the sandbox UI.
4. **Search and add cards** with **zone selection** — type a card name and add it to a chosen zone for either player: Hand, Deck (top or bottom), Graveyard, or Exhaust. Reuse `CardLibrary.all_cards` for the search index.
5. **Move cards between zones** — for any card already in a zone, move it to a different zone (deck → hand, hand → grave, grave → exhaust, etc.). Surface this from the existing pile modals (Phase 14.5) and from the hand row.
6. **Import a deck** — load a saved deck (deck-builder format JSON or paste) into a player's deck zone. Reuse the deck-builder's deck data shape so you can build a deck in one tab and load it in sandbox.
7. **Play cards / move / attack / pass / react / tutor / sacrifice** — every action a real game supports, validated against `legal_actions()` and applied via `resolve_action()`. Click handlers reused from the live game via the global-swap pattern.
8. **Mana / HP cheat inputs** — at any point, set either player's `current_mana`, `max_mana`, or `hp` to any value. This is full cheat mode for testing — no rules enforcement on the inputs.
9. **Save / load state — local AND server**
   - **Client side:** save current state to localStorage (autosave on every action), download as JSON file, paste a base64 share code, copy a base64 share code
   - **Server side:** named save slots persisted to disk on the server (`data/sandbox_saves/<slot_name>.json`), with `save_slot`, `load_slot`, `list_slots`, `delete_slot` Socket.IO handlers
10. **Undo / redo** — keep an in-memory stack of previous states; rewind one step at a time, at least 50 deep. Every state-mutating sandbox action (apply action, add/move card, set mana/hp, set active player, import deck, load slot) pushes to the undo stack.
11. **Reset board** — wipe to a fresh game with empty hands so I can build from scratch

## Constraints — REUSE existing code, no duplicate functions
- **Game engine: reuse**
  - `src/grid_tactics/game_state.py` — `GameState`
  - `src/grid_tactics/action_resolver.py` — `resolve_action`
  - `src/grid_tactics/legal_actions.py` — `legal_actions`
  - `src/grid_tactics/card_library.py` — `CardLibrary`
  - `src/grid_tactics/actions.py` — `Action` + constructors
  - **NO ports, NO copies, NO parallel implementations.**
- **Frontend renderer: reuse `game.js`** board/hand rendering. The sandbox screen should call the same render functions, not duplicate them. The sandbox is a different *screen* but it renders the same `gameState` shape.
- **Server architecture:**
  - Add a thin `SandboxSession` class — does NOT inherit from `GameSession` (avoids leaking 2-player invariants), but holds the same `GameState` + `CardLibrary` + history stack.
  - Exposes: `apply_action`, `to_dict`, `from_dict`, `add_card_to_hand(player, card_id)`, `set_active_player`, `undo`, `redo`, `reset`.
  - Wired via NEW Socket.IO handlers: `sandbox_create`, `sandbox_apply_action`, `sandbox_save`, `sandbox_load`, `sandbox_undo`, `sandbox_redo`, `sandbox_add_card`, `sandbox_set_active_player`, `sandbox_reset`.
  - **NEW** namespace or event names — must NOT collide with existing `submit_action` / `state_update` events.
- **`RoomManager` modification:** ONE additive change — add a `_sandboxes: dict[str, SandboxSession]` keyed by socket SID. Sandboxes are NOT in `_rooms` and NOT in `_games`. They're a parallel dict so the existing room-code lobby flow is untouched.
- **No changes** to: existing `GameSession`, real-game `submit_action` handler, lobby UI, deck builder UI, spectator code, view filter (sandbox uses god view always — no hidden info filtering needed in sandbox).

### Serialization format
The existing `GameState.to_dict()` round-trips fully. JSON shape includes:
- `board` (flattened 7×7)
- `players` (hp, mana, hand, deck, grave, exhaust as numeric IDs)
- `active_player_idx`, `phase`, `turn_number`
- `minions` (list of dicts)
- `pending_action` (for tutor/react state)

No new fields needed.

### Action types
`actions.py` ActionType enum: `PASS`, `DRAW`, `MOVE`, `ATTACK`, `PLAY_CARD`, `PLAY_REACT`, plus special: `DECLINE_POST_MOVE_ATTACK`, `TUTOR_SELECT`, `DECLINE_TUTOR`, `CONJURE_DEPLOY`, `DECLINE_CONJURE`, `TRANSFORM`, `ACTIVATE_ABILITY`. Sandbox supports ALL of these via `resolve_action`.

## Out of scope
- Multi-tab sandbox sharing (one sandbox per browser tab is fine)
- Auto-tutor / AI opponent in sandbox
- Anything that touches RL training, the dashboard, or the wiki
- Modifying `GameSession`, `RoomManager` lobby code, view filter, or `submit_action` handler
- Creating new card data — sandbox only consumes existing CardLibrary
- Authentication / per-user save slots — server save slots are global, single-user, file-on-disk. The whole project will eventually be password-protected at the edge; sandbox does not implement its own auth.
- Flipping the grid / view-toggle / spectator-style perspective swap — sandbox is always god-view, P1 top P2 bottom, fixed

## Open questions for the planner
1. Where does "add card to hand" sit in the engine? `Player.hand` is a tuple in the immutable Python engine — adding a card means creating a new GameState with a modified hand. The planner should design a small sandbox-only helper that uses `dataclasses.replace` rather than mutating in place. The same helper generalizes to "move card between zones" (the source zone is removed-from, the destination zone is appended-to, both via `replace`).
2. How does undo interact with the React phase? If we're mid-react, undoing should jump back to before the triggering action. Plan: undo stack stores full snapshots, no surgery needed.
3. Share-code format: gzip + base64 of `to_dict()` JSON? Length is fine for chat (< 4KB for typical states).

## Decisions made during planning (user-confirmed)
These answers came from a planning-phase Q&A and are binding spec, not assumptions:

- **D1. Layout is fixed dual-perspective.** P1 hand at top, P2 hand at bottom, both face-up. No view toggle, no flip-the-grid. This is god mode at all times — sandbox does NOT use the view filter.
- **D2. "Toggle active player" mutates `state.active_player_idx` directly.** It's a real game-state change, not a view change. The engine's normal active-player transitions during react / pending-tutor still happen automatically via `resolve_action`; the manual toggle is for setting up scenarios.
- **D3. Card placement supports all four zones.** Hand, Deck (top or bottom — top is the next draw), Graveyard, Exhaust. No direct-to-board placement (you put a minion card in hand, then play it from hand to a board tile via the normal play flow). Cards can also be MOVED between zones after placement.
- **D4. Deck import** reuses the existing deck-builder JSON shape. The sandbox can load any saved deck into a player's deck zone in one click.
- **D5. Cheat mana / HP inputs.** Both players have editable `current_mana`, `max_mana`, `hp` numeric inputs that mutate state immediately on change. No validation against engine rules — this is full cheat mode.
- **D6. Save slots are local AND server.** Client side: localStorage autosave + JSON download + base64 share code (already in spec). Server side: NEW — named save slots persisted to `data/sandbox_saves/<slot>.json` on the server with `sandbox_save_slot` / `sandbox_load_slot` / `sandbox_list_slots` / `sandbox_delete_slot` Socket.IO handlers. Slots are global and single-user (the project will be password-protected at the edge later).
- **D7. Sandbox tab is always visible.** No query-param gate, no localhost-only check. Edge auth (project-wide password) is a future concern, not Phase 14.6's problem.
- **D8. Save format versioning** — out of scope for now. If a future engine change breaks old saves, that's acceptable; sandbox is dev tooling.
