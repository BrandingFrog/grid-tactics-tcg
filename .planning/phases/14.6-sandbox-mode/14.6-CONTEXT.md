# Phase 14.6 Context: Sandbox Mode (Dev Tooling)

## What
A new "Sandbox" tab in the top nav (placed BEFORE the existing Wiki tab — Wiki moves to the right) where the developer can manually set up and play through game scenarios for testing. This is dev tooling, not a player-facing feature.

## Why
- Manual UAT of new mechanics is currently painful — you have to find another tab, create a real lobby, draw real hands, hope you get the cards you need
- Playwright tests need reproducible scenarios — being able to share save files / state codes makes bug reports trivial
- Debugging combat / react / tutor / spectator edge cases needs the ability to set up exact board states
- Useful as a future regression-testing tool: save a JSON file → replay it after refactors

## Functional requirements
1. **New Sandbox tab** in `game.html` top nav, placed before the Wiki link. Wiki link moves over.
2. **Toggle active player** — switch between player 1 and player 2 view (god mode by default — see both hands, both decks)
3. **Search and add cards to a hand** — type a card name, click to add to either player's hand. Reuse `CardLibrary.all_cards` for the search index.
4. **Play cards / move / attack / pass** — same actions as a real game, validated against `legal_actions()` and applied via `resolve_action()`
5. **Save / load state** — JSON snapshot using existing `GameState.to_dict()` / `from_dict()`. Save to localStorage and as downloadable file. Also a "share code" (compressed base64) that can be pasted into a chat for debugging.
6. **Undo / redo** — keep an in-memory stack of previous states; rewind one step at a time, at least 50 deep
7. **Reset board** — wipe to a fresh game with empty hands so I can build from scratch

## Constraints — REUSE existing code, no duplicate functions
- **Game engine: reuse**
  - `src/grid_tactics/game_state.py` — `GameState`
  - `src/grid_tactics/action_resolver.py` — `resolve_action`
  - `src/grid_tactics/legal_actions.py` — `legal_actions`
  - `src/grid_tactics/card_library.py` — `CardLibrary`
  - `src/grid_tactics/actions.py` — `Action` + constructors
  - **NO ports, NO copies, NO parallel implementations.**
- **Frontend renderer: reuse `game.js`** board/hand rendering. The sandbox screen should call the same render functions, not duplicate them. The sandbox is a different *screen* but it renders the same `gameState` shape.
- **Server architecture:**
  - Add a thin `SandboxSession` class — does NOT inherit from `GameSession` (avoids leaking 2-player invariants), but holds the same `GameState` + `CardLibrary` + history stack.
  - Exposes: `apply_action`, `to_dict`, `from_dict`, `add_card_to_hand(player, card_id)`, `set_active_player`, `undo`, `redo`, `reset`.
  - Wired via NEW Socket.IO handlers: `sandbox_create`, `sandbox_apply_action`, `sandbox_save`, `sandbox_load`, `sandbox_undo`, `sandbox_redo`, `sandbox_add_card`, `sandbox_set_active_player`, `sandbox_reset`.
  - **NEW** namespace or event names — must NOT collide with existing `submit_action` / `state_update` events.
- **`RoomManager` modification:** ONE additive change — add a `_sandboxes: dict[str, SandboxSession]` keyed by socket SID. Sandboxes are NOT in `_rooms` and NOT in `_games`. They're a parallel dict so the existing room-code lobby flow is untouched.
- **No changes** to: existing `GameSession`, real-game `submit_action` handler, lobby UI, deck builder UI, spectator code, view filter (sandbox uses god view always — no hidden info filtering needed in sandbox).

### Serialization format
The existing `GameState.to_dict()` round-trips fully. JSON shape includes:
- `board` (flattened 7×7)
- `players` (hp, mana, hand, deck, grave, exhaust as numeric IDs)
- `active_player_idx`, `phase`, `turn_number`
- `minions` (list of dicts)
- `pending_action` (for tutor/react state)

No new fields needed.

### Action types
`actions.py` ActionType enum: `PASS`, `DRAW`, `MOVE`, `ATTACK`, `PLAY_CARD`, `PLAY_REACT`, plus special: `DECLINE_POST_MOVE_ATTACK`, `TUTOR_SELECT`, `DECLINE_TUTOR`, `CONJURE_DEPLOY`, `DECLINE_CONJURE`, `TRANSFORM`, `ACTIVATE_ABILITY`. Sandbox supports ALL of these via `resolve_action`.

## Out of scope
- Multi-tab sandbox sharing (one sandbox per browser tab is fine)
- Persistent server-side save slots (localStorage on the client + downloadable JSON is enough)
- Auto-tutor / AI opponent in sandbox
- Anything that touches RL training, the dashboard, or the wiki
- Modifying `GameSession`, `RoomManager` lobby code, view filter, or `submit_action` handler
- Creating new card data — sandbox only consumes existing CardLibrary

## Open questions for the planner
1. Where does "add card to hand" sit in the engine? `Player.hand` is a tuple in the immutable Python engine — adding a card means creating a new GameState with a modified hand. The planner should design a small sandbox-only helper (e.g. `sandbox_state_with_card_added(state, player, card_id)`) that uses `dataclasses.replace` rather than mutating in place.
2. How does undo interact with the React phase? If we're mid-react, undoing should jump back to before the triggering action. Plan: undo stack stores full snapshots, no surgery needed.
3. Share-code format: gzip + base64 of `to_dict()` JSON? Length is fine for chat (< 4KB for typical states).
