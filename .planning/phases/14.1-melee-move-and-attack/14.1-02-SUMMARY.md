# Phase 14.1 Plan 02: Tensor Engine Pending Post-Move Attack Summary

**One-liner:** Mirrored Wave 1 in the GPU tensor engine — removed `_auto_attack_after_move`, added a `[N]` int32 `pending_post_move_attacker` tensor with `-1` sentinel, gated ATTACK on it, routed PASS-while-pending as DECLINE, and deferred the react phase transition while pending is set.

## Field Added

`TensorGameState.pending_post_move_attacker: torch.IntTensor` — shape `[N]`, dtype `int32`, sentinel `-1` = "no pending attack" for that game in the batch. Mirrors `GameState.pending_post_move_attacker_id` from Wave 1 (None ↔ -1).

## Files Changed

- `src/grid_tactics/tensor_engine/state.py`
  - Added `pending_post_move_attacker` field to `TensorGameState` dataclass.
  - Added it to `clone()` and `create_initial_state()` (init to -1).

- `src/grid_tactics/tensor_engine/engine.py`
  - `reset_batch()` clears the field for reset games.
  - `_step_action_phase()`:
    - Reinterprets `PASS` (slot 1001) as `DECLINE_POST_MOVE_ATTACK` when `pending_post_move_attacker >= 0`. Applies decline before normal handlers and excludes those games from PASS/fatigue accounting.
    - Gates the REACT phase transition on `pending_post_move_attacker < 0`, so a MOVE that sets pending stays in ACTION phase until the player resolves with ATTACK or DECLINE.

- `src/grid_tactics/tensor_engine/actions.py`
  - **Removed** `_auto_attack_after_move` (~78 LOC) and its call site in `apply_move_batch`.
  - `apply_move_batch` now computes, for the moved minion, whether any in-range enemy exists from the new tile (melee = manhattan==1 & orthogonal). For melee minions with at least one target, sets `pending_post_move_attacker = safe_slot.to(int32)` via `torch.where`. Ranged minions never enter pending.
  - `apply_attack_batch` gated: when `pending_post_move_attacker[n] >= 0`, only attacks where `a_slot == pending_post_move_attacker[n]` are valid for game `n`. After the attack resolves, pending is cleared via `torch.where`.
  - Added top-level `_apply_decline_post_move_attack(state, mask)` helper.

- `tests/test_tensor_engine_parity.py` (new file, 235 LOC, 6 tests).

## Tests Added

`tests/test_tensor_engine_parity.py::TestPendingPostMoveAttack`:

1. `test_parity_melee_move_pending_state` — melee MOVE into adjacent enemy sets pending and defers react.
2. `test_parity_ranged_move_no_pending` — ranged MOVE never sets pending; react fires immediately.
3. `test_parity_melee_move_no_targets_no_pending` — melee MOVE into open space leaves pending unset; react fires.
4. `test_parity_pending_attack_resolves` — ATTACK from pending clears pending, applies damage, fires react.
5. `test_parity_pending_decline` — PASS-as-DECLINE clears pending, fires react, applies no fatigue.
6. `test_attack_blocked_when_pending_and_wrong_slot` — ATTACK from a non-pending attacker is gated to a no-op.

Tests directly place minions into tensor state via slot writes (the established pattern from `test_tensor_engine.py`), then drive single-game batches through `step_batch` and assert the same observable invariants that the Python Wave 1 tests cover.

## Tensor-Engine Quirks Discovered

- `apply_move_batch` rebuilds attacker card stats inline (it does not share the previous range/owner work that `_auto_attack_after_move` did) — we add a small inline lookup for `atk_range`/`atk_owner` to compute `has_target` cleanly.
- DECLINE has no dedicated action-int slot; it is always encoded via slot 1001 (PASS) and disambiguated by `pending_post_move_attacker >= 0`. The engine performs this disambiguation in `_step_action_phase` _before_ the regular PASS/fatigue path. This required adding an `is_decline` mask and an `~is_decline` exclusion to the existing fatigue branch.
- The tensor engine does not have a `from_python_state` / `to_python_state` bridge, so true side-by-side parity comparison was not possible. The tests instead verify the same observable invariants the Wave 1 Python tests assert.
- React-phase transition gating uses `s.pending_post_move_attacker < 0` rather than the old unconditional transition.
