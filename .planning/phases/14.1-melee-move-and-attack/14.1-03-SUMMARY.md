# Phase 14.1 Plan 03: Pending-Aware Legal-Action Mask Summary

**One-liner:** Taught both Python and tensor legal-action masks about
`pending_post_move_attacker(_id)` so MaskablePPO and the human UI can
only see ATTACK-from-pending or DECLINE while a melee minion is mid-move,
and wired slot 1001's dual meaning into the `ActionEncoder`.

## Slot 1001 Dual Meaning (locked in)

Action-space layout `[0:1262]` is **PRESERVED** (no resizing).

| State                                  | Slot 1001 meaning              |
| -------------------------------------- | ------------------------------ |
| `pending_post_move_attacker_id is None`| `ActionType.PASS`              |
| `pending_post_move_attacker_id` set    | `ActionType.DECLINE_POST_MOVE_ATTACK` |

`ActionEncoder.encode` maps both `PASS` and `DECLINE_POST_MOVE_ATTACK`
to slot 1001. `ActionEncoder.decode` disambiguates by reading
`state.pending_post_move_attacker_id`. The `action_resolver` then
dispatches based on the same field. The tensor engine equivalent
(`_step_action_phase`, established in Wave 2) reads
`state.pending_post_move_attacker >= 0` and routes the slot to
DECLINE before the regular PASS/fatigue branch.

- `src/grid_tactics/rl/action_space.py`
  - `ActionEncoder.encode`: `DECLINE_POST_MOVE_ATTACK` -> `PASS_IDX (1001)`.
  - `ActionEncoder.decode`: slot `1001` decodes to
    `decline_post_move_attack_action()` when
    `state.pending_post_move_attacker_id is not None`, else `pass_action()`.
  - `build_action_mask` is unchanged — it derives mask from
    `legal_actions()` and inherits the new behaviour for free.

- `src/grid_tactics/tensor_engine/legal_actions.py`
  - `compute_legal_mask_batch()` calls a new override at the end:
    `_apply_pending_post_move_attack_override(mask, state, alive)`.
  - Override is fully vectorized over `N`:
    1. `has_pending = alive & (pending >= 0)` — early-return if none.
    2. Look up the pending attacker's `(row, col, owner)` via gather on
       `state.minion_row / minion_col / minion_owner`.
    3. Build `[N, 25]` enemy-cell mask via `scatter_(minion_flat, is_enemy)`.
    4. Compute the 4 cardinal-adjacent target cells per game and gather
       enemy presence at each (`[N, 4]`).
    5. Zero ALL slots for pending games (`mask[has_pending] = False`).
    6. Re-enable ATTACK slots
       `ATTACK_BASE + a_flat * 25 + nflat` via `scatter_`.
    7. Re-enable slot `PASS_IDX` for all pending games.
  - Uses the 4-direction adjacency shortcut (rather than the full
    pairwise distance table) because the Wave 1 invariant guarantees
    the pending attacker is always melee (range 0, manhattan == 1 &
    orthogonal, which on a grid is exactly the 4-cardinal neighbours).

## Decisions Made

- **Slot 1001 reuse, not new slot.** The action space stays at 1262.
  Adding a new int would have invalidated all existing RL checkpoints
  (which already became stale in Wave 1) and required mirror updates in
  `constants.py`, observation wrappers, and the dashboard. Slot 1001 is
  unambiguously context-sensitive on a single state field.
- **Branch in `legal_actions()` BEFORE phase check.** The pending state
  is orthogonal to ACTION/REACT phases (Wave 1 keeps the engine in
  ACTION phase while pending; the react window deferral is downstream).
  Branching first keeps the helper self-contained and the normal phase
  enumerators untouched.
- **Tensor override is post-hoc, not in-line.** Computing the normal
  mask first and then zeroing/re-writing is a few extra ops but avoids
  threading `pending` through every sub-helper and matches the natural
  "rare special case" character of pending state (most batch elements
  will not be pending).
- **4-cardinal adjacency in tensor override** (not the full distance
  table) is correct because Wave 1 only sets pending for melee minions
  (`attack_range == 0`). Saves a `[N, 25, 25]` pairwise allocation in
  the override path.
- **Defensive `pending references missing minion -> only DECLINE`.**
  Should never trigger (Wave 1 invariant), but keeps the UI/agent
  recoverable if state is ever corrupted.
- **`pytest.importorskip("stable_baselines3")` on encoder tests.**
  `grid_tactics.rl.__init__` imports SB3 unconditionally; the encoder
  itself does not need SB3, but the package import does. Skipping is
  preferable to refactoring `rl/__init__.py` (out of scope).
