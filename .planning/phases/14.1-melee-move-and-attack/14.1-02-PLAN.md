  <files>src/grid_tactics/tensor_engine/actions.py</files>
  <action>
1. In the MOVE handler (around lines 180-210):
   - DELETE the call `_auto_attack_after_move(state, valid, safe_slot, dst_row, dst_col, card_table)` (~line 206) and DELETE the entire `_auto_attack_after_move` function (~lines 209-285).
   - After updating positions, compute `is_melee = valid & (card_table.attack_range[atk_cid] == 0)`.
   - Compute `has_target_after_move`: a `[N]` boolean — for each game, does the moved minion at `(dst_row, dst_col)` have at least one enemy in attack range from the new tile? Reuse the same range geometry as the existing legal-actions code (melee = manhattan==1 & ortho). Iterate the 25 minion slots once: `enemy_in_range[n] |= state.minion_alive[n,s] & (state.minion_owner[n,s] != atk_owner[n]) & (manhattan(dst, slot_pos) == 1) & ortho`.
   - Set the pending tensor: `state.pending_post_move_attacker = torch.where(is_melee & has_target_after_move, safe_slot.to(torch.int32), state.pending_post_move_attacker)`.
   - For ranged or melee-without-targets, leave `pending_post_move_attacker` unchanged (-1 or whatever it was, which should be -1 since unrelated games' pending should already be cleared from prior actions).

2. Add a new top-level handler `_apply_decline_post_move_attack(state, mask)`:
   - For each game in `mask`, set `state.pending_post_move_attacker[mask] = -1`. Validity check: only games where `pending_post_move_attacker >= 0` are truly valid; `mask` should already exclude others (legal_actions enforces this in Wave 3, but defensive: `mask = mask & (state.pending_post_move_attacker >= 0)`).

3. In the ATTACK handler: add a guard at the top. If `state.pending_post_move_attacker[n] >= 0` for game n, then ATTACK is only valid if the action's attacker slot equals `pending_post_move_attacker[n]`. After attack resolves, clear pending for those games: `state.pending_post_move_attacker[n] = -1`.
   For games where `pending_post_move_attacker[n] == -1`, ATTACK works as today (no constraint added).

4. Audit the dispatch table / step_batch entry point to route the new `ActionType.DECLINE_POST_MOVE_ATTACK = 8` to `_apply_decline_post_move_attack`. Update any switch / if-elif chain.

5. React-window timing in the tensor engine: mirror the Wave 1 rule. After MOVE with pending set, do NOT trigger the react phase advance for those games. After ATTACK or DECLINE that clears pending, DO trigger react. Find the existing react-trigger logic in `step_batch` or post-action hook and gate it on `pending_post_move_attacker == -1` for the relevant games.
