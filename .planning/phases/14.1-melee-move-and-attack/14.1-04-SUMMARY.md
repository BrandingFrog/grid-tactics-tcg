# Phase 14.1 Plan 04: Frontend Two-Step Move-and-Attack UI Summary

Advance-Wars-style two-step melee move-then-attack UI: after a melee minion moves forward, the board automatically enters attack-pick state showing a soft range footprint (educational) plus a bright pulsing valid-target highlight (actionable), with a Decline Attack button as the explicit out.

| 2 | CSS: .attack-range-footprint + .attack-valid-target classes | 131cffb | Two visually distinct layers (soft hint vs bright actionable pulse) |

| 4 | Visual + UX verification on dev server | deferred | See note below |

## Decisions Made

- **Server pre-computes tile lists.** Range footprint and valid-target lists are derived server-side and serialized into state. This avoids reimplementing the (already-Wave-1-validated) range geometry in JavaScript and keeps the engine the single source of truth.
- **Mode auto-enters from server state.** The client does not infer "we just moved a melee minion" — it simply checks `pending_post_move_attacker_id != null` in every state frame. This is the same pattern used for react-window mode and means reconnection will Just Work.
- **Decline reuses PASS slot 1001.** Per Phase 14.1-02/03 decisions, the client sends DECLINE_POST_MOVE_ATTACK as action int 8 in the websocket payload; server-side ActionEncoder maps it to slot 1001 and decode disambiguates via pending state.
- **Two distinct visual layers, not one.** A single "highlight" hides the educational value. The footprint teaches range; the bright pulse teaches "click here." Combining them would either hide threat geometry or make every footprint tile look clickable.
