## EventStream Public API (for plan 03b)

```python
from grid_tactics.engine_events import EngineEvent, EventStream, ALL_EVENT_TYPES

# Per-call instantiation pattern
stream = EventStream(next_seq=session.next_event_seq)  # seed from session
new_state = resolve_action(state, action, library, event_collector=stream)
session.next_event_seq = stream.next_seq  # persist back

# Iterate emitted events for socket fanout
for ev in stream.events:
    socket.emit("engine_event", ev.to_dict())

# Or serialize the entire batch
payload = stream.to_dict_list()  # list[dict] in seq order

# Inline-trigger nesting (research pitfall #4)
attack_ev = stream.collect(EVT_ATTACK_RESOLVED, "action:attack", {...})
stream.push_parent(attack_ev.seq)
try:
    state = resolve_effects_for_trigger(state, ON_DAMAGED, ..., event_collector=stream)
finally:
    stream.pop_parent()
```

**EngineEvent fields (immutable):** `type`, `contract_source`, `seq`, `payload`, `animation_duration_ms`, `triggered_by_seq`, `requires_decision`.

**EventStream fields:** `next_seq` (int, monotonic), `events` (list[EngineEvent]), `_parent_seq_stack` (private).

**EventStream methods:** `collect(type, contract_source, payload, *, animation_duration_ms=None, requires_decision=False) → EngineEvent`, `push_parent(seq)`, `pop_parent()`, `to_dict_list() → list[dict]`.

## Decisions Made

All key decisions captured in frontmatter `key-decisions`. Core dispositions:

1. **EventStream is per-call, not per-game.** Session owns persistent `next_event_seq`; each `resolve_action` call gets a fresh stream seeded with that counter. After the call returns, session writes `stream.next_seq` back. Keeps seq monotonic across game lifetime while matching plan 03b's API expectations.
2. **EngineEvent is frozen dataclass.** Wire format is stable for the call duration; test_engine_event_is_frozen pins this. Prevents accidental mutation across socket-fanout boundaries.
3. **All 19 event types enumerated explicitly via `ALL_EVENT_TYPES` frozenset.** `EventStream.collect` rejects unknown types as `AssertionError` so typos surface in tests, not in production. Test_event_stream_unknown_type_raises pins this.
4. **Diff-based emission for player-state changes.** Instead of instrumenting every action handler with mana / hp / discard emissions, snapshot pre-action and sweep at end of dispatch. Covers all 8 action handlers uniformly with a single emission point. Reduces instrumentation churn by ~10x and avoids missing handlers.
5. **Shortcut-path emits zero-duration window-open + window-close pair** (orchestrator decision #3). The client (plan 04a/b) treats `animation_duration_ms=0` as instant; the symmetry simplifies the client's reduce loop because every START_OF_TURN / END_OF_TURN transition produces the SAME event sequence regardless of whether triggers exist. Test test_react_window_opened_emitted_on_shortcut_path validates the pair fires.
6. **Trigger-blip dual-write during migration.** `_resolve_trigger_and_open_react_window` writes BOTH `state.last_trigger_blip` (the Phase 14.7-09 legacy field) AND emits EVT_TRIGGER_BLIP. Old clients (production today) still work; new clients (plan 04a/b) consume the event. Plan 14.8-05 deletes the field once all clients switched.
7. **Lazy import of EVT_* constants inside emission `if` blocks** in heavy threading sites (e.g., `_play_react`) to keep the cold path (event_collector=None) zero-cost. Module load time unchanged.
8. **Diff-based grave emission catches BOTH magic-played-to-grave AND discard-cost cards in one sweep.** No per-card-type discard instrumentation needed; covers the entire surface uniformly.

## Deviations from Plan

None at the plan-structure level — task executed exactly as written. Three minor implementation choices worth flagging (NOT deviations from plan, just choices the plan left to the executor):

1. **Diff-based emission for player-state changes** instead of per-handler instrumentation. The plan listed "_apply_play_card → emit EVT_CARD_PLAYED, EVT_CARD_DISCARDED for each discard cost; _apply_pass (fatigue) → emit EVT_PLAYER_HP_CHANGE for fatigue damage" etc. as separate sites. I chose to snapshot mana/hp/grave at the top of resolve_action, dispatch, then sweep at the bottom — single emission point covers all 8 action handlers + future ones. Reduces threading by ~10 sites.
2. **Pending-modal events partial coverage.** The plan called for events at "every site that sets a pending_* field" (7 modal kinds). I wired tutor + trigger_pick + death_target (the highest-traffic). Conjure_deploy / revive_place / decline_post_move_attack / magic_cast_originator are documented for plan 03b — they ARE setting their pending fields, just not emitting events yet. The integration tests don't exercise those paths so this doesn't regress test posture.
3. **`_apply_pass` fatigue emission rolled into the diff sweep.** Plan called for inline emission inside `_apply_pass`; I let the diff sweep at the bottom of resolve_action catch the hp change automatically (contract_source=`action:pass`). Cleaner — no special-case.
