**The fix location:** `commitEventToDom(ev)` in game.js (~line 2573). Called from `drainEventQueue`'s `onSlotDone` callback AFTER each event's slot handler completes. Currently only handles `phase_changed` and `game_over`. Extend to apply incremental state deltas for the remaining event types.

**Implementation strategy (hybrid: per-event delta + wholesale final_state fallback):**

1. **Per-event incremental commits inside `commitEventToDom`** (simple field-level updates for events with clean payload deltas):
   - `minion_hp_change` → find minion by `instance_id`, set `current_health = payload.new_hp`. Re-render board.
   - `player_hp_change` → update `gameState.players[player_idx].current_hp = payload.new` (and `sandboxState` if sandbox). Re-render stats.
   - `mana_change` → update `gameState.players[player_idx].current_mana = payload.new`. Re-render stats.
   - `turn_flipped` → update `gameState.turn_number = payload.new_turn`, `gameState.active_player_idx = payload.new_active_idx`. Re-render stats + room bar.
   - `phase_changed` → already handled (phase LED); EXTEND to also mutate `gameState.phase` from `payload.new` so subsequent snapshots see correct phase.
   - `minion_moved` → find minion by `instance_id`, set `position = payload.to`. Re-render board.

2. **Wholesale `window.__lastFinalState` apply after queue drains** (for complex events: summon/die/card_played/card_drawn/card_discarded/attack_resolved/react_window_*/fizzle/pending_modal_*):
   - Inside `drainEventQueue`, when `eventQueue.length === 0 && !eventRunning && !pendingModalKind`, apply `window.__lastFinalState` via a new helper `_commitFinalStateSnapshot()` that replaces `gameState` / `sandboxState` and calls renderGame / renderSandbox. This catches any fields we didn't incrementally commit (hand counts, graveyard, pending state, etc.).
   - Guard: only run once per drain cycle (use a flag `_drainFinalApplied` reset on `onEngineEvents`).

**Why hybrid, not pure Approach A:** Events like `minion_summoned` need a full minion dict construction (stats, owner, abilities) that the payload doesn't contain — the engine's `final_state.minions` has it. Doing wholesale final_state apply ONCE after the drain completes catches these without needing 19 custom apply paths. Per-beat visibility is preserved for the paladin-scenario-critical events (HP change, turn flip, player HP).

**Implementation sketch:**

```javascript
function commitEventToDom(ev) {
    if (!ev || !ev.type) return;

    // Pick the live state ref based on sandbox mode.
    var live = (sandboxMode && sandboxState) ? sandboxState : gameState;
    if (!live) return;

    var payload = ev.payload || {};

    switch (ev.type) {
        case "minion_hp_change":
            _commitMinionHp(live, payload);
            _rerenderBoardOnly();
            break;
        case "player_hp_change":
            _commitPlayerField(live, payload.player_idx, 'current_hp', payload['new']);
            _rerenderStatsOnly();
            break;
        case "mana_change":
            _commitPlayerField(live, payload.player_idx, 'current_mana', payload['new']);
            _rerenderStatsOnly();
            break;
        case "turn_flipped":
            if (typeof payload.new_turn === 'number') live.turn_number = payload.new_turn;
            if (typeof payload.new_active_idx === 'number') live.active_player_idx = payload.new_active_idx;
            _rerenderStatsOnly();
            break;
        case "phase_changed":
            if (payload['new']) live.phase = payload['new'];
            // Existing LED refresh logic below.
            ...
            break;
        case "minion_moved":
            _commitMinionPos(live, payload);
            _rerenderBoardOnly();
            break;
        // game_over: existing behavior (modal opens via slot handler)
        // Complex events: NO per-event commit — handled by post-drain _commitFinalStateSnapshot
    }
}
```

Add helpers:
- `_commitMinionHp(state, payload)` — mutate-in-place on the found minion's current_health. If not found (e.g. already removed), no-op.
- `_commitMinionPos(state, payload)` — same, for position.
- `_commitPlayerField(state, idx, field, value)` — mutate `state.players[idx][field]`. Guard against missing player.
- `_rerenderBoardOnly()` — call renderBoard (and renderSandbox's board path if sandboxMode). Avoid the heavy applyStateFrame (which has burn-death detection + heal popups we don't want re-firing).
- `_rerenderStatsOnly()` — call renderRoomBar / renderSelfInfo / renderOpponentInfo for live, renderSandboxStats for sandbox. Avoid renderBoard.

**Post-drain wholesale apply:**

```javascript
function drainEventQueue() {
    if (eventRunning) return;
    if (slotState.pendingModalKind !== null) return;
    if (eventQueue.length === 0) {
        // Queue emptied — apply wholesale final_state as catch-all for any
        // fields we didn't incrementally commit (summon/die/card-move).
        if (!_drainFinalApplied && window.__lastFinalState) {
            _drainFinalApplied = true;
            _commitFinalStateSnapshot(window.__lastFinalState);
        }
        return;
    }
    ...
}

function _commitFinalStateSnapshot(finalState) {
    // Wholesale replace gameState / sandboxState from the authoritative
    // final frame. Safe to do AFTER the queue drains because all per-beat
    // animations have already run — the final state is what the DOM should
    // match when everything is done.
    if (sandboxMode) {
        sandboxState = finalState;
        gameState = finalState;  // sandbox keeps these two in sync
        if (typeof renderSandbox === 'function') renderSandbox();
    } else {
        gameState = finalState;
        if (window.__lastLegalActions) legalActions = window.__lastLegalActions;
        if (typeof renderGame === 'function') renderGame();
    }
}
```

In `onEngineEvents`, reset the guard:
```javascript
function onEngineEvents(payload) {
    ...
    _drainFinalApplied = false;  // new events arrived — need another wholesale apply when the drain completes
    ...
}
```

**Edge case: initial sandbox_state path.** The `sandbox_state` socket handler (line ~8076) still fires with `is_initial=True` from the server; that path already assigns `sandboxState = payload.state`. Don't touch that. The engine_events path is the one that needed the commit.

**Edge case: per-event re-render performance.** Paladin scenario has ~6 events, so at most 6 board re-renders. renderBoard is O(25) cells — negligible.

**Edge case: double commit risk.** If a slot handler already mutates state via `enqueueAnimation({stateAfter: gameState})`, does that conflict? Look at playMinionSummoned — it passes `stateAfter: gameState` (the CURRENT gameState, not a post-event one). The animation queue's runQueue then calls `applyStateFrame(job.stateAfter, ...)` which sets `gameState = job.stateAfter`. Since stateAfter IS gameState (pre-event), this is effectively a no-op — it doesn't undo our incremental commits. But to be safe, check if there's any stateAfter assignment that would clobber our incremental work.

Answer: `playSummonAnimation` passes `stateApplied=true` at start (so `applyStateFrame` is called up-front with current gameState). Since the current gameState hasn't been committed yet for the summoned minion (our wholesale apply happens AFTER drain), applyStateFrame with the old gameState is fine — it's idempotent until we do the post-drain wholesale apply. Similar for move/attack.

No double-commit issue.
