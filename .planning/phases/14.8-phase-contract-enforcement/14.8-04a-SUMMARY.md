## Accomplishments

### Infrastructure (Task 1, commit `67b268e`)

- **eventQueue / eventRunning / lastSeenSeq / slotState globals** — Inserted near the AnimationQueue declaration (~line 2456) so it's adjacent to Phase 14.3 animation infrastructure. eventQueue holds inbound EngineEvent dicts in seq order; eventRunning gates re-entry while a slot animates; lastSeenSeq starts at -1 sentinel; slotState reserves spellStageChain / pendingModalKind / pendingModalDeadline for plan 04b.

- **resetEventQueue()** — Single helper that clears queue, eventRunning, lastSeenSeq, slotState. Called from 6 sites: `onGameStart` (live PvP fresh game), `sandboxActivate` (entering sandbox screen), `sandboxDeactivate` (leaving sandbox screen), `sandbox-reset-btn` click handler, `sandbox-load-file` reader.onload (file load), `sandbox-paste-btn` click handler (share code load), `sandbox-slot-load-btn` click handler. Each matches a server-side `SandboxSession._next_event_seq = 0` or `GameSession.next_event_seq = 0` reset point.

- **onEngineEvents(payload) Socket.IO subscriber** — Bound once in `initSocket` via `socket.on('engine_events', onEngineEvents)`. Single subscription covers both live PvP and sandbox (same socket); the payload's `is_sandbox` flag is informational. Iterates `payload.events`, dedupes via `ev.seq <= lastSeenSeq` (logs `console.warn` and skips), pushes accepted events into `eventQueue`, stashes `final_state` + `legal_actions` into `window.__lastFinalState` / `window.__lastLegalActions` for error-recovery snapshot fallback, kicks `drainEventQueue()`.

- **drainEventQueue()** — Shifts one event from the head of the queue, sets `eventRunning=true`, calls `playEvent(ev, onSlotDone)`. The slot's `done` callback re-enters: `eventRunning=false` → `commitEventToDom(ev)` → `drainEventQueue()`. Wrapped in try/catch so a crashing handler logs the error and advances the queue instead of deadlocking. Modal gate at the top: if `slotState.pendingModalKind !== null`, return without draining — reserved for 04b's pending_modal_opened/resolved pair.

- **commitEventToDom(ev)** — Stub hook. The plan body shows it patching mana / phase / legal_actions per event, but those mutations are owned by the snapshot path (state_update emits the full state). Patching twice would race. Stub preserved for plan 04b/05 to fill in once snapshot path is removed.

- **playEvent(ev, done) dispatcher** — Switch with all 19 case branches. 10 routed to `playMinionSummoned` / `playMinionDied` / `playMinionHpChange` / `playMinionMoved` / `playAttackResolved` / `playCardDrawn` / `playCardPlayed` / `playCardDiscarded` / `playInstant` (mana_change) / `playPlayerHpChange`. 9 stubbed with `console.warn("[eventQueue] <type>: stub — implemented in 04b") + setTimeout(done, 0)`: react_window_opened, react_window_closed, phase_changed, turn_flipped, trigger_blip, pending_modal_opened, pending_modal_resolved, fizzle, game_over. Default branch warns on unknown event type.

- **playInstant(ev, done)** — Zero-duration fallback for events whose state is committed by the snapshot path (mana_change in 04a). Just `setTimeout(done, 0)`.

- **window.__eventQueueDebug** — Mirrors the existing `window.__animDebug` pattern. Exposes queue / running / lastSeenSeq / slotState getters + reset() hook for browser devtools poking.

## Posture: snapshot path stays alive

The plan body describes an `applyStateFrameSelective(payload, skipDetect=[...])` shim that suppresses the detect-* helpers in `applyStateFrame` for the 10 event types now owned by eventQueue. This was NOT implemented in 04a because:

1. The detect logic is **inline** in `_applyStateFrameImmediate` (heal/burn HP popups at line ~3171, fatigue nudge, turn banner trigger, trigger blip dispatch) — not factored out into named detect-* helpers. Factoring is itself plan 04b/05 work.
2. Plan failure_handling explicitly licenses the double-render tradeoff: *"If the dispatcher's slot scheduling causes obvious visual glitches (jank, double-firing): document and continue — plan 04b will replace the snapshot path which is the source of double-render."*
3. Server emits state_update FIRST and engine_events SECOND in the same socket flush, so the snapshot path's animation jobs queue AHEAD of the event-driven ones in the same `animQueue` — they run **sequentially**, not in parallel. The visible posture is "N animations from snapshot path, then N animations from eventQueue, in order" — twice as long but not overlapping.

This means after 04a, the user MAY see each summon / move / attack animation play twice in sequence. That's documented and acceptable. Plan 04b suppresses the snapshot-driven path; plan 05 deletes it.

## Reset Point Wiring

Six client-side reset call sites match server-side counter resets:

| Client site | Trigger | Server counterpart |
|-------------|---------|--------------------|
| `onGameStart` | `socket.on('game_start', ...)` | RoomManager.request_rematch builds new GameSession (next_event_seq=0) |
| `sandboxActivate` | User opens Sandbox tab | First sandbox_create/sandbox_load fresh-creates SandboxSession (_next_event_seq=0) |
| `sandboxDeactivate` | User leaves Sandbox tab | (defensive — no server reset needed; clears stale events) |
| `sandbox-reset-btn` click | User clicks Reset in sandbox toolbar | SandboxSession.reset() resets _next_event_seq to 0 |
| `sandbox-load-file` reader.onload | User loads JSON file | SandboxSession.load_dict() resets _next_event_seq to 0 |
| `sandbox-paste-btn` click | User pastes share code | SandboxSession.load_dict() resets _next_event_seq to 0 |
| `sandbox-slot-load-btn` click | User clicks Load on a server-saved slot | SandboxSession.load_dict() resets _next_event_seq to 0 |

Without these, the first event after a reset would arrive with seq=0 ≤ lastSeenSeq=N (from the previous session) and be silently dropped by the dedupe gate.
