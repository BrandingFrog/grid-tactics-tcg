**Replace the stub branches in playEvent (introduced in 04a) with full implementations.**

For each, the rule is: read payload, kick off the existing animation that previously fired from applyStateFrame's detect-helpers OR from the old ad-hoc gate code, call `done()` after the animation completes.

The 9 handlers in this plan:

1. **playReactWindowOpened(ev, done)** — push to spellStageChain. Reuse the existing _spellStage / _showSpellStage logic. Payload: `{react_context, react_player_idx, source_event_seq}`. The chain LIFO behavior from Phase 14.7-09 is preserved as a slot-internal state machine. If react_window_opened arrives while a stage is already open (chain extension), don't open a new stage — push to chain. Done() after fade-in completes.

2. **playReactWindowClosed(ev, done)** — pop spellStageChain. If chain is now empty, fade out the spell stage; otherwise just decrement chain. Reuse the existing _hideSpellStage exit-fade callback. Done() after fade-out completes.

3. **playPhaseChanged(ev, done)** — typically zero-duration (phase indicator updates instantly via commitEventToDom — extend Task 1 of 04a's commitEventToDom to also handle phase_changed). The visual flash for start/end phase indicators (Phase 14.7-09's `_flashPhaseLed`) fires here. Payload: `{prev, new}`. Done immediately.

4. **playTurnFlipped(ev, done)** — refactor existing `_showTurnBanner` to be called from here. Payload: `{prev_turn, new_turn, new_active_idx}`. Banner shows for ev.animation_duration_ms (default 1500ms). Done() after fade-out.

5. **playTriggerBlip(ev, done)** — refactor existing `_fireTriggerBlipAnimation` to be called from here. Payload: `{trigger_kind, source_minion_id, source_position, target_position, effect_kind}`. The function already exists from Phase 14.7-09; just change its call site. Done() after the ~900ms blip sequence completes.

6. **playPendingModalOpened(ev, done)** — set `slotState.pendingModalKind = ev.payload.modal_kind`; show the modal via the existing `syncPending*UI` function (pick the right one based on modal_kind: tutor_select → showTutorModal; death_target_pick → highlight valid death targets; trigger_pick → showTriggerPickerModal; conjure_deploy → highlight deploy zones; revive_place → highlight revive zones; magic_cast_originator + post_move_attack → no modal, just a hint). Then call done() — the queue PAUSES (drainEventQueue checks pendingModalKind before shifting next event). The user's input event handler (which fires the resolution action) is NOT changed; the SERVER will emit `pending_modal_resolved` after processing the resolution action, which arrives via engine_events and triggers playPendingModalResolved.

7. **playPendingModalResolved(ev, done)** — clear `slotState.pendingModalKind = null`; close the modal via the existing modal-close functions; call done(). The drain loop resumes.

8. **playFizzle(ev, done)** — show a brief puff-of-smoke at the source position (CSS keyframe; ~350ms). New CSS class `.fizzle-puff`. Payload: `{source_event_seq, reason}`. Done after animation.

9. **playGameOver(ev, done)** — show the game over modal via existing showGameOverModal. Payload: `{winner, reason}`. Done immediately (modal stays until user dismisses).

Also extend `commitEventToDom` (added in 04a) so that `phase_changed` updates the phase indicator and `game_over` triggers the game-over modal:

```javascript
function commitEventToDom(ev) {
    if (ev.type === "mana_change") {
        updateManaDisplay(ev.payload.owner, ev.payload.new);
    } else if (ev.type === "phase_changed") {
        updatePhaseIndicator(ev.payload.new);
    } else if (ev.type === "game_over") {
        showGameOverModal(ev.payload.winner, ev.payload.reason);
    }
    if (window.__lastLegalActions) {
        renderLegalActions(window.__lastLegalActions);
    }
}
```
