## Decisions Made

All key decisions are captured in frontmatter `key-decisions`. Core dispositions:

1. **Smoking-gun #1 (resolve_summon_declaration_originator)** — chose RE-TAG with new system source over WIDEN action:play_card to allow REACT. Reason: action contracts are entry-points (player-submitted); the drain-step mutation is engine-driven. Keeping action:play_card tight to ACTION-only preserves the strong invariant.
2. **Smoking-gun #2 (enter_end_of_turn from REACT)** — chose WIDEN over RE-TAG. Reason: enter_end_of_turn IS the canonical end-of-turn entry point; its first action is `state = replace(state, phase=END_OF_TURN)`. The REACT-on-entry is a transient predecessor state, not a different mutation. Tight tagging would require splitting into 2 functions for no semantic gain.
3. **Test-fixture phase fixes** (test_status_effects burn-tick, test_effect_resolver fizzle) preferred over contract widening. Reason: the production code paths only ever call those helpers from the strict phase; the tests were synthesizing wrong-phase states for unit-isolation convenience. Aligning tests with production behavior keeps contracts tight.
4. **conftest uses `os.environ.setdefault` (not a pytest fixture)** because phase_contracts caches the mode at first read — a fixture would set the env after the cache is populated. setdefault at module load runs before any other import.
5. **Module-level default in phase_contracts.py remains `off`** — back-compat for non-test consumers (e.g. RunPod training that imports the module without conftest). Test for that updated to reflect new semantics.

## Deviations from Plan

None at the plan-structure level — both tasks executed exactly as written. Three minor in-task choices worth flagging:

1. **TestTriggerInvariants implemented as a single aggregate test, not a parametrized one.** The plan called for parametrization across (card, effect, phase). I chose an aggregate that iterates internally and collects ALL violations into one rich failure message — this gives better diagnostics for the M2 budget-gate triage path (see ALL violations in one test failure rather than 100s of separate failures). If a future plan needs per-card-per-trigger-per-phase isolation, splitting the iteration into pytest.mark.parametrize is mechanical.
2. **`_fire_trigger_for_card` helper signature corrected vs plan example.** Plan example had `(state, card_def, minion, trig, library)` — actual `resolve_effects_for_trigger` signature is `(state, trigger, minion, library)`. Fixed to match production signature.
3. **Pending-modal sentinels:** plan didn't specify exact values; chose minimal valid sentinels (int 0 for player_idx fields, dict for pending_death_target) that satisfy `getattr(state, field) is not None` without requiring fully-realistic state.

## Issues Encountered

**One unexpected source category surfaced beyond the 4 plan-01 documented smoking-guns: the `trigger_kind` short-form mismatch.** The plan-01 implementation tagged with `f"trigger:{trigger.trigger_kind}"` where trigger_kind is `"end_of_turn"` (a short label used in PendingTrigger for serialization compactness). The PHASE_CONTRACTS table uses `trigger:on_end_of_turn` (the long-form TriggerType-derived key). This mismatch produced 18 violations in the engine subset and 7380 in the broader suite (because every end-of-turn trigger drain in any test fires it). The fix is a 4-entry dict mapping short → long form. This is the kind of thing that can ONLY be caught by a runtime invariant test — static analysis can't see the string-construction mismatch.

**One source's full contract-allowed set was widened beyond plan-01's intent for picker-modal resume paths.** Both `trigger:on_start_of_turn` and `trigger:on_end_of_turn` now allow REACT in addition to their natural phase. Plan-01's table listed only START_OF_TURN / END_OF_TURN per the original research audit. The simultaneous-trigger picker modal (introduced in 14.7-05) opens a REACT window for player input, then the TRIGGER_PICK action resumes `_resolve_trigger_and_open_react_window` with state.phase=REACT. Without REACT in the allowed set, every picker-resumed trigger drain fires a violation. This is documented inline in the table.
