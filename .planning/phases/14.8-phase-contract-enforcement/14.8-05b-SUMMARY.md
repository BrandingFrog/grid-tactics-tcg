## Approach Chosen

**Hybrid: Per-event incremental commits + post-drain wholesale apply** (neither pure A nor pure B).

- **Per-event path** (in `commitEventToDom`, fires between slot handler and next event) — handles 6 event types with clean payload deltas:
  - `minion_hp_change` → `_commitMinionHp` (find by instance_id, set current_health = payload.new_hp) + renderBoard
  - `player_hp_change` → `_commitPlayerField(state, idx, 'hp', payload.new)` + renderStats
  - `mana_change` → `_commitPlayerField(state, idx, 'current_mana', payload.new)` + renderStats
  - `turn_flipped` → `state.turn_number = payload.new_turn` + `state.active_player_idx = payload.new_active_idx` + renderStats
  - `phase_changed` → `state.phase = payload.new` + phase-LED refresh (already existed)
  - `minion_moved` → `_commitMinionPos` (find by instance_id, set position = payload.to) + renderBoard

- **Wholesale path** (in `drainEventQueue`, fires ONCE when queue drains to empty AND not running AND no pending modal) — applies `window.__lastFinalState` wholesale to sandboxState/gameState + calls renderSandbox/renderGame. Catches summon/die/card_*/attack/react_window_*/fizzle/pending_modal_* which need complex object updates (new minion dicts, hand/graveyard recomputation).

- **Guard** (`_drainFinalApplied`) ensures the wholesale apply fires AT MOST ONCE per `onEngineEvents` frame. Reset whenever new events arrive.

**Why not pure A:** `minion_summoned` needs a full minion dict (owner, stats, ability state) that the payload doesn't carry — would require client-side reconstruction from `cardDefs` + card_numeric_id. `card_drawn` / `card_played` / `card_discarded` need hand/graveyard list mutation which is error-prone. `attack_resolved` already has `minion_hp_change` follow-up events for the HP impact.

**Why not pure B:** Paladin scenario needs HP=30 to persist during the spell stage (not jump to 32 immediately when the first engine_events frame lands) AND then tick to 32 at the trigger_blip beat. Pure post-drain apply would hold HP at 30 until the FULL chain drains (~5s) then jump it to 32 — still visually acceptable but loses per-beat fidelity for turn_flipped (which fires mid-drain).

## Decisions Made

- **Approach: hybrid (per-event deltas + post-drain wholesale catch-all)** over pure A or pure B. Rationale: pure A requires ~19 slot-specific state-reconstruction paths (summon, die, card_drawn/played/discarded, etc.) that are error-prone; pure B loses per-beat fidelity for turn_flipped / HP changes (which DO have clean payload deltas and should commit at their event's beat).
- **Guard pattern**: `_drainFinalApplied` flag prevents redundant wholesale applies on every drain call (e.g. when a modal opens and closes multiple times mid-chain). Reset to false whenever `onEngineEvents` receives a new frame.
- **Targeted re-render** (renderBoard-only or stats-only, not applyStateFrame): avoids double-firing heal popups + burn-death detection that the per-event slot handlers already own. Also cheaper (O(25) cells vs full game re-render + diff).
- **Keep `onStateUpdate` / `onSandboxState` alive for initial snapshots** (plan 05 decision retained): the server still emits `is_initial=True` frames on sandbox create/load/reset/slot-load, which these handlers consume. No changes to those paths.

- **Found during:** Task 1 Playwright verification (commit trace showed 10 events, 0 of which were `minion_hp_change` for the paladin heal)
- **Issue:** The paladin's `on_end_of_turn` `passive_heal` effect modifies `current_health` in `_apply_heal_to_minion` (effect_resolver.py:188) but does NOT emit `EVT_MINION_HP_CHANGE`. The trigger_blip event fires for the visual pulse, but no HP-change event follows.
- **Impact on this plan:** The per-event incremental commit for HP changes won't fire for passive heals — HP stays at 30 until `_commitFinalStateSnapshot` runs at queue-drain. Result: HP ticks to 32 at t=4496ms (right after turn_flipped commits at t=4441ms) rather than at the earlier trigger_blip beat (~t=1930ms).
- **Visual acceptability:** Still meets the user's success criterion "HP=32 happens before spell stage closes" (stage hides at t=5368ms, HP=32 at t=4496ms — ~870ms margin). HP stayed at 30 for >1500ms (until t=4496ms actually — 4.5s). Turn flips to 2 with banner.
- **Out of scope:** The fix is in the engine (emit `minion_hp_change` alongside the passive heal in action_resolver.py's end-of-turn trigger flow). Logged for follow-up plan 06 UAT or a future engine-side fixup.
- **Files modified (none this plan):** This observation does not require a fix in plan 05b. It's an engine-layer enhancement.
