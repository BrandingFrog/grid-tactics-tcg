# Phase 14.8: Phase Contract Enforcement — Research

**Researched:** 2026-04-21
**Domain:** Server engine invariants + wire-format/animation pipeline (sandbox AND live PvP)
**Confidence:** HIGH for engine inventory/audit (read directly), MEDIUM for client architecture (read directly but several event sources interlock), MEDIUM for the proposed phase decomposition (informed by codebase but not validated against the planner)

> No external library research needed — the work is entirely in-house Python + vanilla JS. The "library" question this phase asks is: "what's the right primitive for an event-sourced state machine?" That's a design question, not a library lookup. Any external pattern (event sourcing, command bus, FSM library) is overkill here — the engine is small (~5k LOC) and already has frozen-dataclass + replace() semantics that make a hand-rolled contract table the cleanest answer.

## Summary

Phase 14.8 makes "out-of-phase mutation" architecturally impossible on both sides of the wire. The current architecture (post-14.7-09) has the invariant present in spirit — every mutation site sits behind some gate — but the gate logic is scattered across `react_stack.py`, `action_resolver.py`, and `effect_resolver.py`, and the wire format is a post-resolution state snapshot that the client diffs to reconstruct what happened. The recent paladin-heal-during-rat-react bug and its v1/v2 fixes are the clearest evidence: the client has accumulated 4+ ad-hoc gates (`_sandboxFrameQueue`, `_pendingPostStageFrame`, `_pendingTriggerBlip`, `_pendingTurnBanner`) trying to reconstruct an event ordering from snapshot diffs. Each gate catches one symptom; none catch the underlying cause.

The phase has two halves that must land together:

1. **Server: contract source on every mutation.** Every state-mutating call sits inside one of four trigger contexts (`trigger:`, `status:`, `action:`, `system:`). A central `PHASE_CONTRACTS` table maps each contract to its allowed `TurnPhase` set; the engine asserts at apply time and raises `OutOfPhaseError` on violation. The wire format becomes an ordered event stream tagged with `contract_source` and `animation_duration_ms`, replacing the current snapshot-diff approach.

2. **Client: single eventQueue, slot-based.** Replace the ad-hoc gates with one queue. Each event declares its slot kind; the queue serializes commits. DOM only reflects events that have been "played" (animation finished). Same code path for sandbox AND live PvP — the only difference today is the per-frame emit in sandbox, which becomes irrelevant once events replace snapshots.

**Primary recommendation:** Land the server side first as a non-breaking refactor (new `EngineEvent` stream emitted ALONGSIDE the existing snapshot, contract assertions added with feature flag `ENFORCE_CONTRACTS=False` by default). Then migrate the client to consume events. Then flip the flag to `True` and remove the snapshot path. This staged migration keeps the 500+ pytest tests passing throughout.

## Inventory of Contract Sources Today

### Trigger-bound contracts (`trigger:<name>`)

`TriggerType` enum has 11 values. For each, here is the canonical phase the trigger should fire in. **Critical:** anchor contracts on `trigger`, NOT on `effect_type` (per user guidance — `passive_heal` is the EFFECT, the trigger is what binds the phase).

| Trigger | Allowed Phase(s) | How fired today | Notes |
|---|---|---|---|
| `ON_PLAY` | ACTION (caster's) OR REACT (during react resolve) | `_cast_magic` (originator on stack), `_play_react`, react resolve LIFO loop | Same trigger fires in two phases legitimately — a magic ON_PLAY resolves during the LIFO drain (REACT), but a react card's ON_PLAY also resolves there. The contract is "ACTION or REACT" not a single phase. |
| `ON_SUMMON` | REACT only (window B / AFTER_SUMMON_EFFECT) | `resolve_summon_effect_originator` | Spec §4.2: summons open compound react windows. ON_SUMMON ALWAYS resolves inside Window B. |
| `ON_DEATH` | **Any phase** (action / react / start / end) | `_cleanup_dead_minions` enqueues into `pending_trigger_queue_*`, `_resolve_trigger_and_open_react_window` drains | Death is the wildcard. A minion can die during ACTION (combat, sacrifice), during REACT (negate kills your minion), during START_OF_TURN (burn tick), during END_OF_TURN (other end-of-turn effect). Contract MUST allow all phases — but the contract source is still `trigger:on_death`, not a wildcard `*`. The contract table just lists all 4 phases for this trigger. |
| `ON_ATTACK` | ACTION only | `_apply_attack` calls `resolve_effects_for_trigger` inline | Pyre Archer's burn-on-attack. Currently fires INLINE, not via the trigger queue — possible 14.8 cleanup target. |
| `ON_DAMAGED` | ACTION only (today, via `_apply_attack`) | `_apply_attack` calls `resolve_effects_for_trigger` inline | Same — inline. NO card uses this trigger today (search of `data/cards/` returned zero matches). It's reserved schema. |
| `ON_MOVE` | ACTION only (caster's) | `_apply_move` calls `resolve_effects_for_trigger` inline | Furryroach + Rathopper rally-forward. Inline today. |
| `ON_DISCARD` | ACTION only (during a play-with-discard-cost) | `_apply_play_card`'s discard cost loop | Magic Dark Matter Stash uses this. Inline. |
| `AURA` | All (continuously checked) | `tick_status_effects` reads `BURN_BONUS` aura at tick time; aura is a steady-state read, not a fired effect | Auras are a passive read, not a mutation. The contract here is "auras may be READ in any phase" — they don't mutate. Only the `BURN_BONUS` aura currently exists. |
| `ON_START_OF_TURN` | START_OF_TURN only | `fire_start_of_turn_triggers` → `_enqueue_turn_phase_triggers` → `drain_pending_trigger_queue` → `_resolve_trigger_and_open_react_window` | Fallen Paladin's heal. Properly phase-bound. |
| `ON_END_OF_TURN` | END_OF_TURN only | `fire_end_of_turn_triggers` → same pipeline | Emberplague Rat's adjacent burn, Dark Matter Battery's player damage. Properly phase-bound. |

### Status-bound contracts (`status:<name>`)

Statuses tick at fixed phase moments regardless of how they were applied. Per user guidance, these get their own contract category.

| Status | Allowed Phase | How fired today | Notes |
|---|---|---|---|
| `status:burn` | START_OF_TURN only (active player's burning minions) | `tick_status_effects` called from `enter_start_of_turn` step 2 | Boolean `is_burning` flag on minion. Tick is `BURN_DAMAGE + burn_bonus` (Flame Wyrm aura). Persists until death. |

Today there is exactly ONE status type. Burn is the entire status family. Future status types (`is_frozen`, `is_poisoned`, etc.) would each get their own row. The schema needs to be extensible — model it as `StatusType` IntEnum mirroring `TriggerType`.

| Action | Allowed Phase | Pending gate (if any) | What it mutates |
|---|---|---|---|

| `ATTACK` | ACTION | None | Minion HP (both sides), may fire ON_ATTACK / ON_DAMAGED triggers inline |
| `DRAW` | ACTION | None | Hand. **Note:** DRAW is reserved in action space slot 1000 but not legal post-auto-draw. |
| `PASS` | ACTION (fatigue) OR REACT (close window) | None | ACTION-PASS: fatigue damage. | `PLAY_REACT` | REACT only | None | Hand → grave, push react entry on stack, swap `react_player_idx` |
| `SACRIFICE` | ACTION | None | Board (remove minion), opponent player HP |

| `DECLINE_POST_MOVE_ATTACK` | ACTION | `pending_post_move_attacker_id` set | Clears pending |
| `TUTOR_SELECT` | Phase-agnostic | `pending_tutor_player_idx` set | Deck → hand, OR enters `pending_conjure_deploy` |

| `DEATH_TARGET_PICK` | Phase-agnostic | `pending_death_target` set | Board (DESTROY) or minion (PROMOTE) |
| `REVIVE_PLACE` | Phase-agnostic | `pending_revive_player_idx` set | Board, grave |
| `DECLINE_REVIVE` | Phase-agnostic | `pending_revive_player_idx` set | Clears pending |

**Pitfall surfaced:** "Phase-agnostic" pending-modal actions are legal in any of {ACTION, REACT, START_OF_TURN, END_OF_TURN} because the engine sets `phase` to whatever the modal interrupted. The contract for these isn't really "phase-bound" — it's "pending-state-bound." This is a 5th implicit category that the user's 4-category model elides. **Recommendation: model these as `action:<name>` with allowed_phase=ANY but ALSO require `pending_field` to be non-None.** The contract assertion becomes a tuple `(phase ∈ allowed_phases) AND (pending_field set if required)`.

### System-bound contracts (`system:<name>`)

Engine-driven mutations not initiated by an action. From `react_stack.py` + `action_resolver.py`.

| System event | Allowed Phase | Where | Notes |
|---|---|---|---|
| `system:turn_flip` | END_OF_TURN → START_OF_TURN boundary | `_close_end_of_turn_and_flip` | Flips `active_player_idx`, increments `turn_number`, mana regen, auto-draw, discard tracking. |
| `system:enter_start_of_turn` | START_OF_TURN entry | `enter_start_of_turn` | Sets phase, ticks burns, fires Start triggers, opens react window OR shortcuts to ACTION. |
| `system:enter_end_of_turn` | END_OF_TURN entry | `enter_end_of_turn` | Sets phase, fires End triggers, opens react window OR shortcuts to turn flip. |
| `system:enter_react` | Any → REACT | Many sites in action_resolver / react_stack | Sets `phase=REACT`, `react_player_idx`, `react_context`. **This is the most-called system mutation today** — every action-resolver branch ends with one. |
| `system:close_react_window` | REACT → ACTION/END/turn-advance | `close_start_react_and_enter_action`, `close_end_react_and_advance_turn`, `resolve_react_stack` dispatch | Clears react bookkeeping; chains into the next phase transition. |
| `system:cleanup_dead_minions` | After ANY mutation that may zero an HP | `_cleanup_dead_minions` | Removes from board, adds to grave, enqueues on_death triggers. Reentrant — guarded by `_cleanup_skip_drain`. |
| `system:check_game_over` | After cleanup | `_check_game_over` | Pure read + state.replace(is_game_over=True). |
| `system:fizzle` | Any | Inside `resolve_effect` (returns `state` unchanged) | Identity-preserving no-op when target invalid. Worth a contract source so the wire stream can EMIT a fizzle event for the client to play a "puff of smoke" animation. |

**Insight:** `system:enter_react` and `system:close_react_window` are the highest-volume system events. They aren't observable mutations in the gameplay sense (they don't change board / HP / hand), but they DO change the `phase` field which everything else depends on. A contract violation here is the bug class that paladin-heal-v2 was trying to prevent: state committed before the previous react window's animation finished playing.

1. Make the implicit contract EXPLICIT (pass `contract_source` through the call chain)

3. Emit each mutation as an event in the wire stream (not as a snapshot diff at the end)

Net new behavior is the assertion + the event emit. The actual game logic is unchanged.

## Wire-Format Analysis

### Today: post-resolution snapshot

**Server emits:**
- `game_start` (PvP) / `sandbox_state` (sandbox): Full filtered `GameState.to_dict()` + `legal_actions` + `card_defs` + (sandbox-only) `last_action` enrichment
- `state_update` (PvP): Same shape, no `card_defs`
- `game_over`: Same + winner

**Client reconstructs intent by diffing:**
- `detectSpellCast(prev, next)`: new react_stack top OR new card in grave → fire spell-stage-open
- `detectSpellStageClose(prev, next)`: react_stack just emptied → fire spell-stage-close
- `detectReactWindowClose(prev, next)`: phase!=REACT && stack empty → close stage
- `deriveAnimationJob(prev, next)`: reads `last_action` (server-enriched) for ATTACK/SACRIFICE/MOVE; falls back to `pending_action` diff for PLAY_CARD/MOVE
- `deriveCardFlyJobs(prev, next)`: hand multiset diff → fly ghosts
- `deriveDrawJobs(prev, next)`: hand additions → draw animations
- `derivePlayerHpDeltaAnims(prev, next)`: HP change → floating popup
- `last_trigger_blip` field: read on each frame to fire trigger blip animation
- Hand-rolled detection of turn flip (turn_number increment) → banner

**Problems with this:**
1. The client guesses what the engine did from delta. When the engine collapses multiple state changes into one resolve_action call, the client sees ONE diff that represents N events.
2. Transient fields like `last_trigger_blip` exist solely to tell the client "an event happened" — they're a hack to get event semantics on top of a snapshot wire format.
3. The sandbox per-frame emit (commit 9c414f9) was added BECAUSE single-emit collapsed events; but per-frame emit then races with the spell-stage chain animation (paladin-heal bug).
4. There's no explicit ordering between `last_trigger_blip` firing and the underlying state mutation it represents — both arrive in the same payload.

### Proposed: event stream

**Wire shape (proposal):**
```python
class EngineEvent(TypedDict):
    type: str  # "minion_hp_change" | "react_window_open" | "turn_flip" | "trigger_blip" | "fizzle" | etc.
    contract_source: str  # "trigger:on_end_of_turn" | "status:burn" | "action:play_card" | "system:turn_flip"
    seq: int  # monotonic event sequence number per game
    payload: dict  # event-specific shape (see below)
    animation_duration_ms: int  # how long the client should hold this slot
    # Optional fields:
    triggered_by_seq: int | None  # event ordering — this event is a child of an earlier seq
    requires_decision: bool  # true for events that open a pending modal
```

**Server emits per resolve_action:**
```python
{
  "type": "engine_events",
  "events": [<EngineEvent>, ...],   # ordered, non-empty
  "final_state": <GameState.to_dict()>,  # for sanity/round-trip; client should not consume this directly
  "legal_actions": [...],
}
```

**Event types proposed (initial set, derived from the audit above):**

| Event type | Triggered by | Animation slot kind | Payload |
|---|---|---|---|
| `minion_summoned` | `action:play_card` | summon | `{instance_id, card_id, position, owner}` |
| `minion_died` | `system:cleanup_dead_minions` | (none — covered by trigger blips) | `{instance_id, position}` |
| `minion_hp_change` | many | hp_popup | `{instance_id, prev_hp, new_hp, delta}` |
| `minion_moved` | `action:move` | move | `{instance_id, from, to}` |
| `attack_resolved` | `action:attack` | attack | `{attacker, defender, attacker_dmg, defender_dmg, killed}` |
| `card_drawn` | many | draw | `{owner, card_id_if_visible}` |
| `card_played` | `action:play_card` / `action:play_react` | (covered by spell-stage) | `{owner, card_id, target_pos}` |
| `card_discarded` | `action:play_card` (cost) | card_fly | `{owner, card_id}` |
| `mana_change` | many | (none) | `{owner, prev, new}` |
| `player_hp_change` | many | hp_popup | `{owner, prev, new, delta}` |
| `react_window_opened` | `system:enter_react` | spell_stage | `{react_context, react_player_idx, source_event_seq}` |
| `react_window_closed` | `system:close_react_window` | spell_stage_close | `{return_phase}` |
| `phase_changed` | `system:enter_*` | turn_banner OR phase_led | `{prev, new, owner_if_turn_flip}` |
| `turn_flipped` | `system:turn_flip` | turn_banner | `{prev_turn, new_turn, new_active_idx}` |
| `trigger_blip` | `trigger:on_*` | trigger_blip | `{trigger_kind, source_position, target_position, effect_kind}` (current `last_trigger_blip` shape) |
| `pending_modal_opened` | various | (gates input — pauses queue) | `{modal_kind, owner_idx, options}` |
| `pending_modal_closed` | various | (releases input) | `{modal_kind}` |
| `fizzle` | `system:fizzle` | optional puff | `{source_event_seq, reason}` |
| `game_over` | `system:check_game_over` | game_over_modal | `{winner}` |

**Key compatibility considerations:**

1. **`game_start`** keeps its current shape (initial snapshot + card_defs); the event stream takes over from there. A reconnect mid-game replays from a snapshot + the events since.
2. **`last_action`** enrichment (server-side, `enrich_last_action`) becomes UNNECESSARY — it's reconstructed from `attack_resolved` / `minion_moved` / etc. events.
3. **Pending-state enrichment** (`enrich_pending_tutor_for_viewer`, etc.) becomes a `pending_modal_opened` event payload. The current 6 pending-* enrichers fold into one event type with a `modal_kind` discriminator.
4. **`last_trigger_blip`** field on GameState is DELETED — replaced by `trigger_blip` event. This removes the transient-clear-on-next-frame hack.
5. **`react_open` / `react_close`** Socket.IO events don't currently exist in the codebase (already integrated into `state_update`). Proposed: keep that integration — events flow on one channel.
6. **Spectator fanout** uses the same event stream. Spectator god-mode just unfilters the payload of certain events (e.g. `card_drawn` shows the card_id even for opponent).

**Backwards compat for the snapshot:** Include `final_state` in every emit during the migration so old clients can fall back. Once the new client is the only client, drop `final_state` from non-game_start emits.

### Right "single queue" abstraction

The user's success criterion #5 says "single eventQueue replacing the ad-hoc gates." But 11 distinct slot kinds exist with very different durations (0ms for fatigue nudge through ~3.4s for spell stage close). A single FIFO queue won't work — that would force the turn banner to wait behind a spell stage, which is the existing behavior; but it would also force a fatigue nudge to wait, which would feel sluggish.

**Recommendation: ONE queue with TYPED SLOTS.** The queue is a single ordered list of events. Each event declares its `slot_kind` (matching the table above). The slot kind determines:
- Whether the event BLOCKS subsequent events (spell_stage YES, fatigue_nudge NO, hp_popup YES briefly)
- The animation duration (server-provided in `animation_duration_ms`)
- The DOM-commit timing (commit BEFORE / DURING / AFTER the visual)

This collapses `_sandboxFrameQueue`, `_pendingPostStageFrame`, `_pendingTriggerBlip`, `_pendingTurnBanner`, `_spellStageQueue`, and the existing `animQueue` into ONE queue with one drain function. Live PvP and sandbox use the same code path — sandbox's only difference today (per-frame emit from server-side drain) becomes irrelevant because each EVENT is what's being emitted, and event ordering is server-authoritative.

**Specifically, the proposed client architecture:**

```js
var eventQueue = [];         // EngineEvent[]
var eventRunning = false;
var slotState = {};          // per-slot timing / chain state (e.g. spell_stage chain)

function onEngineEvents(payload) {
    // Replaces both onStateUpdate and the sandbox_state handler.
    payload.events.forEach(e => eventQueue.push(e));
    drainEventQueue();
}

function drainEventQueue() {
    if (eventRunning) return;
    if (eventQueue.length === 0) return;
    var ev = eventQueue.shift();
    eventRunning = true;
    playEvent(ev, function onDone() {
        commitEventToDom(ev);  // state commit happens HERE, after the visual finished
        eventRunning = false;
        drainEventQueue();
    });
}
```

`commitEventToDom(ev)` does NOT commit the full GameState — it applies the EVENT'S MUTATION to the DOM. Example: `minion_hp_change` reads `{instance_id, new_hp}` and updates that minion tile only. The `final_state` snapshot is used as a sanity check (e.g. on reconnect) but is not the source of truth for the running render.

### Critical subtlety: events that DON'T animate

Not every event has a visual (e.g. `mana_change`, `phase_changed` in the trivial case, `fizzle`). Those events still need to commit to the DOM, but instantaneously. The drain loop handles this by `playEvent` for non-visual events being a `setTimeout(done, 0)` — same pattern as the existing AnimationQueue's `noop` job. Net: the queue maintains ordering; trivial events drain in microseconds; visual events block.

### React-card spell stage chain handling

The existing spell-stage chain is interesting — it OPENS once (on the first cast) and accumulates cards LIFO. With event ordering, each `react_window_opened` event PUSHES to the chain; each `react_window_closed` event POPS or fully resolves the chain. The chain is a slot-internal state machine that consumes multiple events. The drain function honors that the slot is "still active" for the chain's duration.

## Pitfalls to Flag for the Planner

These are non-obvious complications the planner needs to address explicitly.

### Pitfall 1: ON_DEATH is "any phase" — but it shouldn't be a wildcard

A minion can die in any phase. The contract `trigger:on_death` therefore allows {ACTION, REACT, START_OF_TURN, END_OF_TURN}. But this is NOT a wildcard contract — the source is still specifically `trigger:on_death`. Death triggers fired during the wrong context (e.g. fired during a tutor modal) would still be a violation.

**Resolution:** Allowed-phase set is a tuple, not "any." `trigger:on_death` has `allowed_phases=(ACTION, REACT, START_OF_TURN, END_OF_TURN)`. Wildcard is forbidden — every contract source enumerates its phases.

### Pitfall 2: Same-tick simultaneous effects across multiple contract sources

Example: P1 plays Acidic Rain (action:play_card → trigger:on_play). The cast kills two enemy minions both with Lasercannon (trigger:on_death). Both deaths enqueue. The drain opens the trigger picker modal (action:trigger_pick required). The picked trigger fires DESTROY on a minion that ALSO has on_death (chain reaction).

The wire stream needs to preserve this exact ordering:
1. `card_played` (Acidic Rain, action:play_card)
2. `react_window_opened` (AFTER_ACTION, system:enter_react)
3. ...react chain resolves, possibly more events...
4. `react_window_closed`
5. `minion_died` × 2 (system:cleanup)
6. `pending_modal_opened` (TRIGGER_PICK, system:drain_triggers)
7. ...wait for player input...
8. `trigger_blip` (action:trigger_pick → trigger:on_death)
9. `react_window_opened` (AFTER_DEATH_EFFECT)
10. ...etc.

**Resolution:** Each event has a `seq` integer. The client commits in `seq` order. `triggered_by_seq` lets the UI render parent/child relationships if needed (e.g. a chain visual showing Acidic Rain → killed-rat → deathblast). **The planner must ensure the engine emits events in the seq order of resolution**, NOT in the order they were enqueued. The drain functions naturally do this; just emit at the resolve point, not the enqueue point.

### Pitfall 3: Phase-agnostic pending-state actions

`TUTOR_SELECT`, `DECLINE_TUTOR`, `DEATH_TARGET_PICK`, `REVIVE_PLACE`, `DECLINE_REVIVE`, `TRIGGER_PICK`, `DECLINE_TRIGGER`, `CONJURE_DEPLOY`, `DECLINE_CONJURE` are all legal in ANY phase because they fire from a pending modal. The contract for these is "pending field X must be set" not "phase must be Y."

**Resolution:** The CONTRACTS table for these actions has `allowed_phases=(ACTION, REACT, START_OF_TURN, END_OF_TURN)` AND `requires_pending=<field_name>`. The assertion checks both. Pending-modal events lift the slot's blocking effect on the queue (they're a special "input-required" event type).

### Pitfall 4: Inline ON_ATTACK / ON_DAMAGED / ON_MOVE / ON_DISCARD

These triggers fire INLINE from within action resolvers (`_apply_attack`, `_apply_move`, `_apply_play_card` discard loop), not via the `pending_trigger_queue_*` priority queue. Their contract is correct (`trigger:on_attack` inside `action:attack`), but the wire stream needs to flatten the call tree into a sequence of events. The `triggered_by_seq` field captures the parent-child relationship.

**Resolution:** The event emitter wraps each `resolve_effect` call so any state change inside it emits an event whose `triggered_by_seq` points to the parent action's `seq`. The engine doesn't need to be restructured to use the queue — inline calls keep working, the emitter just sees them.

### Pitfall 5: Fizzle is a non-event

`resolve_effect` returns `state is prev_state` (identity-preserving) when fizzle. There's no `replace()` call to wrap. The event emitter would miss it.

**Resolution:** `resolve_effect` must EXPLICITLY emit a `fizzle` event when returning unchanged. The engine layer can detect this by `id(state) == id(prev_state)` post-call, or `resolve_effect` can return a `(new_state, events)` tuple. The latter is cleaner but more invasive.

### Pitfall 6: The "pending" action gates run before phase dispatch

`resolve_action` checks `pending_death_target`, `pending_revive_player_idx`, `pending_trigger_picker_idx`, `pending_tutor_player_idx`, `pending_conjure_deploy_card`, `pending_post_move_attacker_id` BEFORE looking at `state.phase`. So an action like `DEATH_TARGET_PICK` arrives when `phase=REACT` (because the dying minion was killed during a react chain) but the dispatcher routes it to `apply_death_target_pick` regardless of phase. The contract for `action:death_target_pick` therefore can't assert `phase=ACTION`.

**Resolution:** As in Pitfall 3 — these actions have `allowed_phases=ANY` plus `requires_pending=<field>`. The assertion is "pending field set" not "phase is X."

### Pitfall 7: Cleanup is recursive (chain reactions)

`_cleanup_dead_minions` may be called multiple times during one `resolve_action`. The first call enqueues on_death triggers; the trigger drain may resolve an effect that kills another minion; that triggers another `_cleanup_dead_minions`. The reentrancy is guarded by `_cleanup_skip_drain`. Each cleanup pass emits `minion_died` events with the same `triggered_by_seq` as the cleanup parent. The wire stream needs to handle nested events correctly.

**Resolution:** Events are flat (one ordered list per resolve_action), not nested. The `triggered_by_seq` field is metadata for UIs that want to render a chain visual; the queue itself just drains in sequence. The engine emits 5 `minion_died` events in a row if 5 die in one chain — the client renders them sequentially.

### Pitfall 8: `_runTurnFlipVisuals` re-entrancy guard

Today's `_lastBannerTurnKey` dedupe in `_showTurnBanner` exists because the same `turn_number` can arrive multiple times (initial connect, undo/redo, sandbox reset). With the event stream, the dedupe lives in the SERVER — `turn_flipped` is emitted exactly once per turn flip, never on reconnect. The client doesn't need the dedupe; the queue's natural ordering handles it.

**Resolution:** Drop `_lastBannerTurnKey` in the client. On reconnect, the server sends the `final_state` snapshot WITHOUT replaying historical events. The client renders the snapshot directly without firing animations.

### Pitfall 9: Spectator and viewer filtering

`filter_state_for_player(state_dict, viewer_idx)` hides opponent hand contents. With events, each event needs a per-viewer filter. Example: `card_drawn` event for opponent should hide `card_id`; for own player, show it. `pending_modal_opened` for TUTOR_SELECT should only emit the deck options to the picker, not to the opponent.

**Resolution:** The event emitter takes a `viewer_idx` argument; per-viewer filtering happens at emit time. The current `enrich_pending_*_for_viewer` functions translate directly to per-event filtering. Spectator god mode bypasses the filter (existing pattern).

### Pitfall 10: The originator pattern (14.7-01) defers magic effects

Magic ON_PLAY effects don't resolve at cast time — they sit at the bottom of the react stack and resolve LIFO. The contract for the deferred effect is still `trigger:on_play` but the PHASE at resolution time is REACT (mid-stack-drain). This is the case I called out in the Trigger table: `ON_PLAY` legitimately fires in both ACTION and REACT.

**Resolution:** As above — `trigger:on_play` allowed_phases=(ACTION, REACT). Engineers writing card definitions don't need to think about this; it's encoded in the central CONTRACTS table.

### Pitfall 11: Auto-PASS drain in events.py (line 720-752) and sandbox_session.py (line 230-248)

The server runs an auto-advance loop that calls `resolve_action(state, pass_action(), library)` repeatedly when legal_actions is empty. Each iteration today produces one `state_update` (PvP) or one `sandbox_state` frame (sandbox). With events, each PASS produces multiple events. The drain emits ALL events in order; the client queues them and plays them out in sequence.

**Resolution:** The auto-advance loop's emit point becomes "emit accumulated events from this iteration." Sandbox's `on_frame` callback becomes "emit events from this iteration." Both paths flow through the same serializer (success criterion #4).

### Pitfall 12: The 14.7-09 transient `last_trigger_blip` field is the smoking gun

The field exists because the engine needs to tell the client "this trigger just opened a window" but the snapshot wire format doesn't have a way to say that. So a transient field is set, then cleared on the next `resolve_action`. With events, this becomes a normal `trigger_blip` event in the stream. The field should be DELETED in 14.8 plan 06.

## Open Questions

4. **`pending_modal_opened` for sub-action gates (POST_MOVE_ATTACK, MAGIC_CAST_ORIGINATOR)?** These aren't really "modals" in the UI sense — they're engine-internal pending states that the client renders inline. Should they emit events? Recommend YES for completeness — the client can choose to ignore them. Confirm.

Recommend YES — sandbox actions are first-class events tagged `sandbox:cheat_mana` etc. They don't go through `resolve_action`, so they need their own contract source category (`sandbox:`). Confirm whether this 5th category is acceptable, or if the user wants to keep sandbox edits as plain snapshot patches.

6. **Reconnect / replay scope?** Plan 14.8-08 is optional. Confirm whether 14.8 should include it or defer to Phase 15.

7. **Do auras need contract sources?** Auras are READ continuously (e.g. `BURN_BONUS` is read in `tick_status_effects`). They never mutate, so they have no `replace(state, ...)` site. The contract framework as designed doesn't apply. Confirm: leave auras out, OR add `aura:<name>` for the read-time validation.
