## Issues Encountered

**One ambiguity noticed during tagging (NOT an error — just worth flagging for plan 14.8-02 to validate):**

- Several inner helpers in `resolve_react_stack`'s LIFO loop tag with `contract_source="trigger:on_play"` even though they execute in REACT phase. This IS legal per the table (`trigger:on_play` allows {ACTION, REACT}), so shadow mode does NOT warn. This was the orchestrator's intent in decision #9 — explicit pair.
- However, `resolve_summon_declaration_originator` tags with `action:play_card` (because it's the resolution arm of an action initiated in ACTION) but executes in REACT phase. This DOES warn in shadow mode (`action:play_card allowed=['ACTION'], phase=REACT`). Documented inline as a "smoking gun" for plan 14.8-02 to disposition: either (a) widen `action:play_card` to allow REACT (current intent: NO — actions are action-phase entry points), or (b) re-tag the originator resolution path with a new system source like `system:resolve_summon_originator`. Plan 14.8-02 will pick.

Same situation for `enter_end_of_turn` called from inside `resolve_react_stack` after the AFTER_ACTION react window closes — the function is correctly tagged `system:enter_end_of_turn` (phases ACTION + END_OF_TURN), but the calling phase is REACT. Same disposition pending.

These are exactly the violations plan 14.8-02 was designed to surface. The 102 shadow-mode warnings counted on the engine test subset reduce to a small number of unique source/phase pairs (likely 3-5).

1. `source=action:play_card phase=REACT` — fired from `resolve_summon_declaration_originator` (Window A → Window B compound summon). Stack trace consistently bottoms out at `resolve_react_stack:1458 → resolve_summon_declaration_originator:1140`.

Plan 14.8-02 inherits these as the working baseline to drive to zero. No tag was wrong in strict mode TODAY (default = off), so no engine behavior is broken. The 8 pre-existing failing tests are unchanged.
